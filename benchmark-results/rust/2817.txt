{"title":"Using Git to manage a web site","author":null,"date":"2008-12-17T00:00:00+00:00","main_content":"The HTML source for my (i.e., this) web site lives in a\nGit repository on my local workstation.\nThis page describes how I set things up so that I can make changes live\nby running just \"git push web\".\n\nThe one-line summary:\npush\ninto a\nremote\nrepository that has a detached work tree, and a\npost-receive hook\nthat runs \" git checkout -f \".\n\nThe local repository\n\nIt doesn't really matter how the local repository is set up, but for the\nsake of argument, let's suppose you're starting one from scratch.\n\n$ mkdir website && cd website\n$ git init\nInitialized empty Git repository in /home/ams/website/.git/\n$ echo 'Hello, world!' > index.html\n$ git add index.html\n$ git commit -q -m \"The humble beginnings of my web site.\"\n\nAnyway, however you got there, you have a repository whose contents you\nwant to turn into a web site.\n\nThe remote repository\n\nI assume that the web site will live on a server to which you have ssh\naccess, and that things are set up so that you can ssh to it without\nhaving to type a password (i.e., that your public key is in\n~/.ssh/authorized_keys and you are running\nssh-agent locally).\n\nOn the server, we create a new repository to mirror the local one.\n\n$ mkdir website.git && cd website.git\n$ git init --bare\nInitialized empty Git repository in /home/ams/website.git/\n\nThen we define (and enable) a post-receive hook that checks out the\nlatest tree into the web server's DocumentRoot (this directory must\nexist; Git will not create it for you):\n\n$ mkdir /var/www/www.example.org\n$ cat > hooks/post-receive\n#!/bin/sh\nGIT_WORK_TREE=/var/www/www.example.org git checkout -f\n$ chmod +x hooks/post-receive\n\nNote: earlier versions of this howto depended on setting the git config\nvariables core.worktree to the target directory, core.bare to false, and\nreceive.denycurrentbranch to ignore. But these changes are not needed if\nyou use GIT_WORK_TREE (which didn't work when I first wrote the howto),\nand the remote repository can remain bare.\n\nBack on the workstation, we define a name for the remote mirror, and\nthen mirror to it, creating a new \" master \" branch there.\n\n$ git remote add web ssh://server.example.org/home/ams/website.git\n$ git push web +master:refs/heads/master\n\nOn the server, /var/www/www.example.org should now contain\na copy of your files, independent of any.git metadata.\n\nThe update process\n\nNothing could be simpler. In the local repository, just run\n\n$ git push web\n\nThis will transfer any new commits to the remote repository, where the\npost-receive hook will immediately update the\nDocumentRoot for you.\n\n(This is more convenient than defining your workstation as a remote on\nthe server, and running \" git pull \" by hand or from a cron\njob, and it doesn't require your workstation to be accessible by ssh.)\n\nNotes\n\nA few more things bear mentioning.\n\nFirst, the work tree ( /var/www/www.example.org above) must\nbe writable by the user who runs the hook (or the user needs sudo access\nto run git checkout -f, or something similar).\n\nAlso, the work tree does not need to correspond exactly to your\nDocumentRoot. Your repository may represent only a\nsubdirectory of it, or even contain it as a subdirectory.\n\nIn the work tree, you will need to set the environment variable\nGIT_DIR to the path to website.git before you\ncan run any git commands (e.g. \" git status \").\n\nSetting receive.denycurrentbranch to \" ignore \"\non the server eliminates a warning issued by recent versions of git when\nyou push an update to a checked-out branch on the server. (Thanks to\nMiklos Vajna for pointing this out.)\n\nYou can push to more than one remote repository by adding more URLs\nunder the [remote \"web\"] section in your\n.git/config.\n\n[remote \"web\"]\nurl = ssh://server.example.org/home/ams/website.git\nurl = ssh://other.example.org/home/foo/website.git\n\nThere are also other hooks. See\ngithooks(5)\nfor details. For example, you could use pre-receive to\naccept or deny a push based on the results of an\nHTML validator. Or you could do\nmore work in the post-receive hook (such as send email to\nco-maintainers; see contrib/hooks/post-receive-email ).\n\nI wrote this after reading Daniel Miessler's piece on\nUsing Git to maintain your website (no longer available). He pushes to a\nbare repository on the server and pulls changes into a second clone that\nis used as the DocumentRoot. My implementation has fewer moving parts,\nand keeps.git separate from the DocumentRoot.\n\nNote: some people have reported that this strategy doesn't work under\ngit 1.5.4.3 (because the git checkout -f fails). I know\nit does work with 1.6.x. I haven't investigated further.\n\nVariants\n\nThese are links to articles that readers have sent me. I have glanced at\nthem before linking, but I haven't tried out the instructions myself.\n\nBrazilian Portuguese\nby Thiago Belem.\n\nA 1and1-specific guide\nby Woody Hayday.\n\nA one.com-specific guide\nby Pieter Jan De Smedt.\n\nQuestions and suggestions are welcome."}
