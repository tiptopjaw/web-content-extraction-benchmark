{"title":"Skills in OpenAI Codex","author":"Mastodon","date":"2025-12-19T00:00:00+00:00","main_content":"I've been working through some testing of OpenAI's new skills support in their codex CLI tool.\n\nSkills are currently gated behind a feature flag. You turn them on by running codex --enable skills.\n\nAs I was trying to debug why a skill I'd installed wasn't triggering, Codex told me a bit about its' \"Discovery\" skill.\n\nClaude Code finds skills in the skills subdirectories of installed plugins and the skills directory of your.claude directory at startup. It then shoves those into the system prompt as a list of available skills. At least the last time I looked, there was no way to enable specific skills for specific projects or subdirectories.\n\nIt looks like Codex is doing something very different. It allows you to list additional skills in project documentation. I think I like this?\n\nOpenAI ship some \"global\" skills in ~/.codex/skills/.system - a 'plan' skill and a 'skill-creator' skill. You can drop your own global skills into ~/.codex/skills/. And because it looks like they're just using a system prompt, the \"Discovery\" meta-skill, and the \"File read\" tool rather than custom skill tools, any sort of directory layout inside ~/.codex/skills/ works. They're just looking for any directory in that tree with a 'SKILL.md' file and going from there.\n\nThis is Codex's detailed description of what's going on. (You can likely find the 'real' version of this in the publicly available source code.)\nThe complete “Discovery” entry I have in this chat is:\n\n- Skill Creator: Guide for creating effective skills. This skill should be\nused when users want to create a new skill (or update an existing skill)\nthat extends Codex's capabilities with specialized knowledge, workflows, or\ntool integrations. (file: /Users/jesse/.codex/skills/.system/skill-creator/\nSKILL.md)\n- plan: Plan lifecycle management for Codex plans stored in $CODEX_HOME/plans\n(default ~/.codex/plans). Use when a user asks to create, find, read,\nupdate, delete, or manage plan documents for implementation work or\noverview/reference documentation. (file: /Users/jesse/.codex/skills/.system/\nplan/SKILL.md)\n- Discovery: Available skills are listed in project docs and may also appear\nin a runtime \"## Skills\" section (name + description + file path). These are\nthe sources of truth; skill bodies live on disk at the listed paths.\n- Trigger rules: If the user names a skill (with $SkillName or plain text) OR\nthe task clearly matches a skill's description, you must use that skill for\nthat turn. Multiple mentions mean use them all. Do not carry skills across\nturns unless re-mentioned.\n- Missing/blocked: If a named skill isn't in the list or the path can't be\nread, say so briefly and continue with the best fallback.\n- How to use a skill (progressive disclosure):\n1. After deciding to use a skill, open its SKILL.md. Read only enough to\nfollow the workflow.\n2. If SKILL.md points to extra folders such as references/, load only the\nspecific files needed for the request; don't bulk-load everything.\n3. If scripts/ exist, prefer running or patching them instead of retyping\nlarge code blocks.\n4. If assets/ or templates exist, reuse them instead of recreating from\nscratch.\n- Description as trigger: The YAML description in SKILL.md is the primary\ntrigger signal; rely on it to decide applicability. If unsure, ask a brief\nclarification before proceeding.\n- Coordination and sequencing:\n- If multiple skills apply, choose the minimal set that covers the request\nand state the order you'll use them.\n- Announce which skill(s) you're using and why (one short line). If you\nskip an obvious skill, say why.\n- Context hygiene:\n- Keep context small: summarize long sections instead of pasting them;\nonly load extra files when needed.\n- Avoid deeply nested references; prefer one-hop files explicitly linked\nfrom SKILL.md.\n- When variants exist (frameworks, providers, domains), pick only the\nrelevant reference file(s) and note that choice.\n- Safety and fallback: If a skill can't be applied cleanly (missing files,\nunclear instructions), state the issue, pick the next-best approach, and\ncontinue."}
