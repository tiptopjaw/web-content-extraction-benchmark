{"title":"Syntactic musings on the fallibility effect","author":"Yoshua Wuyts","date":"2025-12-17T00:00:00+00:00","main_content":"defining the problem space\n\nfallible functions\n\nfallible blocks and closures\n\noperating on fallibility\n\nnotes on effect naming\n\nmotivating reification into an abstract type\n\nclosing words\n\nI believe it must have been about three or four years ago when Rust added the\nunstable yeet keyword on nightly which can be used to return new errors in\ntry -functions. More recently Rust has added the unstable bikeshed keyword on\nnightly which gives try -blocks and closures the ability to express which kind\nof error they operate on. We’ve been deferring making decisions of error handling syntax for a while now, and I think that’s actually quite reasonable.\n\nBut if we want things to head for stable, we’re eventually going to want to\ndecide on syntax. And so I think it’s not a bad idea to start working through\nthe entire syntactic space here. And because things are unstable I guess it’s\nunavoidable to also work through the semantic space here. I guess this is a\nlong-winded way of saying: in this post I’m sharing my opinions on Rust’s effect\nfor fallibility. And I’m doing that mostly because I think it’s fun to think about.\n\nDefining the problem space\n\nFor Rust we can categorize the error handling space into three categories. The first category are the places which carry the effect type notation, of which there are three. These are items in the language where for which we want to say: “Hey, this right here is operating with an error handling context”. Those items are:\n\nblocks ( {} )\n\nfunctions ( fn f() {} )\n\nclosures ( || {} )\n\nThe second category are the effect operators. These are used inside of the items marked with the fallibility effect, and allow us to operate on them. There are three kinds of operations we need to support:\n\ncreate (e.g. yield for iteration)\n\npropagate (e.g..await for async)\n\nconsume (e.g. block_on for async, for..in for iteration)\n\nBoth of the categories we’ve seen so far apply to all of Rust’s control-flow effects (async, fallibility, and iteration). But the third category is unique to fallibility only, and that is: specialization. Fallibility has two modes it can operate in:\n\nabstract, where the effect is reified into an abstract -> impl Try\n\nconcrete, where the effect is reified into a specific type (e.g. -> io::Result )\n\nYou can almost think of fallibility as being two separate effects in a trenchcoat. Or maybe as “concrete” being a specialized version of “abstract”. But since both flavors of fallibility are so similar in both purpose and usage, to an end-user we will both to feel similar to one another.\n\nFallible functions\n\nSo for the effect type we have to account for functions, closures, and blocks.\nFor both of these we need to be able to support both abstract and concrete\nvariants. Or put differently: we need to be able to optionally be able spell out\nboth the explicit type when needed.\n\nStarting with functions. Swift uses the throws keyword to annotate a function as fallible. I believe this matches Java, Kotlin, and Scala. This keyword can optionally carry a concrete type, and I think that’s perfect. Here is how I would adapt that for Rust:\nfn foo () {} // `-> ()` (base)\nfn foo () throws {} // `-> impl Try`\nfn foo () throws Err(io::Error) {} // `-> io::Result<()>`\nfn foo () throws None {} // `-> Option<()>`\nfn foo () throws None -> i32 {} // `-> Option<i32>`\n\nThe notation here borrows from pattern types. None and Err imported as part of the Rust prelude, and just like we can write None and Err are in function bodies, we should be able to use them freely in signatures as well. And since we know they are part of an impl Try, we know what the actual type is. The only open question I have is express that in the impl Try trait itself - but the Try trait is still unstable, so that’s something we can figure out.\n\nIn terms of location this follows what Swift does, listing effects after the argument list but before the arrow. This to me makes sense because effects describe properties of the function itself. And because throws says something about the function’s outputs, I expect it to be listed after the argument list. And because we don’t want it to be confused with the logical return type, it makes sense for it to be listed before the return type.\n\nSomething to be aware of also is that it’s notationally important that we are\nalways able strip the effect from the function signature and still end up with a\nvalid function signature. That’s an important property we need to be able to\ngroup effects together and reason about them in sets or as variables. Those\ninclude things like: effect aliases, associated effects, and effect generics:\n// Adding or removing effects from functions should\n// not affect the rest of the function signature:\nasync fn foo () -> i32; // async effect\nconst fn foo () -> i32; // const effect\nfn foo () throws None -> i32; // fallible effect\nfn foo () -> i32; // no effects (baseline)\n// That keeps the door open to enable higher-order reasoning\n// about effects. Using a Flix-inspired notation for fun:\neffect ef = async + const + throws None;\nfn foo () -> i32 \\ ef;\n\nMaybe there’s something to be said for going all the way with our notation\nand start the migration to a general-purpose effect notation. But I’m somewhat intentionally trying to scope things down here and argue for more localized decisions on how we notate fallibility. But I can be convinced to do something more generic/consistent if people feel strongly we should do it all in one go.\n\nFallible blocks and closures\n\nBlocks and closures are very similar in how they’re written; with the exception that closures must list their arguments and may list their return type. I think our block notation is a little goofy overall, and wish it would be able to list return types too. In my opinion Swift does this better in their do statements, and like I kind of wish we would have that for blocks. But we don’t do that right now, so I think we should start by spelling closures and blocks as follows:\n// abstract definitions\nthrows {} // block notation\nthrows || {} // closure notation\nthrows || -> i32 {} // explicit return type\nasync throws || {} // mixing effects\n// concrete definitions\nthrows None {} // block notation\nthrows None || {} // closure notation\nthrows None || -> i32 {} // explicit return type\nasync throws None || {} // mixing effects\n\nPersonally I don’t love that the effects are listed here before the argument list, while in fn() {} -functions the effects are listed after the argument list (and before the arrow). And so it’s weird to me if closures don’t follow this same syntactic pattern. But I think making that more consistent is something we can probably address later.\n\nWhat probably stands out here though is that I’m opting to fully omit the try keyword. The way I see it there are three options for how can we spell things here:\n\ntry {} / try throws None {}\n\ntry throws {} / try throws None {}\n\nthrows {} / throws None {}\n\nPersonally I prefer the last one, omitting the try keyword entirely. Though I like try as a verb, it doesn’t carry any additional information that throws doesn’t already. And so I remind myself of Stroustrup’s Rule and away try goes.\n\nOperating on fallibility\n\nThere are three kinds of operations we need to support for fallibility: create,\npropagate, and consume. The latter of the two we already have in the language, so let’s start there. To “forward” the fallibility effect in code, you can use what is officially called “the try propagation expression” - though it is more commonly known as “the question mark operator”:\nlet x = foo ()?; // `?` propagates the fallibility effect\n\nTry propagation expressions can only be used in functions which themselves return an impl Try (either abstract or concrete). If we have a function which itself is infallible, or has a different Try type, we will want to consume the fallible effect. To do that we can use the match keyword. If we are working with an abstract impl Try implementation we will first need to cast it to a concrete type. But otherwise the logic is the same for both:\n// `match` consumes a concrete `Result` type\nmatch foo () {\nOk(x) => {.. }\nErr(err) => {.. }\n}\n// `match` consumes an abstract `impl Try` type\n// NOTE: this should probably become a combinator on `Try`\nlet res = match foo (). branch () {\nBreak(b) => FromResidual::from_residual(b),\nContinue(c) => Try::from_output(c),\n};\nmatch res {\nOk(x) => {.. }\nErr(err) => {.. }\n}\n\nThen finally we have the missing “create” operator which allows you to create a new impl Try type. On unstable today this is called yeet, and though I think the name is funny, I think it should probably be called throw. Here is how you would use it inside of functions:\nthrow None; // -> `return None`\nthrow Err(\" hello \") // -> `return Err(\"hello\".into())`\n\nAnd that is all three control-flow operations accounted for:?, match, and throw.\n\nNotes on effect naming\n\nThough I like the throw terminology overall, I think it could do with some tidying up. If we’re going to go with throw, I want us to really lean into it. To me using try as a noun feels really unintuitive, and so I’d much rather we use nouns and verbs based on the throw operation instead:\n\nTry → Throwable\n\ntry / try..bikeshed → throws\n\nyeet → throw\n\n? (try propagation operator) →? (rethrow operator)\n\nTo me this degree of regularity is incredibly appealing. Aside from the general-purpose match operation, all of these items seem clearly related to each other. And I think that’s really valuable! In a previous post I made a similar case for the iteration effect as well. Here is what I would change if I could (though I’m not saying we should):\n\nIntoIterator → Iterate\n\nFromIterator → Collect\n\nIterate is the name of the operation, which itself returns an iterator which can be iterated over. The async effect could probably also do with some normalization, since we’re currently mixing the terms “async”, “future”, and “poll” in ways that take time to internalize. I think some normalization of terminology here could be nice too (I’m musing, not prescribing):\n\nIntoFuture → Wait\n\nFuture → Waitable\n\nFuture::poll → Waitable::wait\n\nPoll → WaitableState\n\nI guess though if we had a “throwable” and a “waitable”, then we might also want\nan “iterable”. I do like “iterable” less than “iterator”, but personally I like\nconsistency even more. Though in the grand scheme of things: this specific name doesn’t matter too much either.\n\nMotivating reification into an abstract type\n\nI didn’t really get into it before in this post, but I think we’re currently underestimating just how valuable it is for functions to be able to choose whether they are able to return an abstract impl Try or a concrete Try type. In my 2019 error handling survey I highlighted the auto-enums crate which is able to automatically enable heterogenous types to be returned from function bodies.\n\nI see this combination of features becoming the language-native way to provide\n80% of the value of the anyhow::Error type. Instead of casting error types to a Box<dyn Error + Send + Sync + 'static>, this would cast the error to an anonymous enum Error {} with a variant per-type. That means:\n\nno allocations are involved\n\neach error variant is explicitly tracked\n\nsyntactically easy to write\n\nnatural upgrade path to explicit error types (e.g. thiserror )\n\nHere is an example of how I see that playing out. Function bar here can either\nthrow i32 or throw &'static str. The idea is that the compiler would recognize this, create an anonymous enum that can hold both types, and then return that as the error type:\nfn foo ( x: bool ) throws { // `throws i32 | &'static str`\nbar (x)?;\n}\nfn bar ( x: bool ) throws {\nif x {\nthrow 12 // `: i32`\n} else {\nthrow \" oops \" // `: &'static str`\n}\n}\n\nThe exact rules of how this would work would be a little subtle. We likely\nwouldn’t be able to just to coerce errors using.into() the way we do today. But I think in the short term, we could get away with some basic rules to hold space for this in the future. So bare fn() throws {} functions:\n\nShould return an abstract impl Try type\n\nShould not call.into as part of the? desugaring to coerce types\n\nAnd I believe that’s about it. But I’m keen to hear from Scott (T-Lang) to tell me exactly what I’ve missed and why this might not be enough.\n\nClosing words\n\nAnd that’s a complete design for a fallibility effect notation done. I know not everyone will like throw as much as I do, but my hope is to steer any discussion here at least to some regularity. Say if we want to go with yeet, I believe the right terminology should then become yeets / Yeetable /”re-yeet”. Personally I tend to care more about the system and framework we apply, than about the exact keyword we end up using.\n\nSomething that’s stood out while writing this is that the fallibility effect doesn’t seem to have a moral equivalent to IntoIterator / IntoFuture right now. I guess we have the automatic Into conversion that is applied when we use?. But that is more general-purpose, and I’m having some trouble working through the exact implications of not having a separate trait.\n\nAnyway, that’s it. I had a lot of fun writing this, and I hope it makes for an interesting read. Happy December!"}
