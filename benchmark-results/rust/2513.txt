{"title":"Bytes vs unicode codepoint handling in Python 3","author":null,"date":null,"main_content":"Yesterday there was\nan interesting discussion on Twitter\nabout Python 2 vs Python 3's handling of byte strings and unicode strings.\n\nI gave this some more thought and so you get to read\n(if you want) Walls Of Text about UTF-8, bytes and\nunicode right now, right here.\n\nOne interesting point that Andre made was that if we ignored\nall the legacy (non-UTF-8) byte encodings, the Python 2\napproach of implicitly converting strings to bytes (always\nusing UTF-8 as the byte encoding) would be better.\n\nIn my opinion (and experience), Python 3's way of dealing\nwith this is actually quite sane and The Right Thing if\nyou think about the operations you do on strings (and\nbyte strings) and how they differ in behavior.\n\nThe Python 2 way was (even if we would ignore non-UTF-8\nbyte encodings) error-prone and caused lots of headache.\n\ntl;dr: Because Python overloads len() and [] based\non the type of data, bytes vs. unicode string needs to\nbe encoded into the type system, because the programmer\nneeds to pick how they want to interpret the data.\n\nUpdate 2021-02-25: Found a link to the Rust documentation\nand how it handles strings:\nBytes and Scalar Values and Grapheme Clusters! Oh My!\n\nDifferent trade-offs -- you can't index into a String\nin Rust (because it stores everything as UTF-8 encoded\nbytes), but you can iterate over unicode codepoints with\n.chars() (because then you can anyway decode UTF-8\ncodepoints as you iterate over the string). Tradeoffs.\n\nSome assumptions\n\nThis discussion ignores that the language might have to\ndeal with legacy encodings (e.g. locale-based filesystem\nencoding). If you're running with a non-UTF-8 locale on\nyour own machine in 2021, rethink your life choices. Then\nagain, your code might need to run on some legacy hardware\nof some customer, but that could be solved via third party\nlibraries, etc...\n\nSo we're left with a clean environment, all byte-based text\nI/O is encoded with UTF-8 (this includes filenames in the\nfilesystem, file contents as well as network sockets, etc..).\nYes, this includes HTTP sockets, let's pretend that HTTP would\nhave been defined with UTF-8 as its default/fallback encoding,\nnot Latin-1 (or again, it could be solved in libraries, anyway).\nAnd let's ignore that a HTML document can specify its encoding\nin a meta tag, which means it has to be parsed as Latin-1 first\nand then re-parsed when that meta tag is encountered, and it's\nall bad anyway. That's The Web for you.\n\nAutomagic conversion\n\nSo let's assume that byte strings ( str in Py2, bytes in\nPy3) could always be auto-converted to/from unicode strings\n( unicode in Py2, str in py3) whenever needed, and the\nonly encoding used for this would be UTF-8. This would\nactually work, and solve some problems, you couldn't deal\nwith Latin-1 \"in memory\", but would have to somehow convert\nit in the interface points (local file, device I/O and\nnetwork socket I/O), but the core language and types would\nonly ever support UTF-8.\n\nThis would indeed work fine. But this assumes that the only\nthing we're ever doing with text strings is copying them as\na whole. So let's see what operations we would do on a string.\n\nSplitting on a separator\n\nThis would work fine for both UTF-8 encoded byte strings as\nwell as unicode strings. Space (ASCII 0x20, encoded as 0x20\nin UTF-8, also unicode codepoint U+0020) would have a unique\nrepresentation, and splitting it at the bytes level would\nalways work correctly (Python 3.9.1 here):\n>>> hey = 'hello wörld'.encode()\n>>> hey\nb'hello w\\xc3\\xb6rld'\n>>> hey.split()\n[b'hello', b'w\\xc3\\xb6rld']\n>>> [x.decode() for x in hey.split()]\n['hello', 'wörld']\n\nSimilarly, splitting on any other ASCII character (or 7-bit\ncodepoint that has a single-byte representation in UTF-8)\nwould work independent of whether bytes or str is used.\n\nAssuming we can split on arbitrary-length strings and not\njust a single byte, splitting on the UTF-8-encoded\nrepresentation of a string would still work fine:\n>>> 'aöböc'.split('ö')\n['a', 'b', 'c']\n>>> [x.decode() for x in 'aöböc'.encode().split('ö'.encode())]\n['a', 'b', 'c']\n\nFinding substrings\n\nThis would always work fine. If haystack and needle would\nbe a different representation, auto-converting either to\nthe type of the other would give the same result\n(continuing from above):\n>>> hey.find('wörld'.encode())\n6\n>>> hey.decode().find('wörld')\n6\n\nNote that depending on whether the UTF-8 representation or\nthe unicode string representation is used, the offset might\nbe different:\n>>> 'ö abc'.find('abc')\n2\n>>> 'ö abc'.encode().find('abc'.encode())\n3\n\nWhich brings us to the next topic...\n\nBytes, Codepoints, On-Screen Characters\n\nFrom this point, we have to think of different \"units\":\n\nBytes: Treat each byte as one \"unit\", this is useful\nfor working with binary data / C structs, etc...\n\nCodepoints: Each unicode codepoint is a \"unit\", this\nis useful if you care about \"text\" and e.g. an umlaut\n(which would be encoded as 2 bytes in UTF-8) should be a\ntreated as single character\n\nCharacter on screen: Unicode has modifiers, a zero-\nwidth joiner and other fun stuff (e.g. a 👩🏿‍🏫 \"dark skin tone\nfemale teacher\" is in some cases - depending on the font\nand operating system used - a single character on screen,\nbut is composed of 4 unicode codepoints: U+1F469, U+1F3FF,\nU+200D, U+1F3EB, which encodes into 15(!) UTF-8 bytes,\nand on another machine, this 15-byte sequence could be\nrendered as 2 characters on screen: 👩🏿 \"U+1F469, U+1F3FF\ndark skin tone woman\" and 🏫 \"U+1F3EB school\" or depending\non the font as 3 characters on screen: 👩 \"U+1F469 woman\",\n🏿 \"U+1F3FF dark skin tone\" and 🏫 \"U+1F3EB school\" -\nthe \"U+200D zero width joiner\" is not rendered on-screen,\nbut I guess if you have an old enough font renderer, it\nmight be rendered as tofu in some cases, so you would\nget 4 characters like □□□□ there)\n\nFor 7-bit data (ASCII characters), all 3 definitions are\nusually the same, a lowercase \"a\" is encoded as byte 0x61\nin UTF-8, has the\nunicode codepoint U+0061 and is one character on-screen.\n\nFor the unicode codepoints that are in Latin-1\nwhich are assigned codepoints 128-255, UTF-8 encodes those\nas 2 bytes (0xC2 0x80 for 128, 0xC3 0xBF for 255, the \"ÿ\")\nand so on. Those would, however, still be rendered as a\nsingle character per unicode codepoint on the screen\n(ignoring control characters here).\n\nThis by the way also explains why UTF-8 encoded\ndata interpreted as Latin-1 leads to well-known garbage that\nyou might have seen in shipping labels:\n>>> 'Währinger Straße'.encode('utf-8').decode('latin-1')\n'WÃ¤hringer StraÃ\\x9fe'\n\nCodepoint 228 (\"ä\", U+00E4) encodes as 0xC3 0xA4 in UTF-8,\nand 0xC3 in Latin-1 is \"Ã\" and 0xA4 is \"¤\". This is why you\nshould only ever use UTF-8, but maybe you NEED to convert it\nto Latin-1 at the point where you communicate with your\nlegacy serial label printer or whatever, but don't make your\nchoice of label printer dictate how you store data on disk\nand in memory.\n\nByte count, string length, length \"on screen\"\n\nIn Python, the len() function is overloaded to work with\nall its built-in types: You can get the length of a list,\na tuple, a dict, a bytes object or a (unicode) string with\nthe same function.\n\nDepending on the type of object, len() does different\nthings (first is a (unicode) string, second is the string\nencoded as UTF-8 byte string):\n>>> len('Währinger Straße')\n16\n>>> len('Währinger Straße'.encode())\n18\n\nSo if the type system would automatically convert bytes\nto strings, should len() always work on bytes or on\nstrings?\n\nAnd what about our good old emoji with modifiers case?\n>>> len('👩🏿‍🏫')\n4\n>>> len('👩🏿‍🏫'.encode())\n15\n\nIt prints the length as 4 unicode codepoints, which is\nnot wrong, but at the same time doesn't really correspond\nto what's rendered on-screen (depending on your terminal,\nOS and font, it might not even render as a single character,\nit does for me on macOS and the built-in Terminal.app as\nwell as on Safari when viewing this web page).\n\nThe thing is, at the point where the REPL of Python \"talks\" to the system\n(usually some pseudo TTY), it doesn't know much about the terminal and OS/font\nthat is used. There's the $TERM variable and various locale-specific\nvariables (that hopefully make Python use a UTF-8 locale, or it would all be even\nworse!), and you can make some assumptions, but if I SSH into another machine,\nuse a serial line or even inspect the console output of a program running in CI\nin a web browser (which itself might use OS-specific fonts and rendering), all\nbets are off in terms of \"how is this set of Unicode codepoints displayed to\nthe user?\".\n\nSo as far as I'm aware, we can't really answer the\n\"how many characters will this be on screen?\" question\nproperly from the language -- you can still do it if you\nuse a GUI toolkit and/or have some control over how the\ncharacters are rendered, but for command-line tools, you\ncannot generally determine how things will be displayed.\n\nString slicing / substr\n\nLet's now assume that we want to get parts of a string,\nlike \"get the first N characters\". In Python, this can\nbe accomplished with the slicing operator, and just like\nlen() it works not only on strings and bytes, but also\non lists (and tuples):\n>>> a = [1, 2, 3, 4, 5]\n>>> a[:3]\n[1, 2, 3]\n>>> a[1:3]\n[2, 3]\n>>> a[2:]\n[3, 4, 5]\n\nThe problem is, just like with len(), slicing has\ndifferent output depending on whether the string is\nUTF-8 encoded bytes or a (unicode) string:\n>>> 'brötchen'[6:]\n'en'\n>>> 'brötchen'.encode()[6:]\nb'hen'\n\nThere's no Right or Wrong variant, both make sense,\nyou just have to specify if your units are \"bytes\" or\n\"unicode codepoints\" (and again, \"characters on screen\"\nare a hard problem to solve, see above).\n\nIf your bytes are actually binary data, trying to\ninterpret them as UTF-8 and \"counting\" codepoints this way\nwill be very wrong. \"Give me the first 6 bytes\" and\n\"give me the first 6 unicode codepoints\" are two different\nrequests which will give two different results.\n\nThis gets even worse if you expect to be able to cut\nUTF-8 byte strings at any point, which does not work.\n\nFor example, back in the Python 2 days, I had a rare\nbug in my code (which elided strings) that happened\nonly in some corner cases. Consider that you want to\nlimit your string to 15 characters, and if it's longer\nthan that, elide it (I am aware that there is an\nellipsis character and it's different from three dots,\nbut we'll go with the three dots here for simplicity):\n>>> s = 'Sprachenbenützungsproblem'\n>>> if len(s) > 15:\n... s = s[:12] + '...'\n...\n>>> s\n'Sprachenbenü...'\n\nSweet! Now how would this work with bytes?\n>>> s = 'Sprachenbenützungsproblem'.encode()\n>>> if len(s) > 15:\n... s = s[:12] + '...'\n...\n>>> s.decode()\nTraceback (most recent call last):\nFile \"<stdin>\", line 1, in <module>\nUnicodeDecodeError: 'utf-8' codec can't decode byte 0xc3 in position 11: invalid continuation byte\n\nArgh, here we are -- the dreaded UnicodeDecodeError!\n\nThe reason is simple: \"ü\" gets encoded as 2 bytes:\n>>> 'ü'.encode()\nb'\\xc3\\xbc'\n\nIf you decode those two bytes back, it works just fine:\n>>> b'\\xc3\\xbc'.decode()\n'ü'\n\nHowever, if you just decode the first byte, it will\nresult in weirdness (the error is slightly different\nhere, because here the string ends and it expects\nanother byte, whereas above it tries to decode the\nfirst \".\" (in \"...\") as continuation byte, which fails):\n>>> b'\\xc3'.decode()\nTraceback (most recent call last):\nFile \"<stdin>\", line 1, in <module>\nUnicodeDecodeError: 'utf-8' codec can't decode byte 0xc3 in position 0: unexpected end of data\n\nIn any case, the point is that you can't just\narbitrarily cut off UTF-8 byte strings and expect\nto get a valid UTF-8 sequence back. Here again, you\nneed to know if you're dealing with bytes or text\ndata: It is totally okay to cut off a bytes string\nin the middle of 0xC3 and 0xBC if it's not UTF-8\nencoded data, but rather e.g. two bytes encoding\nthe numbers 195 and 188 and you just care about\nthe first number:\n>>> import struct\n>>> packed = struct.pack('<BB', 195, 188)\n>>> packed\nb'\\xc3\\xbc'\n>>> struct.unpack('<B', packed[:1])\n(195,)\n\nCollation\n\nAnother thing that is actually very very tricky is\nsorting alphabetically. How things are sorted might\ndiffer from locale to locale. This is mostly a\nlocalization \"problem\", but since strings in Python\nare usually sorted \"by codepoint\", it means if you\nsort a list of strings, you might not get what you\nexpect:\n>>> sorted(['Argentinien', 'Österreich', 'Zimbabwe', 'Oman'])\n['Argentinien', 'Oman', 'Zimbabwe', 'Österreich']\n\nThese are German-language country names, and Austria\nis \"Österreich\", which I kind of expect to be sorted\nnext to other names starting with \"O\", but \"Ö\" has\ncodepoint 214, whereas \"O\" is 79 and \"Z\" is 90. Not\nto mention that e.g. local web shops would probably\nhave 99% of their customers order to Austria, so it\nmight make sense to sort it all the way at the top.\n\nThis is a UX problem and doesn't directly relate to\nencoding, but since the technical default sorting\norder is unicode codepoints, it kinda relates to\nencoding and shows that it might make sense to have\nsome \"higher-level\" representation where e.g. comparing\ntwo strings takes care of locale-specific collation and\non-screen representation (the \"how many characters on\nthe screen\" question above).\n\nClosing remarks\n\nIt's true that all the problems above come from the\nsimple fact that functions like len() and operators\nlike the slicing operator (e.g. [:3] ) are overloaded\n(the same syntax causes different behavior depending\non the type of the object it's called on).\n\nIf you would have e.g. n_bytes() and n_codepoints()\nas functions, you could indeed treat UTF-8 encoded byte\nstrings as one and the same type, at function-calling\ntime you can decide how you want to interpret it (but\nagain, n_codepoints() called on an arbitrary bytes\nobject can fail if it's not valid UTF-8 encoding, so\nyou can never fully \"hide\" the encoding).\n\nIn a language like C where there's no overloading of\nfunctions, you basically only get \"byte strings\" and\nhave to deal with UTF-8 separately, but that's fine,\nyou can have functions that parse it as UTF-8 and\nreturn the length in codepoints. And IMHO that's a\nmuch nicer solution than wchar_t - for C.\n\nBut since Python has overloaded functions and operators\nthat work on bytes and strings (and you still need to\ndifferentiate between when you want to work on bytes\nand when you want to work on codepoints), it kind of\nrequires that the type system encodes whether some\nvalue is currently bytes or unicode \"text\".\n\nI do agree that UTF-8 should be the One True Encoding\n(if it isn't already...), but similarly I enjoy that\nPython 3 deals with it in a much more sane way than\nwhat Python 2 did given its language design choices\n(overloaded len(), index [i] and slicing [n:m] operator).\n\nAnd yes, it means you get some one-time migration pains when\ngoing from Python 2 to Python 3, but the approach in Python 3\nis much saner.\n\nAnd yes, it means you get to hate on Python if some\ntooling you depend on was still on Python 2 and now\nit breaks.\n\nLess magic, more thinking required from the developer\nwhere thinking -- as you can see above -- is actually\nneeded and cannot be abstracted away fully.\n\nIn fact, maybe it even makes sense to have a \"higher-level\"\ntype on top of unicode strings that can include features such\nas \"visible characters on screen\" for its len(), group\nthose characters for slicing (so the dark skin tone female\nteacher is considered a \"single item\" and not 4 codepoints)\nand have other higher-level things abstracted away for those\ncases where you just want to e.g. \"show the first 10 visible\ncharacters on screen\"."}
