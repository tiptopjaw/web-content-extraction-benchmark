Arni\n\nHardware-Attested Nix Builds\n\nMore confidence in the integrity of your Nix artifacts.\n\nWe have recently run what might be the first hardware-attested Nix build.\nHardware attestation provides cryptographic, independently verifiable evidence\nthat a Nix build was executed as specified. It dramatically reduces the attack\nsurface for integrity attacks on Nix builds, to the point that even full root access to e.g. garnix or cloud provider infrastructure is not alone enough to forge attestations.?\n\nIn other words, this is a big deal!\n\nWhat this all means and how we did it is the subject of this blog post. In\na following post, we'll also talk about how attestation enables us to have\nconfidential Nix builds.\n\nWhat is attestation\n\nAn attestation is a claim about the state of a software system (sometimes\ncalled the target ) in the form of evidence for that claim. The process of\ncollecting evidence through observation is called a measurement, and usually\nthis takes the form of hashes of some system state. The\nprocess of verifying or assessing the evidence is called verification.\n\nA couple of examples will make this more concrete. GitHub (as well as other CIs)\nallows workflows access to an OIDC token that identifies the bearer as a\nparticular CI run, for a particular commit in a particular repo. The measurement\nis the hashing of workflow files, looking up of commits, etc. An external\nservice (e.g. Sigstore) can then generate a signing certificate for any bearer\nof such a token, and the signing certificate will contain the identity\n(run/commit/repo) signed by the service. The CI run can now sign a statement\nsaying \"this build artifact was produced by such-and-such run/commit/repo\",\nwhich is the attestation. If you trust that GitHub and the external service did\ntheir job correctly and haven't been hacked, and that the CI runner wasn't\nhacked, you have very good reason to believe that statement. The signature\nis evidence that the services agree with that statement.\n\nThere's a lot this attestation doesn't specify: What is the hypervisor, what\nother software is running in the host, what version of the kernel is running in\nthe guest, etc. And it may be that other things, such as exactly what versions\ndependencies were downloaded during builds, is not captured in e.g. lockfiles.\nAnd all of this is relevant. It would be good to have that\ninformation also attested. But even better is if that information were not relevant\nbecause it cannot influence the artifact.\n\nAnother even more pertinent example is the signatures associated with Nix store\npaths. The assertion in this case is that a particular output was the result\nof building a particular input (derivation), and the measurement is the hash\nof the contents of the output. Whether you believe that assertion\nmay well depend on who is making it (i.e. who signed it). Nix takes care of locking\nall resolved dependencies, but all the other facts about the build are still\nrelevant, and still missing.\n\nThe conclusion we want to be able to draw from the evidence we have in\nattestations is that our built artifact is correct. What \"correct\" means,\ncomes down to the semantics of your programming language. If your\ninterpreter/compiler implement that semantics correctly, and the OS implements\nits semantics correctly, and nothing else gets in the way, it will be correct.\nBut a lot can get in the way — that's how the attacks get in.\n\nHardware attestation\n\nHardware that supports attestation helps limit those things, and to then make\ndependable claims about everything else. In particular, the hardware supports\ncreating isolated virtualized environments that aren't as easily influenced\nby their host (these are called Trusted Execution Environments, or TEE).? If you\ncreate a tiny microvm with barely anything besides a\nkernel and your program, and it can't be influenced by the host, and you can\nattest that reliably, without the stuff backing the attestation introducing\neven more things to be attested, attestation becomes very convincing evidence\nof the correctness and integrity of your artifacts.\n\nThe way hardware manufacturers achieve this is by enabling software to\ninitialize hardware-backed isolation of memory (and of CPUs and registers),\ncombined with hardware-backed measurements of the contents of that memory, which\nare then signed by a key with hardware-backed assurances that it will not be\nleaked or used inappropriately. Moreover, even tampering with the system\nphysically is made very difficult. The host process can then load a program into\nmemory, and request that the hardware initialize this isolation and measurement\nprocess and begin executing the instruction loaded into memory.\n\nWhen the program executes, it can then request that data about the memory\nmeasurements be signed together with arbitrary data. A verifier can check that\nthe memory measurements correspond to the correct program either by compiling\nthe program again (if the build is reproducible) or by having been the one\nto compile it in the first place. This report is thus\nan attestation that that program created that data. The semantics of the\ndata is clear from the program itself; for example, if a program that builds an\nartifact and then requests that the hash of that artifact be attested, the\nattestation shows that that artifact was built by the program.?\n\nAttestations don't need to be about artifacts (unless by \"artifact\" we mean something\nvery general like \"anything a program produces\", including HTTP requests,\nstdout lines, etc.). An important class of attestations are of this\nkind. Instead, attestation is used to prove that a system you are communicating\nwith is running some specific software. A big driver for developing hardware attestation was so cloud providers could\nconvincingly prove to their clients that they were running the software they\nhad been asked to.?. This type of attestation is sometimes called remote attestation and\ndistinguished from artifact-based software attestation. (The terms are terrible.)\n\nNix\n\nNix already does an excellent job at capturing the explicit build inputs of\nan artifact, and limiting network access. Hardware attestation can make this\nmuch more dependable by limiting the influence of everything else.\n\nOne way we could do hardware-attested Nix builds it to simply start a remote\nbuilder in a hardware-attested VM, and then use the hash of the tuple\n(input hash, output hash) as the extra data given to the hardware attestation.\nThis is better than existing remote builders, and is quite easy to implement\n(it's the first thing we did). But there is still a lot that can go wrong: one\nbuild might escape its sandbox, and then influence the other. Or the Nix daemon\nitself can be exploited.\n\nMuch better is to replace the sandbox that Nix builds run in with a\nhardware-attested, very minimal VM. In the sandbox, hardly anything is needed\nbesides the builder script of the derivation itself, and its dependencies.\nIn fact, you don't need Nix, and, for the most part, you don't even need network access (the exception\nare fixed-output derivations). We can then attest the tuple\n(input derivation, Merkle-tree hash of dependencies, output hash) together\nwith the VM boot measurements. (Note that we don't in the VM verify the\nattestation reports of the dependencies, leaving that to the end user, since\nthat allows for varying trust policies — important if, for example, some\nsystem is compromised.)\n\nThis is an incredibly powerful thing. If hardware-attestation lived to its full\npromise, the attestation report would be completely trustworthy proof that\nthe right thing was built given the VM (e.g., the Linux kernel, bootloader and firmware versions)\nand trust in hardware manufacturers manufacturing process and key management.?\nAnyone could accept this proof and be safe. (We also have a simple patch to\nNix to make signature verification of substitution more configurable, so that\nNix itself could start accepting such evidence, and requiring hardware-attested\nbuilds in your server or machine be made trivial. Upvote the issue!)\n\nOf course, that's too good to be true. There have been attacks on such hardware,\nand some are still exploitable. In fact, it is generally a bad idea to think\nof hardware protections as absolute, and to instead see them as deterrents much\nin the same way that you would likely think of a bank vault — and, like\na bank vault, to consider whether the cost of the attack is lower than the benefit.\nSignificantly, the extant attacks rely\nof having physical access to the machine. This is a dramatically different\nthreat model to software attacks — it's much harder in most cases to gain\nphysical access to a machine, and much more likely to result in being caught.\nIf you are using a cloud provider and your\nworry is nation-state threat actors, or deliberately the cloud provider itself,\nhowever, you should certainly be worried about this. You can, of course, combine\nhardware attestation with on-prem servers (and a couple of cameras to watch\nover them). Despite the fact that remote attestation is usually described as\nsecuring yourself against your cloud providers, I think it makes a lot of sense\nwith servers you manage yourself.\n\nBecause hardware attacks are possible, one should not trust any build with\nthe correct measurements, signed by hardware keys from Intel or AMD. Having\nphysical access to a computer can, with certain attacks, allow one to sign\nspurious measurements, so an attacker would only need physical access to one\nsuch computer. Instead, you should only accept signatures from machines in\nphysical control of entities you trust.\n\nBecause attestation reports contain information about so much that can influence\na build, they also make it easy to assess the impacts of compromises and to\neffectively recover from them. If a particular version of Linux or the firmware\nis the problem, or if a particular machine was physically accessed, you know\nexactly what builds they were responsible for.\n\nConclusion\n\nThere's of course a lot about attestation that we didn't cover or\nsimplified substantially. For hardware, details vary considerably between\nmanufacturers, and between generations; it's useful to pick one manufacturer\nand learn more about it.\n\nHardware attestation can be combined with independent rebuilds to increase\nconfidence even more in the result. In particular, rebuilding every derivation\ntwice, in different cloud providers (or once on-prem), and on AMD and Intel,\nis probably a very good idea, reducing the impact of both manufacturer-specific\nattacks and of cloud provider compromises.\n\nOne thing I didn't talk about is how evidence can be chained to support broader\nconclusions. For example, one could run an attested Nix evaluator that gives\nyou as output only the drv file corresponding to a particular Nix file (or\noutput of the flake file). This can be combined with attestations about builds\nto attest file-to-build-output mappings. These combinations can become complex;\nframeworks such as in-toto can help.\n\nIn the next blog post, we'll talk about how we can have a similarly high level\nof assurance that no program besides the intended one can read the inputs or\noutputs of a build — how we can have, in short, confidential builds.\n\nThere are some issues with per-build sandboxes that we still have to work out\n(for instance, that they take a long time to start up), and setting this system\nup still requires a lot of involvement on our part and consideration of the\nspecific use. Additionally, it's more expensive and slower than non-attested\nbuilds. So we don't expect to enable these for all customers in the near future.\nBut if you think your company might benefit from this, do reach out. A big\nbenefit of this approach is that using the attestations once someone has\nset up the build pipeline is remarkably simple.\n\nOther Posts:\n\nNov 25, 2025 Julian K.
