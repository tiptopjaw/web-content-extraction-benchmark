Every AI coding assistant—Cursor, Claude Code, Copilot, or what we’re building—follows the same pattern.\n\nThe Agent Loop\n\nIt’s a loop. You say something, the LLM responses, if it asks for a tool you run it, and the loop goes around and around.\n\nUser Enters input\n\nSend to the LLM\n\nLLM Runs the simulation, returns tokens\n3a. Includes tool call? Call the tool, go back to step #2\n\nShows the result to the user\n\nGo to step 1.\n\nThat’s it. The “AI” part is the LLM. Everything else is plumbing.\n\nThe Harness: Prompts + Tools + Model\n\nWhat we are building here is called a harness. This is a combination of a prompt, tools, and a model. Think of it as three layers working together:\n\nLayer What It Is What It Does\nModel The LLM (Claude, GPT, Gemini) The “brain”—understands intent, generates responses, decides what tools to call\nTools Functions the model can invoke The “hands”—read files, run commands, search the web\nPrompts Instructions shaping behavior Clear directions on what you want the agent to do\n\nThe model is powerful but blind—it can’t see your filesystem or run code. Tools give it capabilities. Prompts tell it how to use them wisely.\n\nThis combination is the harness. Swap the model, and the same harness behaves differently. Change the prompts, and the same model acts differently. Add tools, and new capabilities emerge.\n\nKey Concepts\n\nTool Calling — Modern LLMs don’t just output text. They can output structured requests like “call the read_file function with path /src/index.ts ”. Your code executes that function and sends the result back. The model never actually runs anything—it just asks.\n\nContext Window — The LLM’s working memory. Every message, every tool result, every instruction competes for space in a fixed-size window (typically 128K-200K tokens). When you cross that limit, things get stupid. Managing context is the central challenge of agent design.\n\nSystem Prompt — Instructions on what you want the agent to do. What sorts of responses are you looking for? Do you want it to be especially terse, or complete and have all of it’s responses fix a clear template? You tweak that here, and a coding agent will have multiple prompts for what you need to do.\n\nStreaming — Instead of waiting for the full response, you receive tokens as they’re generated. This is why you see AI assistants “typing”—it’s not theater, it’s the actual generation happening in real-time.\n\nWhat Makes a Good Agent?\n\nThe model. 99% the model.\n\nNo really\n\nYes really.\n\nOk but..\n\nThe differences are in the harness, and how cleverly you can manage context.\n\nGood context — Making sure it understands your problem, where you are with it, what needs to happen.\n\nGood tools — Safe, predictable, well-described so the model knows when to use them\n\nGood prompts — Clear instructions that guide without over-constraining\n\nGood observability — Logging and cost tracking so you know what’s happening\n\nGood architecture — Context management, subagents for complex tasks, extensibility\n\nThat’s what we’re building.\n\nWhat You’ll Understand After\n\nBy the end of this tutorial, you won’t just have working code—you’ll have mental models:\n\nAgents are loops: The core pattern is simple.
