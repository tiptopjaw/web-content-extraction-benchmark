It works beautifully—until it doesn’t. The broom splits into more brooms, each carrying more water, flooding the room. Mickey has no idea how to stop it because he never understood the magic in the first place.\n\nThis is vibe coding in 2025.\n\nThe Sorcerer’s Apprentice Problem\n\n“Vibe coding” — the practice of describing what you want and letting an AI generate the code — has exploded. Cursor, Copilot, Claude, GPT. The tooling is genuinely impressive. A junior developer can now scaffold an entire application in an afternoon.\n\nBut I’ve been watching what happens in week two.\n\nThe AI generates 500 lines of React. It works. The developer ships it. Then a bug appears. They stare at their own codebase like it’s hieroglyphics. They paste the error back into the AI. It “fixes” it by regenerating 200 lines. Something else breaks. They paste that error. The AI removes a feature that was working. They don’t notice until production.\n\nThe developer has become a passenger in their own codebase.\n\nThis isn’t a skill issue. It’s a paradigm issue. We’ve optimized for speed of generation at the cost of understanding.\n\nWhat Understanding Actually Provides\n\nWhen you understand your code, you get:\n\n1. Debugging capability. You can read a stack trace and know where to look.\n\n2. Modification confidence. You can add a feature without fear of breaking something unrelated.\n\n3. Architectural intuition. You can feel when something is getting too complex.\n\n4. Compound learning. Each project makes you better at the next one. Understanding accumulates. Prompting doesn’t.\n\nIntroducing Mentor Mode\n\nWhat if we flipped the relationship?\n\nVibe Mode: Human describes goal → AI generates code → Human accepts or rejects\n\nMentor Mode: Human describes goal → AI generates code AND teaches every concept → Human understands before moving on\n\nThe name is intentional. In Vibe Mode, the AI is a wizard performing magic. In Mentor Mode, the AI is a teacher whose success is measured by whether the student can eventually work independently.\n\nThe Mentor Mode Prompt\n\nHere’s the template I use: I'm building [PROJECT].\n\nI want to learn as I build. For this session, focus on [SPECIFIC CHUNK].\n\nPlease:\n1. Explain the concept BEFORE showing code\n2. Give me the mental model (how should I think about this?)\n3. Show code in small pieces with comments explaining each piece\n4. Explain any syntax I might not know\n5. Include \"check your understanding\" questions\n6. Give me a small practice exercise\n\nAssume I'm an intelligent adult who wants to understand,\nnot just copy-paste.\n\nWhat Mentor Mode Output Looks Like\n\nWhen I asked for help writing a parser function, instead of just getting code, I got:\n\nThe Concept:\n\n“Parsing is taking text and giving it structure. Your brain does this with English: ‘The dog bit the man’ becomes Subject → Verb → Object. We’re doing the same thing with code.”\n\nThe Mental Model:\n\n“Think of the = sign as a landmark. Everything left of it is the name, everything right is the value. Find the landmark, split there.”\n\nThen the code, with every line explained:\n\njavascript function parseLine(line) {\n// Find the position of the = sign\nconst equalsIndex = line.indexOf('=')\n\n// If no =, this isn't a valid assignment\n// indexOf returns -1 when not found\nif (equalsIndex === -1) {\nreturn null\n}\n\n// Everything before = is the name\n// slice(0, n) takes first n characters\n// trim() removes whitespace\nconst name = line.slice(0, equalsIndex).trim()\n\n// Everything after = is the value\nconst value = line.slice(equalsIndex + 1).trim()\n\nreturn { name, value }\n}\n\nThen came comprehension questions (”What does indexOf return if not found?”) and a practice exercise (”Extend this to handle labels”).\n\nThe difference: After vibe coding, I have working code. After Mentor Mode, I have working code and I understand indexOf, slice, trim, and string parsing patterns I’ll use forever.\n\n(Full example with complete output in Appendix A)\n\nThe Role of Repetition\n\nLearning requires repetition. This is neuroscience, not opinion. The first time you see slice(), you’ll forget it. The third time, it’ll stick.\n\nMentor Mode builds repetition in naturally:\n\nConcept explanation — first exposure\n\nCode with comments — see it applied\n\nSyntax breakdown — see it isolated\n\nComprehension questions — active recall\n\nPractice exercise — apply it yourself\n\nNext chunk uses it again — spaced repetition\n\nIf you’re still forgetting things, slow down. The goal isn’t speed — it’s durable understanding.\n\nWhen to Use Each\n\nWhen to use Vibe Mode:\n\nThrowaway prototype — speed matters, durability doesn’t\n\nOne-off script — you’ll never touch it again\n\nHackathon — ship now, learn never\n\nWhen to use Mentor Mode:\n\nLearning a new framework — investment compounds to future projects\n\nYour own product — you’ll maintain this for years\n\nCareer development — skills transfer, code doesn’t\n\nCode interview prep — you need to explain your code\n\nCommon Objections\n\n“This is too slow. I have deadlines.”\n\nYou have a deadline this week. But you also have a career spanning decades. Mentor Mode is slower on day 1 and faster on day 100.\n\nAlso: how much time do you spend debugging vibe-coded projects? That’s hidden slowness that doesn’t feel like slowness.\n\n“I don’t need to understand everything. That’s why we have AI.”\n\nYou don’t need to understand everything. But you need enough to know when the AI is wrong, debug when things break, and make architectural decisions.\n\n“This is just... learning.”\n\nYes. Vibe coding is the avoidance of learning dressed up as productivity. Mentor Mode is learning accelerated by AI. The insight isn’t that learning is good — it’s that we’ve built workflows that accidentally optimize against it.\n\nSigns You’re Ready to Speed Up\n\nMentor Mode isn’t forever. You’re ready to accelerate when:\n\nYou can predict what the AI will generate before it does\n\nYou spot when the AI’s solution is suboptimal\n\nYou can modify the code without looking anything up\n\nYou could explain it to someone else\n\nThe goal isn’t perpetual slowness. It’s building the foundation that makes speed sustainable.\n\nA Personal Note\n\nI’m Russell, and I came to this the hard way.\n\nSix months ago, I vibe-coded an entire project.
