{"title":"Rust’s Block Pattern","author":"John Nunley","date":"2025-12-18T00:00:00+00:00","main_content":"Here’s a little idiom that I haven’t really seen discussed anywhere, that I think makes Rust code much cleaner and more robust.\n\nI don’t know if there’s an actual name for this idiom; I’m calling it the “block pattern” for lack of a better word. I find\nmyself reaching for it frequently in code, and I think other Rust code could become cleaner if it followed this pattern.\nIf there’s an existing name for this, please let me know!\n\nThe pattern comes from blocks in Rust being valid expressions. For example, this code:\n\nlet foo = { 1 + 2 };\n\n…is equal to this code:\n\nlet foo = 1 + 2;\n\n…which is, in turn, equal to this code:\n\nlet foo = {\nlet x = 1;\nlet y = 2;\nx + y\n};\n\nSo, why does this matter?\n\nLet’s say you have a function that loads a configuration file, then sends a few HTTP requests based on that config file.\nIn order to load that config file, first you need to load the raw bytes of that file from the disk. Then you need to parse\nwhatever the format of the configuration file is. For the sake of having a complex enough program to demonstrate the value\nof this pattern, let’s say it’s JSON with comments. You would need to remove the comments first using the regex crate,\nthen parse the resulting JSON with something like serde-json.\n\nSuch a function would look like this:\n\nuse regex::{ Regex, RegexBuilder };\nuse std::{ fs, sync:: LazyLock };\n\n/// Format of the configuration file.\n#[derive(serde::Deserialize)]\nstruct Config { /*... */ }\n\n// Always make sure to cache your regexes!\nstatic STRIP_COMMENTS: LazyLock < Regex > = LazyLock:: new (|| {\nRegexBuilder:: new ( r\"//.*\" ).multi_line ( true ).build ().expect ( \"regex build failed\" )\n});\n\n/// Function to load the config and send some HTTP requests.\nfn foo ( cfg_file: & str ) -> anyhow:: Result < () > {\n// Load the raw bytes of the file.\nlet config_data = fs:: read ( cfg_file )?;\n\n// Convert to a string to the regex can work on it.\nlet config_string = String:: from_utf8 ( & config_data )?;\n\n// Strip out all comments.\nlet stripped_data = STRIP_COMMENTS.replace ( & config_string, \"\" );\n\n// Parse as JSON.\nlet config = serde_json:: from_str ( & stripped_data )?;\n\n// Do some work based on this data.\nsend_http_request ( & config.url1 )?;\nsend_http_request ( & config.url2 )?;\nsend_http_request ( & config.url3 )?;\n\nOk (())\n}\n\nThis is fairly simple, and just leverages a few Rust crates and language features to parse JSON and then do something with it.\n\nHowever, there are a few weaknesses here. In the foo function, we declare four new variables ( config_data, config_string,\nstripped_data, config ) only for only one of those variables to be used after the configuration parsing ( config ). In addition,\nlet’s say you didn’t know what this code was for going in, and you didn’t have these comments (or you had bad comments). One might\nask why you’re declaring the regular expression STRIP_COMMENTS, or why you’re loading data from a file.\n\nWhen I write code, I try to make it immediately obvious what the purpose of the code is, and why it’s written that way. This is why\nI generally avoid C’s “bottom-up” strategy for organizing code. It’s like being given a few screws and being expected to implicitly\nunderstand that it should be built into a chair. In Rust, I like that you are able to define your top-level functions first, and then\ngo down and define all the bits and pieces after.\n\nAlthough, we can do a little bit better. What if we organized the foo function like this:\n\n/// Function to load the config and send some HTTP requests.\nfn foo ( cfg_file: & str ) -> anyhow:: Result < () > {\n// Load the configuration from the file.\nlet config = {\n// Cached regular expression for stripping comments.\nstatic STRIP_COMMENTS: LazyLock < Regex > = LazyLock:: new (|| {\nRegexBuilder:: new ( r\"//.*\" ).multi_line ( true ).build ().expect ( \"regex build failed\" )\n});\n\n// Load the raw bytes of the file.\nlet raw_data = fs:: read ( cfg_file )?;\n\n// Convert to a string to the regex can work on it.\nlet data_string = String:: from_utf8 ( & raw_data )?;\n\n// Strip out all comments.\nlet stripped_data = STRIP_COMMENTS.replace ( & config_string, \"\" );\n\n// Parse as JSON.\nserde_json:: from_str ( & stripped_data )?\n};\n\n// Do some work based on this data.\nsend_http_request ( & config.url1 )?;\nsend_http_request ( & config.url2 )?;\nsend_http_request ( & config.url3 )?;\n\nOk (())\n}\n\nIn this function, we’ve moved all of the configuration-related code (parsing, loading, even the static regex) into the block.\nThis works because Rust lets you have items, statements and expressions inside of a block, hence why we were able to move everything\ninside. This pattern has three immediate advantages:\n\nThe block starts with the intent of the code ( let config =... ). We can see that we’re working to resolve some kind of\nconfiguration object right off the bat. Only then do we move into the implementation details of the code.\n\nIt reduces pollution of the namespace of both the foo function and the top-level module. Now in foo, the variable names\nconfig_data, config_string et al are no longer used. In addition to allowing these variable names to be re-used, it\nmakes this code a lot more “idiot-proof”. If someone else were to edit the foo function, they would only be able to use\nconfig. They wouldn’t be able to use the raw_data or STRIP_COMMENTS items, which are only meant to be used by the\nconfig parser.\n\nThe variables raw_data and data_string go out of scope at the end of the block, which means they are dropped, freeing\nup resources.\n\nAs an aside, all three of these advantages also come if you were to refactor the block out into its own function. However, this\npattern has two key advantages over that:\n\nThe code flow is still inline with the rest of the function. For shorter blocks, this improves reading comprehension, since it\nmeans you don’t have to go to a different part of the code to fully understand the function.\n\nIf there are a lot of variables that the block would use, it prevents needing to explicitly name those variables as parameters.\n\nThere is one more benefit that’s not exposed in the above example: erasure of mutability. Let’s say you construct some object for\nuse in a later part of the function:\n\nlet mut data = vec! [];\ndata.push ( 1 );\ndata.extend_from_slice ( & [ 4, 5, 6, 7 ]);\n\ndata.iter ().for_each (| x | println! ( \"{x}\" ));\nreturn data [ 2 ];\n\nThe issue is that data is declared as mutable, which means the rest of the function can mutate it. Since a lot of bugs come from\ndata being mutated when it isn’t supposed to be mutated, we’d like to restrict the mutability of the data to a certain area of the\nfunction. This is also possible with the block pattern:\n\nlet data = {\nlet mut data = vec! [];\ndata.push ( 1 );\ndata.extend_from_slice ( & [ 4, 5, 6, 7 ]);\ndata\n};\n\ndata.iter ().for_each (| x | println! ( \"{x}\" ));\nreturn data [ 2 ];\n\nThis effectively “closes” the mutability to a certain section of the function.\n\nClosing Thoughts\n\nI don’t know if this pattern is already well known to the Rust community. Even if it isn’t, I figure it’s still a good idea to bring it\nto people who may be inexperienced in Rust."}
