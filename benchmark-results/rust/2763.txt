{"title":"Code Kata&#x3a; Formatting Compound Strings In ColdFusion","author":"Ben Nadel","date":"2025-12-20T12:27:22+00:00","main_content":"Earlier this week, in my Big Sexy Poems ColdFusion application, I added location information to my session list. The location information includes the city, region, and country code; which I compose using conditional commas and parenthesis based on the data that I have available. This kind of conditional composition and formatting never feels elegant. As such, I wanted to explore the topic of conditional formatting as a code kata in ColdFusion.\n\nBrute Force\n\nHistorically, when implementing conditional composition and formatting of a compound string value such as name, location, address, etc, in ColdFusion, I just brute force it. Meaning, I have a series of if -blocks that conditionally build-up the value one check at a time.\n\nTo illustrate, let's use the brute force method to build up a user's name from the given prefix, first, last, and suffix parts. In the following ColdFusion code you'll see that each if -block:\n\nChecks if the current part is populated?\n\nChecks if the running value is populated?\n\nConditionally includes a delimiter.\n\nConcatenates the part with the running value.\n\nAt the end, we'll output the composed string:\n\n<cfscript>\n\nuser = {\nprefix: \"Mrs.\",\nfirstName: \"Laura\",\nlastName: \"Badonka\",\nsuffix: \"Jr.\"\n};\n\n// -- construct compound name. -- //\n\nname = \"\";\n\nif ( user.prefix.len() ) {\n\nname = user.prefix;\n\n}\n\nif ( user.firstName.len() ) {\n\nname &= name.len()\n? \" #user.firstName#\"\n: user.firstName\n;\n\n}\n\nif ( user.lastName.len() ) {\n\nname &= name.len()\n? \" #user.lastName#\"\n: user.lastName\n;\n\n}\n\nif ( user.suffix.len() ) {\n\nname &= name.len()\n? \", #user.suffix#\"\n: user.suffix\n;\n\n}\n\nwriteOutput( name );\n\n</cfscript>\n\nAs you can see, the name value is incrementally built-up with each if check. And, when we run this ColdFusion code, we get the following output:\n\nMrs. Laura Badonka, Jr.\n\nAs you can see, most of the delimiters were space ( ); but, the last delimiter, prior to the suffix is comma-space (, ). It's ugly, but it's easy to read and it works.\n\nFAIL: List Append\n\nAs I was looking at my string concatenation, it occurred to me that what I'm really doing is building up a list of values using a variety of delimiters. I wanted to see if I could use ColdFusion's native list-functions to simplify this code.\n\nIn the following CFML code, we're doing the same exact thing; only, I'm leaning on ColdFusion to deal with the conditional delimiter:\n\n<cfscript>\n\nuser = {\nprefix: \"Mrs.\",\nfirstName: \"Laura\",\nlastName: \"Badonka\",\nsuffix: \"Jr.\"\n};\n\n// -- construct compound name. -- //\n\nname = user.prefix;\n\nif ( user.firstName.len() ) {\n\nname = name.listAppend( user.firstName, \" \" ); // Space delimiter.\n\n}\n\nif ( user.lastName.len() ) {\n\nname = name.listAppend( user.lastName, \" \" ); // Space delimiter.\n\n}\n\nif ( user.suffix.len() ) {\n\nname = name.listAppend( user.suffix, \", \" ); // Comma-Space delimiter.\n\n}\n\nwriteOutput( name );\n\n</cfscript>\n\nThis time, I'm not conditionally including the delimiter - I'm just telling the ColdFusion list-functions that a delimiter exists; and, I'm letting the list-functions conditionally include the delimiter if the list already has an item within it.\n\nUnfortunately, this doesn't work. When we run this ColdFusion code, we get the following output:\n\nMrs. Laura Badonka,Jr.\n\nThe issue is subtle and I almost didn't notice it at first. If you look closely, however, you'll see that there's no space before the Jr. token. This is because ColdFusion's listAppend() function only uses the first character when a multi-character delimiter is supplied. So when I pass in comma-space (, ) as the final delimiter, the native code only uses the comma.\n\nCustom List Append Function\n\nColdFusion's listAppend() method didn't work; but I liked the concept. So my next thought was to build a ColdFusion User Defined Function (UDF) that borrowed the list concept but was tailored for a multi-character delimiter:\n\nstringAppendIf( base, token, delimiter )\n\nThis UDF is basically performing a list-append with the two values ( base and token ), but will happily include the whole delimiter regardless of how long it is:\n\n<cfscript>\n\nuser = {\nprefix: \"Mrs.\",\nfirstName: \"Laura\",\nlastName: \"Badonka\",\nsuffix: \"Jr.\"\n};\n\n// -- construct compound name. -- //\n\nname = user.prefix;\nname = stringAppendIf( name, user.firstName, \" \" ); // Space delimiter.\nname = stringAppendIf( name, user.lastName, \" \" ); // Space delimiter.\nname = stringAppendIf( name, user.suffix, \", \" ); // Comma-Space delimiter.\n\nwriteOutput( name );\n\n// ------------------------------------------------------------------------------- //\n// ------------------------------------------------------------------------------- //\n\n/**\n* I append the token to the base using the given delimiter. If the token is empty, the\n* base is returned as-is.\n*/\nprivate string function stringAppendIf(\nrequired string base,\nstring token = \"\",\nstring delimiter = \"\"\n) {\n\nif (! token.len() ) {\n\nreturn base;\n\n}\n\nif (! base.len() ) {\n\nreturn token;\n\n}\n\nreturn \"#base##delimiter##token#\";\n\n}\n\n</cfscript>\n\nWith this UDF, I don't need the if -blocks because the UDF will omit any token that's empty. And, when we run this ColdFusion code, we get the following output:\n\nMrs. Laura Badonka, Jr.\n\nThis time, space and comma-space where both included.\n\nCustom Method With Optional Formatting\n\nA user's name is relatively simple. But I want an approach that can be used for more complex data-sets. When I started to think about composing address values, I hit a snag. With an address, two issues arose:\n\nThe \"care or\" has conditional formatting (including of a c/o ).\n\nThe city and state are on the same line.\n\nThe city/state issue wasn't really a bother since I can always nest the stringAppendIf() calls. But, the c/o issue was different. Including the c/o means that I have both a conditional inclusion and a conditional rendering of the \"care of\" value.\n\nTo deal with this, I updated the stringAppendIf() to accept an optional formatted token:\n\nstringAppendIf( base, token [, formattedToken], delimiter )\n\nThis function will use the token to see if the value should be included; and, if so, it will use the formattedToken value as the composed part. In the following ColdFusion code note that in the careOf line, I'm using \"c/o #careOf#\" as the formatted token:\n\n<cfscript>\n\nuser = {\nprefix: \"Mrs.\",\nfirstName: \"Laura\",\nlastName: \"Badonka\",\nsuffix: \"Jr.\"\n};\naddress = {\nrecipient = formatName( argumentCollection = user ),\ncareOf: \"Special Investigations\",\nstreet1: \"21 Jump Street\",\nstreet2: \"Suite 28A\",\ncity: \"Metropolis\",\nstate: \"Evergreen State\",\nzipcode: \"21001\"\n};\n\nwriteOutput( \"<pre>\" & formatAddress( argumentCollection = address ) & \"</pre>\" );\n\n// ------------------------------------------------------------------------------- //\n// ------------------------------------------------------------------------------- //\n\n/**\n* I format the given name.\n*/\nprivate string function formatName(\nstring prefix = \"\",\nstring firstName = \"\",\nstring middleName = \"\",\nstring lastName = \"\",\nstring suffix = \"\"\n) {\n\nvar result = prefix;\nresult = stringAppendIf( result, firstName, \" \" );\nresult = stringAppendIf( result, middleName, \" \" );\nresult = stringAppendIf( result, lastName, \" \" );\nresult = stringAppendIf( result, suffix, \", \" );\n\nreturn result;\n\n}\n\n/**\n* I format the given address as a multi-line string value.\n*/\nprivate string function formatAddress(\nstring recipient = \"\",\nstring careOf = \"\",\nstring street1 = \"\",\nstring street2 = \"\",\nstring city = \"\",\nstring state = \"\",\nstring zipcode = \"\",\nstring linebreak = chr( 10 )\n) {\n\nvar result = recipient;\nresult = stringAppendIf( result, careOf, \"c/o #careOf#\", linebreak );\nresult = stringAppendIf( result, street1, linebreak );\nresult = stringAppendIf( result, street2, linebreak );\nresult = stringAppendIf( result, stringAppendIf( city, state, \", \" ), linebreak );\nresult = stringAppendIf( result, zipcode, linebreak );\n\nreturn result;\n\n}\n\n/**\n* I append the token (or optionally the formatted token) to the base using the given\n* delimiter. If the token is empty, the base is returned as-is.\n*/\nprivate string function stringAppendIf(\nrequired string base,\nstring token = \"\",\nstring formattedToken = \"\",\nstring delimiter\n) {\n\n// The formatted token is optional and allows the token to be given additional\n// formatting if - and only if - it's populated. If the delimiter is null, we'll\n// assume that the formatted token has been omitted and is acting as the delimiter.\nif ( isNull( delimiter ) ) {\n\n// Shift arguments over.\ndelimiter = formattedToken;\nformattedToken = token;\n\n}\n\n// We're using the token, NOT the formatted token, to determine the control flow.\nif (! token.len() ) {\n\nreturn base;\n\n}\n\nif (! base.len() ) {\n\nreturn formattedToken;\n\n}\n\nreturn \"#base##delimiter##formattedToken#\";\n\n}\n\n</cfscript>\n\nThis code is longer because I'm doing more stuff; but hopefully it's not too hard to follow. When we run this ColdFusion code, we get the following output:\n\nMrs. Laura Badonka, Jr.\nc/o Special Investigations\n21 Jump Street\nSuite 28A\nMetropolis, Evergreen State\n21001\n\nNotice that \"care of\" line is prefixed with c/o.\n\nFluent API Wrapper\n\nOne thing I really miss about the listAppend() approach is the fact that they are member methods. In my previous approach, I'm constantly having to pass-around the running value; and it looks quite noisy. In this final approach, I wanted to wrap the stringAppendIf() calls in a light-weight wrapper that can provide that fluent, implied-context API.\n\nI'm creating this fluent wrapper with a UDF, stringAppendPipeNew():\n\n<cfscript>\n\n/**\n* I create a piping version of the stringAppendIf() function.\n*/\nprivate struct function stringAppendPipeNew( string initialValue = \"\" ) {\n\nvar this = {\nvalue: initialValue,\nappendIf: function( token, formattedToken, delimiter ) {\n\nthis.value = stringAppendIf(\nbase = this.value,\ntoken = token,\nformattedToken = arguments?.formattedToken,\ndelimiter = arguments?.delimiter\n);\n\nreturn this;\n\n}\n};\n\nreturn this;\n\n}\n\n</cfscript>\n\nThis ColdFusion function returns a struct that contains a closure function. This closure function allows the appendIf() logic to refer back to itself using the closed-over this reference. And, all this function does is proxy the stringAppendIf() UDF, passing in the running this.value as the implied first argument.\n\nHere's the full code:\n\n<cfscript>\n\nuser = {\nprefix: \"Mrs.\",\nfirstName: \"Laura\",\nlastName: \"Badonka\",\nsuffix: \"Jr.\"\n};\naddress = {\nrecipient = formatName( argumentCollection = user ),\ncareOf: \"Special Investigations\",\nstreet1: \"21 Jump Street\",\nstreet2: \"Suite 28A\",\ncity: \"Metropolis\",\nstate: \"Evergreen State\",\nzipcode: \"21001\"\n};\n\nwriteOutput( \"<pre>\" & formatAddress( argumentCollection = address ) & \"</pre>\" );\n\n// ------------------------------------------------------------------------------- //\n// ------------------------------------------------------------------------------- //\n\n/**\n* I format the given name.\n*/\nprivate string function formatName(\nstring prefix = \"\",\nstring firstName = \"\",\nstring middleName = \"\",\nstring lastName = \"\",\nstring suffix = \"\"\n) {\n\nreturn stringAppendPipeNew()\n.appendIf( prefix )\n.appendIf( firstName, \" \" )\n.appendIf( middleName, \" \" )\n.appendIf( lastName, \" \" )\n.appendIf( suffix, \", \" )\n.value\n;\n\n}\n\n/**\n* I format the given address as a multiline value value.\n*/\nprivate string function formatAddress(\nstring recipient = \"\",\nstring careOf = \"\",\nstring street1 = \"\",\nstring street2 = \"\",\nstring city = \"\",\nstring state = \"\",\nstring zipcode = \"\",\nstring linebreak = chr( 10 )\n) {\n\nreturn stringAppendPipeNew()\n.appendIf( recipient, linebreak )\n.appendIf( careOf, \"c/o #careOf#\", linebreak )\n.appendIf( street1, linebreak )\n.appendIf( street2, linebreak )\n.appendIf( stringAppendIf( city, state, \", \" ), linebreak )\n.appendIf( zipcode, linebreak )\n.value\n;\n\n}\n\n/**\n* I create a piping version of the stringAppendIf() function.\n*/\nprivate struct function stringAppendPipeNew( string initialValue = \"\" ) {\n\nvar this = {\nvalue: initialValue,\nappendIf: function( token, formattedToken, delimiter ) {\n\nthis.value = stringAppendIf(\nbase = this.value,\ntoken = token,\nformattedToken = arguments?.formattedToken,\ndelimiter = arguments?.delimiter\n);\n\nreturn this;\n\n}\n};\n\nreturn this;\n\n}\n\n/**\n* I append the token (or optionally the formatted token) to the base using the given\n* delimiter. If the token is empty, the base is returned as-is.\n*/\nprivate string function stringAppendIf(\nrequired string base,\nstring token = \"\",\nstring formattedToken = \"\",\nstring delimiter\n) {\n\n// The formatted token is optional and allows the token to be given additional\n// formatting if - and only if - it's populated. If the delimiter is null, we'll\n// assume that the formatted token has been omitted and is acting as the delimiter.\nif ( isNull( delimiter ) ) {\n\n// Shift arguments over.\ndelimiter = formattedToken;\nformattedToken = token;\n\n}\n\n// We're using the token, NOT the formatted token, to determine the control flow.\nif (! token.len() ) {\n\nreturn base;\n\n}\n\nif (! base.len() ) {\n\nreturn formattedToken;\n\n}\n\nreturn \"#base##delimiter##formattedToken#\";\n\n}\n\n</cfscript>\n\nIf we run this ColdFusion code, we get the same output:\n\nMrs. Laura Badonka, Jr.\nc/o Special Investigations\n21 Jump Street\nSuite 28A\nMetropolis, Evergreen State\n21001\n\nWith the fluent-API wrapper, I feel like I almost don't even need the formatName() and formatAddress() functions anymore. The add almost no value to the fluent-API itself, I might as well just use it directly. That said, a reusable abstraction is always nice to have.\n\nIt's All Trade-Offs\n\nMy first approach (brute force) and my last approach (fluent API) are wildly different. But neither is the \"right\" approach. Each of these approaches comes with its own trade-offs. There's something to be said about the sheer readability of the brute force approach - top down, imperative awesomeness. The fluent API creates tighter, less noisy code; but, it's also harder to understand.\n\nI'm definitely curious to know what kind of formatting techniques everyone else is using.\n\nWant to use code from this post?\nCheck out the license.\n\nShort link:\n\nhttps://bennadel.com/4856"}
