{"title":"12 eBPF-Powered CLI Utilities That Every Modern Linux Sysadmin Should Master","author":"Neville Ondara","date":"2025-12-16T09:57:52+00:00","main_content":"If you’ve been a Linux admin for any amount of time, you’ve probably lived this scene: your servers behave flawlessly all week… and then the moment you sit down with a coffee, the CPU spikes, a critical service slows to a crawl, and suddenly everyone’s pinging you asking why the service or app or website feels “laggy.”\n\nYou check top. Everything looks normal.\n\nYou restart a few services. Nothing changes.\n\nYou try strace. It helps, but not without slowing everything else down.\n\nYou consider diving into tcpdump and instantly regret it. Whoc could read al that?\n\nThis is where eBPF comes in. eBPF (Extended Berkeley Packet Filter) lets you attach tiny, sandboxed programs directly to the Linux kernel, no restarts, no recompiling, no intrusive overhead. In practice, you gain X-ray vision into your system’s behavior, in real time, with almost zero performance penalty.\n\nAnd thanks to the amazing CLI tools built around eBPF, you can debug problems that were previously “impossible” to catch.\n\nBefore you see those eBPF tools, let me tell you that you need to take the kernel seriously. Here's why.\n\nKnow this before trying eBPF\n\nBecause eBPF hooks directly into the kernel, a few conditions need to be met for things to work smoothly. Otherwise, these new tools will either fail to load or behave unpredictably.\n\nKernel version matters\n\nYou must use Linux kernel version 4.1 or newer. Now, most modern distributions ship with a kernel version 4.1 and higher, but there could be some production servers still rocking the older kernel with enterprise support. It is rare but possible.\n\nI generally go with this rule: newer the kernel, the better the eBPF support.\n\nKernel must be built with the right options\n\nEven if the Linux kernel version is good enough, certain kernel flags need to be enabled. Most mainstream distros already include them, but it’s worth knowing what matters:\nCONFIG_BPF\nCONFIG_BPF_SYSCALL\nCONFIG_BPF_JIT or CONFIG_HAVE_EBPF_JIT\nCONFIG_BPF_EVENTS (for kprobes and tracepoints)\nCONFIG_IKHEADERS or access to kernel headers\n\nWithout these, you'll see errors like “failed to load BPF program” or “unknown map type”. Some eBPF tools may also silently fail to trace anything.\n\nIf your environment uses custom-compiled kernels, double-check the config.\n\nYou likely need root (or near-root) privileges\n\nMost tracing tools rely on the ability to load eBPF programs into the kernel. That typically requires root, or CAP_SYS_ADMIN, and sometimes CAP_BPF or CAP_PERFMON on newer systems.\n\n💡\n\nIf you run a tool and it exits with zero output, missing privileges is usually the culprit.\n\n🚧 Install all eBPF tools at once\n\nWhile I'll discuss more than ten eBPF tools, you don't need to install them individually. You can get them with different meta packages.\n\nI'll include the commands for Debian/Ubuntu. Use your distribution's package manager: sudo apt install -y bpfcc-tools bpftrace\nsudo apt install -y linux-tools-common linux-tools-$(uname -r)\n\nThis gives you:\nbpftrace: a high-level, DTrace-like language for on-demand kernel debugging\nBCC tools (bpfcc-tools): 50+ ready-made eBPF commands\nbpftool: for inspecting running eBPF programs ( linux-tools-common )\n\nNow let’s jump into the tools that actually save you when production is on fire.\n\n1. bpftrace: The ultimate custom debugger\n\nEver wished you could ask the kernel, “What’s going on in that function right now?” That’s bpftrace.\n\nIt’s a tiny scripting language designed for one-off investigations. Perfect for those weird bugs that strike only once an hour.\n\nI once debugged a short-lived slowdown in a connection-pooling service. CPU was fine; disk was fine. But threads were waking slowly. With bpftrace, I traced the kernel scheduler itself and caught the exact moment those wakeups spiked.\n\nHere’s the script I used to measure wakeup-to-schedule latency: sudo bpftrace -e '\ntracepoint:sched:sched_wakeup {\n@start[args->pid] = nsecs;\n}\n\ntracepoint:sched:sched_switch /@start[args->next_pid]/ {\n@lat = hist(nsecs - @start[args->next_pid]);\ndelete(@start[args->next_pid]);\n}\n\ninterval:s:1 {\nprint(@lat);\nclear(@lat);\n}\n'\n\nIt prints a live histogram of scheduler latency, pure gold when you’re hunting mysterious performance drops.\n\n2. execsnoop: Catching fleeting processes\n\nSome processes spawn and die so fast that ps never has a chance to show them.\nCron jobs, failing scripts, runaway children, execsnoop catches them all. sudo execsnoop-bpfcc\n\nWhenever I suspect a service is forking out of control or spawning aggressive, short-lived processes, this is the first tool I reach for.\n\n3. opensnoop: Permission denied demystified\n\nApplications often give you vague errors like “file not found” or “permission denied,” without telling you what file they’re actually talking about.\n\nopensnoop answers that question instantly: sudo opensnoop-bpfcc -p <PID>\n\nYou’ll see every file the process tries to open, whether it succeeds or fails, and the exact error code. It’s the fastest way to debug broken config files, missing libraries, incorrect working directories, you name it.\n\n4. tcplife: Tracing microservice sessions\n\nIn microservice environments, most TCP sessions last milliseconds. If one suddenly takes 500 ms, your whole stack feels it.\n\ntcplife shows you the duration of every TCP session as it closes: sudo tcplife-bpfcc\n\nNo packet captures. No walls of text. Just clean summaries of connection lifetimes and endpoints. It’s my go-to tool for latency spikes between internal services.\n\n5. gethostlatency: Fixing intermittent slowness\n\nHalf of the “network is slow” problems I’ve debugged were actually DNS delays.\n\ngethostlatency traces every hostname resolution: sudo gethostlatency-bpfcc\n\nIf you start seeing ~1000 ms DNS lookups, it’s usually a bad resolver in /etc/resolv.conf, a dead VPN DNS server, IPv6 fallback delays, or simply a slow DNS service.\n\nNo more guessing. The output tells you exactly which process is waiting on which hostname.\n\n6. biotop: Disk I/O task manager\n\nIf your system feels sluggish, disk I/O is often the real culprit, not CPU, not RAM.\n\nbiotop gives you a real-time, updating list of processes doing disk reads/writes: sudo biotop-bpfcc\n\nIt’s an instant way to catch noisy neighbors on your storage, from runaway databases to log services eating your disks alive.\n\n7. biolatency: The I/O performance historian\n\nbiolatency shows you exactly how long your disk takes to handle I/O, grouped into latency buckets.\n\nThe histogram is extremely handy: sudo biolatency-bpfcc 10\n\nIf your I/O starts piling up in the higher-latency buckets, that’s your sign. Your storage stack needs attention.\n\n📋\n\nIf you wonder why there is 'bio' in the above two tools, well, B is for BPF and IO is for Input/Output.\n\n8. tcpretrans: Spotting network reliability issues\n\nTCP retransmissions happen when the network drops packets. Normally? Hard to see.\ntcpretrans Shows you every one: sudo tcpretrans-bpfcc\n\nEach event shows PID, connection, when it happened, current TCP state.\n\nIf your network team is insisting everything is “fine,” tcpretrans is how you politely prove it isn’t.\n\n9. tcpconnect: Tracing outbound connections\n\nIf you’ve ever gotten a security alert about “suspicious outbound traffic,” this tool is your detective: sudo tcpconnect-bpfcc\n\nWatch outbound connections appear in real time and instantly spot anything that looks… “off.” 👀\n\nFantastic for debugging microservices, auditing unknown traffic, catching malware or confirming an app talks to the right endpoints.\n\n10. tcpaccept: Tracing inbound connections\n\nIn contrast to tcpconnect, tcpaccept traces inbound connections as they land on your system, making it perfect for debugging load balancer behavior or verifying firewall rules. sudo tcpaccept-bpfcc\n\nYou’ll see inbound connections appear in real-time, making it super easy to spot misrouted traffic or rogue services grabbing traffic they shouldn’t be. ✨\n\n11. biosnoop: I/O operations in detail\n\nWhen you need a microscope for block I/O, biosnoop is the way: sudo biosnoop-bpfcc\n\nYou get device, sector, latency, PID and size.\n\nIt’s too verbose for everyday use, but irreplaceable when debugging low-level storage weirdness.\n\n12. bpftool: Inspect and debug eBPF programs already running\n\nMost eBPF tools run tracing programs. bpftool shows you what’s already running, which is helpful when debugging a system that already has monitoring agents, security tools, or custom eBPF programs loaded. If you want to understand your system’s eBPF state, this is the tool.\n\nList All Loaded eBPF Programs: sudo bpftool prog show\n\nYou can immediately see program IDs, type ( tracing, kprobe, xdp, etc.), what it’s attached to, memory usage and when it loaded.\n\nIf a mystery program shows up? Now you know where to look for.\n\nWrapping Up\n\nWe can continue debating if sysadmin is still a viable career choice or not but one thing we all can agree on is that everyone needs to adapt and improve their skill sets, even the most seasoned sysadmins.\n\nClassic Linux tools are always there, but eBPF is the natural upgrade as it looks directly in the kernel. Once you start using these tools, it’s hard to imagine managing Linux systems without them.\n\n📋\n\nWe have an upcoming hands-on eBPF course on our other website, Linux Handbook.
