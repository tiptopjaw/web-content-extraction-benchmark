The web had proven it could handle real\napplications — Gmail, Google Maps, and a generation of startups had shown that.\nBut we were hitting the limits of how we'd been building things. jQuery and\nserver-rendered pages got you far, but as applications grew more ambitious, the\ncracks started showing.\n\nThe problem was state. When your server renders HTML and jQuery sprinkles\ninteractivity on top, keeping everything in sync becomes a nightmare. Imagine a\npage with a comment section — the server renders the initial comments, but when\na user posts a new one, you need to somehow create that same HTML structure in\nJavaScript and insert it in the right place. Now imagine that comment count\nappears in three different places on the page. And there's a notification badge.\nAnd the comment might need a reply button that does something else. Single-page\napplications — SPAs — would handle all the rendering in the browser. The server\nwould just be an API that returned JSON. JavaScript would manage the entire user\ninterface.\n\nThe first wave of frameworks had already arrived.\nBackbone.js gave you models and views with some\nstructure. Ember.js, which I used on my\nfirst major frontend project, was the most ambitious: it tried to be the Rails\nof JavaScript, with strong conventions and a complete solution for routing,\ndata, and templating.\n\nThese frameworks were a step forward, but they all struggled with the same\nfundamental problem: keeping the DOM in sync with your data was complicated.\nTwo-way data binding sounded great in theory — change the data and the UI\nupdates, change the UI and the data updates — but in practice it led to\ncascading updates that were hard to reason about. When something went wrong,\ngood luck figuring out what triggered what.\n\nThen React arrived.\n\nFacebook open-sourced React in 2013, and the initial reaction was... mixed. The\nsyntax looked weird. JSX — writing HTML inside your JavaScript — seemed like a\nstep backward. Hadn't we spent years separating concerns? Why would we mix\nmarkup and logic together?\n\nBut React had a different mental model, and once it clicked, everything else\nfelt clunky. Instead of mutating the DOM directly, you described what the UI\nshould look like for a given state. When the state changed, React figured out\nwhat needed to update. It was declarative rather than imperative — you said \"the\nbutton should be disabled when loading is true\" rather than writing code to\ndisable and enable the button at the right moments.\n\nThe virtual DOM made this efficient. React would build an in-memory\nrepresentation of your UI, compare it to the previous version, and calculate the\nminimum set of actual DOM changes needed. This sounds like an implementation\ndetail, but it freed you from thinking about DOM manipulation entirely. You just\nmanaged state and let React handle the rest.\n\nComponents were the other big idea. You'd build small, reusable pieces — a\nButton, a UserAvatar, a CommentList — and compose them into larger interfaces.\nEach component managed its own state and could be reasoned about in isolation.\nThis wasn't entirely new, but React made it practical in a way previous\nframeworks hadn't.\n\nReact took a couple of years to reach critical mass. Even modern browsers were slow to implement all the new features.\nYou couldn't just write ES6 and ship it.\n\nEnter Babel. Babel was a transpiler that converted your\nshiny new ES6 code into the old ES5 that browsers actually understood. You could\nwrite modern JavaScript today and trust the build process to make it work\neverywhere. This was transformative, but it also meant that writing JavaScript\nnow required a compilation step. The days of editing a file and refreshing the\nbrowser were over.\n\nThis is how Node.js finally conquered every developer's machine — not through\nserver-side applications, but through build tools. You might never write a Node\nbackend, but you absolutely had Node installed because that's what ran your\nbuild process.\n\nThe tooling evolved rapidly. Grunt came first — a task\nrunner where you'd configure a series of steps: transpile the JavaScript,\ncompile the Sass, minify everything, copy files around. Configuration files grew\nlong and unwieldy. Gulp came next, promising simpler\nconfiguration through code instead of JSON. You'd pipe files through a series of\ntransformations. It was better, but still complex.\n\nThen Webpack changed everything. Webpack wasn't just\na task runner — it was a module bundler that understood your dependency graph.\nIt would start at your entry point, follow all the imports, and bundle\neverything into optimized files for production. It handled JavaScript, CSS,\nimages, fonts — anything could be a module. Code splitting let you load only\nwhat you needed. Hot module replacement let you see changes instantly without\nrefreshing the page.\n\nWebpack was powerful, but the configuration was notoriously difficult. Webpack\nconfig files became a meme. You'd find yourself copying snippets from Stack\nOverflow without fully understanding what they did, adjusting settings until\nthings worked, terrified to touch it again. The joke was that every team had one\nperson who understood their Webpack config, and when that person left, the\nconfig became a haunted artifact that no one dared modify.\n\nThe npm ecosystem exploded alongside all of this. Where you once downloaded\nlibraries manually, now you'd npm install them and import what you needed. The\nconvenience was incredible — need a date library? npm install moment. Need\nutilities? npm install lodash. Need to pad a string on the left side?\nnpm install left-pad.\n\nThat last one led to one of the most absurd moments in JavaScript history. In\nMarch 2016, a developer named Azer Koçulu got into a dispute with npm over a\npackage name. In protest, he\nunpublished all of his packages,\nincluding a tiny eleven-line module called left-pad that did exactly what the\nname suggests: pad a string with characters on the left side.\n\nIt turned out thousands of packages depended on left-pad, either directly or\nthrough some chain of dependencies. When it vanished, builds broke everywhere.\nMajor projects — including React and Babel themselves — couldn't install their\ndependencies. The JavaScript ecosystem ground to a halt for a few hours until\nnpm took the unprecedented step of restoring the package against the author's\nwishes.\n\nEleven lines of code. The whole thing felt absurd and a little scary. We'd built\nan incredibly powerful ecosystem on top of a foundation of tiny packages\nmaintained by random people who could pull them at any time. The left-pad\nincident didn't change how we worked — the convenience of npm was too great —\nbut it did make us realize how fragile it all was.\n\nBy 2016, the complexity had reached a peak, and people were exhausted. A\nsatirical article titled\n\" How it feels to learn JavaScript in 2016 \"\nwent viral. It captured the bewilderment of trying to build a simple web page\nand being told you needed React, Webpack, Babel, Redux, a transpiler for the\ntranspiler, and seventeen other tools before you could write a line of code.\nJavaScript fatigue was real. The ecosystem was moving so fast that the thing you\nlearned six months ago was already outdated. Best practices changed constantly.\nIt felt impossible to keep up.\n\nAnd yet, we kept building. The tools were complex, but they were also capable.\nYou could create rich, interactive applications that would have been impossible\na few years earlier. The complexity was the cost of ambition.\n\nDocker arrived in 2013 and quietly started solving a\ncompletely different problem. Remember \"works on my machine\"? Docker promised to\nend it forever. You'd define your application's environment in a Dockerfile —\nthe operating system, the dependencies, the configuration — and Docker would\npackage it into a container that ran identically everywhere. Your laptop, your\ncolleague's laptop, the staging server, production — all the same.\n\nContainers weren't a new concept. Linux had the underlying technology for years.\nBut Docker made it accessible. You didn't need to understand cgroups and\nnamespaces; you just wrote a Dockerfile and ran docker build. The abstraction\nwas good enough that regular developers could use it without becoming\ninfrastructure experts.\n\nAdoption was bumpy at first. Docker on Mac was slow and flaky for years.\nNetworking between containers was confusing. The ecosystem was fragmented —\nDocker Compose for local development, Docker Swarm for orchestration, and then\nKubernetes emerged from Google and started winning the\norchestration war. By 2017 it was clear that containers were the future, even if\nthe details were still being sorted out.\n\nThe microservices trend rode alongside Docker. Instead of one big application,\nyou'd build many small services that communicated over the network. Each service\ncould be deployed independently, scaled independently, written in whatever\nlanguage made sense. In theory, this gave you flexibility and resilience. In\npractice, you'd often traded the complexity of a monolith for the complexity of\na distributed system. Now you needed service discovery, load balancing, circuit\nbreakers, distributed tracing. Many teams learned the hard way that\nmicroservices weren't free.\n\nThe big web applications were getting really good during this period.\nSlack launched in 2013 and somehow made work chat feel fun.\nIt was fast, searchable, and had integrations with everything. The threaded\nconversations, the emoji reactions, the GIFs — it felt like software made by\npeople who actually used it. Slack replaced email for a lot of workplace\ncommunication, for better or worse.\n\nFigma launched in 2016 and proved that even serious\ncreative tools could run in a browser. Design software had always meant big\ndesktop applications — Photoshop, Sketch, Illustrator. Figma gave you\ncollaborative design in a browser tab. Multiple people could work on the same\nfile simultaneously. No more emailing PSDs around. No more \"which version is the\nlatest?\" Figma was fast enough and capable enough that designers actually\nswitched.\n\nThese applications — Slack, Figma, Notion (which launched\nin 2016 too), and others — demonstrated what modern web development could\nachieve. They were ambitious in a way that validated all the complexity. Sure,\nyou needed Webpack and React and a build process, but look what you could build\nwith them.\n\nThe role of product manager had become standard by now. When I started in 2012,\na PM was something big companies had. By 2016, even small startups expected to\nhave product people writing tickets and prioritizing backlogs. Engineering\nmanagers emerged as a distinct role from tech leads — one focused on people and\nprocess, the other on technical decisions. The flat, informal teams of the\nearlier era were giving way to more structured organizations.\n\nScrum was everywhere, for better or worse. Standups, sprint planning,\nretrospectives, story points, velocity charts. Some teams found it genuinely\nhelpful; others went through the motions without understanding why. The\nceremonies could feel like overhead, especially when poorly implemented, but the\nunderlying ideas — short iterations, regular reflection, breaking work into\nsmall pieces — had real value.\n\nBy 2017, the dust was starting to settle. React had won the framework war, more\nor less. Webpack was painful but standard. ES6 was just how you wrote\nJavaScript. Docker was becoming normal. The ecosystem was still complex, but it\nwas stabilizing. Developers who'd stuck it out had powerful tools at their\ndisposal.\n\nThe next era would bring consolidation. TypeScript would finally make JavaScript\nfeel like a real programming language. Meta-frameworks like Next.js would tame\nsome of the complexity. And deployment would get dramatically simpler. But\nfirst, we had to survive the JavaScript renaissance — and despite the fatigue,\nmost of us came out the other side more capable than we'd ever been.\n\nThe TypeScript Era\n\n\"Types are good, actually\"\n\nAfter the chaos of the JavaScript renaissance, something unexpected happened:\nthings calmed down. Not completely — this is still web development — but the\nfrantic churn of new frameworks and tools slowed to something more manageable.\nThe ecosystem started to mature. And quietly,\nTypeScript changed how we thought about\nJavaScript entirely.\n\nTypeScript had been around since 2012, created by Microsoft to bring static\ntyping to JavaScript. For years, most developers ignored it. JavaScript was\ndynamic and that was fine. Types were for Java developers who liked writing\nboilerplate. Adding a compilation step just to get type checking seemed like\noverkill for a scripting language.\n\nBut as JavaScript applications grew larger, the problems with dynamic typing\nbecame harder to ignore. You'd refactor a function, change its parameters, and\nspend the next hour tracking down all the places that called it with the old\nsignature. You'd read code written six months ago and have no idea what shape of\nobject a function expected. You'd deploy to production and discover a typo —\nuser.nmae instead of user.name — that a type checker would have caught\ninstantly.\n\nTypeScript adoption started accelerating around 2017-2018. Once you tried it on\na real project, it was hard to go back. The autocomplete alone was worth it —\nyour editor actually understood your code and could suggest what came next.\nRefactoring became safe. Interfaces documented your data structures in a way\nthat couldn't drift out of sync with the code.\n\nThe turning point was when the big frameworks embraced it. Angular had been\nTypeScript-first from the start, but Angular was also verbose and enterprise-y.\nWhen React's type definitions matured and Vue 3 was rewritten in TypeScript, it\nbecame clear this wasn't just a niche preference. By 2020, starting a new\nproject in plain JavaScript felt almost irresponsible.\n\nTypeScript also changed who could contribute to a codebase. In a dynamically\ntyped project, you needed to hold a lot of context in your head — or dig through\ncode to understand what types things were. TypeScript made codebases more\napproachable. A new team member could read the interfaces and understand the\ndomain model without archaeology. The types were documentation that the compiler\nkept honest.\n\nThe tooling matured around the same time.\nVS Code, which Microsoft had released in 2015,\nbecame the dominant editor. It was fast, free, and had TypeScript support baked\nin from the start. The combination of VS Code and TypeScript created a\ndevelopment experience that felt genuinely good — intelligent autocomplete,\ninline errors, refactoring tools that actually worked.\nSublime Text and\nAtom faded.\nEven vim devotees started grudgingly admitting that VS Code was pretty nice.\n\nWhile TypeScript was bringing sanity to the language, another layer of\nabstraction was emerging on top of React. The problem was that React itself was\njust a UI library. It told you how to build components, but it didn't have\nopinions about routing, data fetching, server rendering, or code splitting.\nEvery React project had to make these decisions from scratch, and every team\nmade them differently.\n\nNext.js, created by Vercel, stepped\nin to fill this gap. It was a meta-framework — a framework built on top of React\nthat made the common decisions for you. File-based routing: drop a file in the\npages folder and it becomes a route. Server-side rendering: works out of the\nbox. API routes: add a file to the api folder and you have a backend endpoint.\nCode splitting: automatic.\n\nNext.js wasn't the only option. Nuxt did the same for Vue.\nRemix, created by the React Router team, emerged later\nwith a different philosophy that leaned harder on web standards and progressive\nenhancement. SvelteKit gave the\nSvelte community their answer.\nGatsby was popular for static sites. But Next.js\nbecame the default choice for React projects, to the point where \"React app\" and\n\"Next.js app\" became almost synonymous.\n\nThis consolidation was a relief after the fragmentation of the previous era. You\ncould start a new project without spending a week on tooling decisions. The\nmeta-frameworks handled webpack configuration, Babel setup, and all the other\ninfrastructure that used to require careful assembly.\n\nDeployment was getting dramatically simpler too. Vercel and\nNetlify had been around for a few years, but they\nreally hit their stride in this era. The pitch was simple: connect your GitHub\nrepo, and every push triggers a deployment. Pull requests get preview\ndeployments with unique URLs. Production deploys happen automatically when you\nmerge to main. No servers to manage, no CI/CD pipelines to configure, no Docker\nimages to build.\n\nThis wasn't just convenience — it changed workflows. Product managers and\ndesigners could preview changes before they merged. QA could test feature\nbranches in realistic environments. The feedback loop tightened dramatically.\nAnd the free tiers were generous enough that you could run real projects without\npaying anything.\n\nRailway and Render emerged as\nalternatives that handled more traditional backend applications. Heroku, the\npioneer of push-to-deploy, started feeling dated — its free tier got worse, its\ninterface felt neglected, and the DX that had once been revolutionary now seemed\nmerely adequate. The new platforms were eating its lunch.\n\nServerless was the other deployment story of this era.\nAWS Lambda had launched back in 2014, but it\ntook years for the ecosystem to mature. The idea was appealing: instead of\nrunning a server 24/7 waiting for requests, you'd write functions that spun up\non demand, ran for a few seconds, and disappeared. You paid only for what you\nused. Scaling was automatic — if a million requests came in, a million functions\nwould spin up to handle them.\n\nIn practice, serverless had rough edges. Cold starts meant the first request\nafter idle time was slow. State was hard — functions were ephemeral, so you\ncouldn't keep anything in memory between requests. Debugging was painful. The\nlocal development story was poor. Vendor lock-in was real. But for certain use\ncases — APIs with spiky traffic, scheduled jobs, webhook handlers — serverless\nmade a lot of sense. Cloudflare Workers\npushed the model further, running functions at the edge, close to users, with\neven faster cold starts.\n\nCSS was having its own quiet revolution. For years, the answer to CSS complexity\nwas preprocessors — Sass or\nLess — that added variables, nesting, and mixins. Then\ncame CSS-in-JS, where you'd write styles in your JavaScript files, colocated\nwith components. Styled-components and\nEmotion were popular. It solved some real problems around\nscoping and dynamic styles, but it also felt heavyweight. You were shipping a\nruntime just to apply CSS.\n\nTailwind CSS arrived in 2017, but it reached\nmainstream adoption around 2020. The premise was counterintuitive: instead of\nwriting semantic class names and then defining styles separately, you'd use\nutility classes directly in your HTML. A button might have the classes\nbg-blue-500 text-white px-4 py-2 rounded hover:bg-blue-600. Your markup looked\ncluttered with all these classes, and everyone's first reaction was visceral\ndiscomfort.\n\nBut once you tried it, something clicked. You stopped context-switching between\nHTML and CSS files. You stopped inventing class names. You stopped worrying\nabout the cascade and specificity. The utility classes were constrained enough\nthat designs stayed consistent. And because you never wrote custom CSS, your\nstylesheets stayed tiny. The approach wasn't for everyone — designers who loved\ncrafting CSS hated it, and it felt wrong if you'd internalized the separation of\nconcerns gospel — but for developers who just wanted to style things and move\non, Tailwind was a revelation.\n\nGraphQL had its peak during this era. Facebook had\nreleased it in 2015 as an alternative to REST APIs. Instead of hitting multiple\nendpoints and getting back whatever the server decided to send, you'd write a\nquery specifying exactly what data you wanted. The server would return exactly\nthat, no more, no less. No overfetching, no underfetching. Frontend and backend\ncould evolve independently, connected by a typed schema.\n\nThe developer experience was genuinely good. Tools like\nApollo Client managed caching and\nstate. GraphiQL let you explore APIs\ninteractively. The typed schema meant you could generate TypeScript types\nautomatically. For complex applications with lots of nested data, GraphQL was\nelegant.\n\nBut GraphQL also brought complexity. You needed a GraphQL server layer. Caching\nwas harder than REST. N+1 query problems were easy to create. For simple CRUD\napplications, it was overkill. By the end of this era, the hype had cooled.\nGraphQL remained popular for complex applications and public APIs, but many\nteams quietly stuck with REST or moved to simpler alternatives like\ntRPC, which gave you type safety without the ceremony.\n\nKubernetes won the container orchestration war, for better or worse. Docker had\nsolved packaging applications into containers, but running containers in\nproduction at scale needed something more. For a while, Docker Swarm,\nKubernetes, Mesos, and others competed. By 2018,\nKubernetes was the clear winner. Google had designed it based on their internal\nsystems, donated it to the\nCloud Native Computing Foundation, and it became the\nstandard way to run containers in production.\n\nKubernetes was powerful and flexible, but it was also complex. The learning\ncurve was steep. YAML configuration files sprawled across your repository. You\nneeded to understand pods, services, deployments, ingresses, config maps,\nsecrets, and a dozen other abstractions just to deploy a simple app. Managed\nKubernetes services like GKE,\nEKS, and\nAKS helped, but\nthere was still a lot to learn.\n\nFor many teams, Kubernetes was overkill. If you weren't running dozens of\nservices at scale, the operational overhead didn't justify the benefits. The new\nPaaS platforms — Vercel, Render, Railway — were partly a reaction to this.\nThey'd handle the infrastructure complexity for you. You didn't need to know\nabout Kubernetes; you just pushed code. The abstraction level kept rising.\n\nThen COVID happened.\n\nIn March 2020, the world went remote almost overnight. Software teams that had\nbeen office-based suddenly had to figure out how to work distributed. The tools\nexisted — Slack, Zoom, GitHub — but the workflows hadn't\nbeen tested at this scale.\n\nFor web development specifically, the impact was a boom. Everyone needed digital\neverything. Companies that had planned to \"eventually\" build web applications\nneeded them now. E-commerce exploded. Collaboration tools exploded. Anything\nthat could move online did. Developer tools companies raised huge funding\nrounds. Vercel became a unicorn. The demand for software far exceeded the supply\nof people who could build it.\n\nRemote work also changed team dynamics. Asynchronous communication became more\nimportant. Documentation mattered more when you couldn't tap someone on the\nshoulder. Pull request descriptions got more thorough. The tools we'd been\nbuilding all along — GitHub's code review, Notion for docs, Slack for chat —\nturned out to be exactly what distributed teams needed.\n\nThe organizational side of software development kept maturing. Engineering\nladders became standard — clear paths from junior to senior to staff to\nprincipal engineer. The IC (individual contributor) track gained legitimacy as\nan alternative to management. You could become a staff engineer without managing\npeople, recognized for technical leadership rather than people leadership.\n\nEngineering managers and product managers were now expected at any company above\na certain size. The two roles had differentiated clearly: PMs owned what to\nbuild and why, EMs owned how the team worked and people development, tech leads\nowned technical decisions. The flat, informal structures of the early startup\nera persisted at very early-stage companies, but they were the exception now.\n\nDeveloper experience — DX — emerged as a discipline. Companies realized that\nmaking developers productive was worth investing in. Internal platform teams\nbuilt tools and abstractions so product engineers didn't have to think about\ninfrastructure. Good documentation, smooth onboarding, fast CI pipelines — these\nthings started getting attention and resources.\n\nBy 2022, the web development landscape looked remarkably different from five\nyears earlier. TypeScript was the default. Next.js dominated React projects.\nDeployment was trivially easy. Tailwind had won over skeptics. The tooling was\nmature and capable. Yes, there was still complexity — there would always be\ncomplexity — but it felt manageable. The JavaScript fatigue of 2016 had given\nway to something like contentment.\n\nAnd then everything changed again. A company called OpenAI\nreleased something called ChatGPT.\n\nThe AI Moment\n\n\"Wait, I can just ask it to write the code?\"\n\nEvery era in this retrospective had its defining moment — the thing that made\nyou realize the rules had changed. Gmail showing us what AJAX could do. React\nchanging how we thought about UI. TypeScript making JavaScript feel grown-up.\nBut nothing prepared us for November 30, 2022.\n\nThat's when OpenAI released ChatGPT.\n\nWithin a week, everyone was talking about it. Within a month, it had a hundred\nmillion users. The interface was almost comically simple — a text box where you\ntyped questions and got answers — but what came out of it was unlike anything\nwe'd seen. You could ask it to explain quantum physics, write a poem in the\nstyle of Shakespeare, or debug your Python code. And it would just... do it. Not\nperfectly, not always correctly, but well enough that it felt like magic.\n\nDevelopers started experimenting immediately. Could it write a React component?\nYes, pretty well actually. Could it explain why your code wasn't working? Often\nbetter than Stack Overflow. Could it convert code from one language to another?\nSurprisingly competently. The things that used to require hours of documentation\nreading or forum searching now took seconds.\n\nGitHub Copilot had actually arrived\nearlier, in June 2022, but ChatGPT was what made AI feel real to most people.\nCopilot worked differently — it lived inside your editor, watching what you\ntyped and suggesting completions. Start writing a function name and it would\nguess the implementation. Write a comment describing what you wanted and it\nwould generate the code below. It was autocomplete on steroids.\n\nThe experience was strange at first. You'd start typing and this ghost text\nwould appear, offering to finish your thought. Sometimes it was exactly what you\nwanted. Sometimes it was subtly wrong. Sometimes it hallucinated APIs that\ndidn't exist. You had to develop a new skill: evaluating AI suggestions quickly,\naccepting the good ones, tweaking the close ones, rejecting the nonsense.\n\nBut when it worked, it worked remarkably well. Boilerplate code that used to\ntake ten minutes took ten seconds. Unfamiliar libraries became approachable\nbecause Copilot knew their APIs even if you didn't. The tedious parts of\nprogramming — writing tests, handling edge cases, converting data between\nformats — got faster. You could stay in flow instead of constantly switching to\ndocumentation.\n\nCursor took this further. It launched in 2023 as a fork of\nVS Code rebuilt around AI. The difference was integration — instead of AI being\na feature bolted onto your editor, the entire experience was designed around it.\nYou could select code and ask Cursor to refactor it. You could describe a\nfeature in plain English and watch it write across multiple files. You could\nhave a conversation with your codebase.\n\nThe implications were disorienting. Skills that had taken years to develop —\nknowing APIs, remembering syntax, understanding patterns — suddenly mattered\nless. A junior developer with good prompting skills could produce code that\nlooked like it came from someone with a decade of experience. What did seniority\neven mean when the AI knew more syntax than anyone?\n\nThe answer, it turned out, was that seniority still mattered — just differently.\nThe AI could write code, but it couldn't tell you what code to write. It didn't\nunderstand your business requirements, your users, your technical constraints.\nIt didn't know which shortcuts would come back to haunt you and which were fine.\nIt would confidently generate solutions to the wrong problem if you weren't\ncareful. The job shifted from writing code to directing code — knowing what to\nask for, evaluating what came back, understanding the system well enough to spot\nwhen the AI was leading you astray.\n\nThe discourse was polarized, predictably. Some people declared that programming\nwas dead, that we'd all be replaced by AI within two years. Others dismissed the\nwhole thing as hype, insisting that AI-generated code was buggy garbage that\nreal engineers would never use. The truth was somewhere in between and more\ninteresting than either extreme. AI didn't replace developers, but developers\nwho used AI became noticeably more productive. You could attempt projects that\nwould have been too tedious before. You could learn new domains faster. You\ncould build more.\n\nI found myself building things I wouldn't have attempted a few years ago. Side\nprojects that would have taken months became weekend experiments. Areas where I\nhad no expertise — machine learning, game development, unfamiliar frameworks —\nbecame accessible because I could have a conversation with something that knew\nmore than I did. The barrier to trying new things dropped dramatically.\n\nThis was the democratization story continuing, but accelerated. The web had\nalways been about lowering barriers — from needing a server to needing just FTP,\nfrom needing C to needing just PHP, from needing operations knowledge to needing\njust git push. AI was the next step: from needing to know how to code to needing\nto know what you wanted to build.\n\nNon-developers started building things. Product managers prototyping features.\nDesigners implementing their own designs. Domain experts creating tools for\ntheir specific needs. The line between \"technical\" and \"non-technical\" blurred.\nYou still needed developers for anything complex or production-grade, but the\ndefinition of what required a developer was shrinking.\n\nThe indie hacker movement flourished. Solo developers shipping products that\ncompeted with funded startups. People building and launching MVPs in days\ninstead of months. Twitter was full of people documenting\ntheir journeys building SaaS products alone. Some of it was survivorship bias\nand hustle culture, but some of it was genuinely new. The tools had become\npowerful enough that one person really could build something meaningful.\n\nWhile AI was the main story, the ecosystem kept evolving in other ways.\nReact Server Components,\nafter years of development, started seeing real adoption. The idea was to render\ncomponents on the server by default, sending only the HTML to the browser, and\nexplicitly marking components as client-side when they needed interactivity. It\nwas a philosophical shift — instead of shipping JavaScript and rendering on the\nclient, you'd do as much as possible on the server.\n\nhtmx emerged as a more radical alternative. Its premise was\nthat you didn't need a JavaScript framework at all for most applications.\nInstead, you'd add special attributes to your HTML that let elements make HTTP\nrequests and update parts of the page. Your server returned HTML fragments, not\nJSON. No build step, no virtual DOM, no bundle size concerns. For certain types\nof applications — content sites, CRUD apps, anything that didn't need rich\nclient-side interactivity — htmx was delightfully simple.\n\nThis \"use the platform\" sentiment extended beyond htmx. Web standards had gotten\ngenuinely good while we weren't paying attention. CSS had flexbox, grid,\ncontainer queries, cascade layers, and native nesting. You could do most of what\nSass offered in plain CSS now. JavaScript had top-level await, private class\nfields, and all the features we used to need Babel for. The gap between what we\nwanted and what browsers provided had narrowed enough that sometimes you didn't\nneed the abstraction layer.\n\nBun launched in 2022 as an alternative to Node.js, promising\nto be dramatically faster. It was a new JavaScript runtime written in Zig, with\na built-in bundler, test runner, and package manager. The benchmarks were\nimpressive, and the developer experience was good. Whether it would actually\nreplace Node remained to be seen, but it represented the kind of ambitious\nrethinking that keeps ecosystems healthy.\n\nThe job market went through turbulence. The hiring frenzy of 2021-2022, when\nevery tech company was growing aggressively, reversed sharply in late 2022\nand 2023. Layoffs swept through the industry. Companies that had hired thousands\nsuddenly let thousands go. Interest rates rose, funding dried up, and the\ngrowth-at-all-costs mentality shifted to profitability.\n\nFor developers, this was jarring. The market that had felt infinitely abundant —\nwhere recruiters flooded your inbox and companies competed to hire you —\ntightened considerably. Junior roles became harder to find. The question of\nwhether AI would reduce demand for developers mixed uncomfortably with the\nreality that demand was already reduced for macroeconomic reasons.\n\nBy 2024 and 2025, things had stabilized somewhat. The layoffs slowed. Companies\nwere still hiring, just more selectively. AI hadn't replaced developers, but it\nhad changed expectations. You were expected to be productive with AI tools. Not\nusing them started to look like stubbornly refusing to use an IDE.\n\nThe AI capabilities kept improving. Claude,\nGPT-4, and other models got better at understanding\ncontext, writing longer coherent passages of code, and reasoning through\nproblems. Coding agents — AI that could not just suggest code but actually\nexecute it, run tests, and iterate — moved from research demos to usable tools.\nThe pace of improvement showed no signs of slowing.\n\nLooking at web development in late 2025, I feel remarkably optimistic. The tools\nhave never been this good. You can go from idea to deployed application faster\nthan ever. The platforms are mature, the frameworks are capable, the deployment\nis trivial. AI handles the tedious parts and helps you learn the parts you don't\nknow yet.\n\nSwitching computers used to be a multi-day ordeal — reinstalling software,\ncopying files, reconfiguring everything. Now I install a handful of apps and log\nin to services. Everything lives in the cloud. My code is on GitHub, my\ndocuments are in Notion, my designs are in Figma. The computer is almost just a\nterminal into the real environment that exists online.\n\nThirty years ago, my dad set up a Unix server so I could upload HTML files about\nwhatever interested me that week. The tools were primitive, the sites were ugly,\nand everything was held together with <br> tags and good intentions. But the\ncore promise was there: anyone could build and share on the web.\n\nThat promise has only grown stronger. The barriers keep falling. What once\nrequired a team now takes a person. What once took weeks now takes hours. The\nweb remains the most accessible platform for creation that humans have ever\nbuilt.\n\nI don't know exactly where this goes next. AI is moving fast enough that\npredictions feel foolish. But the trajectory of my entire career has been toward\nmore people being able to build more things more easily. I don't see any reason\nthat stops.\n\nIt's a fantastic time to be building for the web."}
