Many new AI gateways can handle basic GenAI use cases, but most have never been tested under the demanding, high-throughput conditions enterprises require. Additionally, once your organization begins working with natural language processing (NLP), embeddings, and unstructured data like images, performance isn’t optional — it’s critical.\n\nKong stands apart by extending its battle-tested API infrastructure to power modern AI workloads.\n\nKong AI Gateway is built on the same highly performant runtime — Kong Gateway — that already supports mission-critical APIs across the world’s largest organizations.\n\nIn this blog post, we'll share performance benchmark results comparing Kong AI Gateway to newer offerings like Portkey and LiteLLM. We’ll walk through the test setup, execution, and what the data reveals about each offering’s performance at scale.\n\nBenchmark architecture\n\nThe benchmark tests were executed in AWS. The server infrastructure ran on an Amazon Elastic Kubernetes Service (EKS) cluster, 1.32. In order to have better control over the AI Gateways and remove the native LLM infrastructure variables, such as latency time and throughput, we mocked an LLM with WireMock to expose OpenAI-based endpoints. WireMock is an open-source tool used to simulate API responses.\n\nThe AI Gateways were exposed to the consumers through a Network Load Balancer (NLB) to protect them from external interference. Similarly, the mocking LLM was exposed with an NLB. In order to not compete for the same hardware (HW) resources, the AI Gateways and WireMock ran in their own EKS Nodes based on the c5.4xlarge instance type with 16 vCPUs and 32GiB of memory.\n\nLastly, K6 played the load generator role, running on an EC2 instance deployed in the same VPC as the EKS Cluster.\n\nHere's a diagram illustrating the benchmark architecture:\n\nThe following versions were used for the benchmark tests:\n\nKong Gateway 3.10\n\nPortkey OSS 1.9.19\n\nLiteLLM 1.63.7\n\nThis Kong's GitHub repository presents all instructions and artifacts necessary for creating the same environment used in the performance tests.\n\nBaseline\n\nThe tests took the WireMock results as the baseline. WireMock was deployed in a single Pod with a configuration to consume all resources available in the EKS Node.\n\nBy submitting the WireMock EKS node to the same throughput, we were able to achieve the following results:\n\n29005.51 RPS\n\nP95 of 24.07ms\n\nP99 of 30.35ms\n\nHere's an AWS CloudWatch picture, during a 10-minute run, showing CPU utilization. The orange line is K6's EC2 as the blue line is WireMock EKS Node. As you can see, we were able to stress WireMock while K6 still had some resources available. That's important to make sure that WireMock does not become a bottleneck during the actual AI gateway runs.\n\nAI gateway consumption and policies\n\nThe main goal was to compare the AI gateways in similar scenarios consuming the same resources. All deployments had the resource configuration with an upper limit of 12 CPUs.\n\nFor the comparison tests, all gateways were configured to play the proxy role only, meaning no policies, like caching or API key-based authentication mechanisms, were configured. As we start adding policies, we expect to see new and different results.\n\nFor all benchmark tests, K6 ran for 3 minutes and always injected the same throughput to all AI Gateways with 400 VUs (each VU representing a consumer) sending requests with 1000 prompt tokens.\n\nRequests per second\n\nThe benchmark results presented the following results for RPS (requests per second).\n\nObservations:\n\nFor Kong, we had two deployments:\n\nSelf-managed Kong Enterprise, with Control Plane and Data Plane separation, fully deployed in EKS.\n\nKong Data Plane deployed in EKS connected to a Konnect Control Plane running in Kong Cloud.\n\nKong Konnect Data Planes showed a performance increase of over 200% when compared to PortKey, and over 800% against LiteLLM.\n\np95 and p99\n\nHere are the p95 latency times:\n\nThe p99 latency times had proportional similarities:\n\nEKS nodes consumption\n\nFor a visual representation of the gateways, here are the CloudWatch screenshots for each one of them during a 10-minute run:\n\nOrange line: K6's EC2\n\nBlue line: WireMock EKS Node\n\nGreen line: AI Gateway EKS Node\n\nThe most notable aspect is that Kong was not just able to consume the HW resources provided, but was also capable of stressing the LLM layer implemented by WireMock.
