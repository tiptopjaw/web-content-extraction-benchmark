{"title":"Selective Applicative Functors","author":"@MonoidMusician","date":null,"main_content":"I havenʼt seen a good accounting of the essence of selective applicative functors.\n\nTheyʼve been longing for a better description, to help explain what should be allowed and what should be disallowed, beyond “hey, here is a function select that seems to do useful things and enable us to write interesting code”.\n\nSelective applicative functors were originally proposed in 2019 in the paper Selective Applicative Functors, by Andrey Mokhov, Georgy Lukyanov, Simon Marlow, and Jeremie Dimino, with this typeclass definition:\n\nThe paper mentions a branch combinator derived from ( <*? ) = select:\n\nbranch:: Selective f => f ( Either a b) -> f (a -> c) -> f (b -> c) -> f c\nbranch x l r = fmap ( fmap Left ) x <*? fmap ( fmap Right ) l <*? r\n\nBut the story has stopped short after select because the familiar tools of theoretical analysis failed to apply to select: there was no account for select in terms of monoidal tensors, and even branch did not meaningfully compose with itself, which means that there was no obvious way that the laws related to more familiar algebraic and categorical structures like monoids (and, it turns out, near-semirings).\n\nHowever, the final answer for what selective applicative functors want to be is really cool.\nWe just have to work a bit harder to see it: we have to consider arrows (composable profunctors) instead of plain functors.\n\nSelective applicative functors want to encode exclusive determined choice.\nThey can choose between a finite number of predetermined case branches based upon previous results during evaluation. select itself has an implicit pure branch with no side-effects, where branch has two branches.\n\nOnce you see it, it makes so much sense from a programming language perspective: it has the shape of an AST for a programming language.\nWe can even go farther and relate it to the other typeclass for control flow, Alternative, which provides <|> for nondeterministic choice.\n\nOverview\n\nHere it is, here is the essence of selective applicatives!\n\nLetʼs start from the familiar territory of monads.\nMonads encode the essence of dynamic control flow: because >>= allows binding any function as a continuation, an action in a monad can construct an arbitrary action to execute next, dynamically.\nThis also forces a clear direction to evaluation: the left effects have to happen before the right effects, because the result of the left action is used to determine the whole action on the right.\nThis also disallows static analysis: the constant functor Const is not an interesting monad. The reason it does not have a Monad instance at all is because it would not be compatible with the much more compelling Applicative instance.\n\nThis is in contrast to applicative functors, which have no “arrow of time”: their structure can be dualized to run effects in reverse because it has no control flow required by the interface. Of course particular applicative functors can have interesting control flow themselves, via combinators other than <*>.\nAnd their static analysis, given by Monoid m => Applicative ( Const m), uses monoids to accumulate information about each action that was sequenced by <*>.\n\nSelective applicative functors sit in the sweet spot of expressing finite control flow: they allow (but do not require) that an implementation choose between a finite number of branches of otherwise static control flow.\n\nThis means that we need something that restricts >>= to finite-case functions to encode exclusive determined choice.\n\nThis will be defined as a CaseTree data type below.\n\nIt turns out that the most natural setting is to consider arrows instead of functors.\nThe functor can be recovered as an arrow out of the unit type (spelled () in Haskell and Unit in PureScript).\n\nAny approach that does not consider arrows seems doomed to failure, mainly for the reason that we want to work with coproducts ( either:: (x -> r) -> (y -> r) -> ( Either x y -> r) ) which involves the domain in an essential manner, while products ( tuple:: (i -> x) -> (i -> y) -> (i -> Tuple x y) ) stay in the codomain and thus are more amenable to restricting to actions on functors.\n\nThe concept that we need, of “finite-case functions”, is a bit tricky to formulate (especially in programming data types: it needs existential types), which I believe is part of why it has been missed.\n\nHowever, once we focus on the arrows instead of the applicative actions in isolation, it can all pop into place and it pays off in revealing details of the structure we were really after.\n\nFor example, we learn that selective applicative functors allow static analysis via near-semirings by using a constant functor.\n(That is, we define static analysis to be an interpretation into a constant functor, and we learn that near-semirings are the algebraic structure we need to make it work.)\n\nThe common formulation of select:: f ( Either u v) -> f (u -> v) -> f v does not capture this story.\nThis select is a specialization to where one branch is a constant identity branch, and the general formulation cannot be recovered perfectly.\nThat is, select allows for finite choice / optional effects, but it cannot encode which branches of control flow are exclusive.\n\nWe can squeeze it all into a single data type for a free arrow (a strong profunctor and a category) like this, subject to some laws.\n\ndata ControlFlow f i r where\nAction:: (f (i -> r)) -> ControlFlow f i r\nPure:: (i -> r) -> ControlFlow f i r\n-- `j` is existential here\nSequencing:: ControlFlow f i j -> ControlFlow f j r -> ControlFlow f i r\nAbsurd:: (i -> Void ) -> ControlFlow f i r\n-- `x` and `y` are existential here\nCaseBranch::\n(i -> Either x y) ->\nControlFlow f x r ->\nControlFlow f y r ->\nControlFlow f i r\n\nWe then recover the free selective applicative as FreeSelective f = ControlFlow f ().\n\nIn search of a monoidal tensor\n\nMonoidal tensors are associative up to isomorphism: x ⊗ ( y ⊗ z ) ≅ ( x ⊗ y ) ⊗ z.\nMaking them monoidal is an identity object: I ⊗ x ≅ x ≅ x ⊗ I.\nThey may also satisfy symmetry in two ways, see the difference between braided monoidal categories and symmetric monoidal categories: x ⊗ y ≅ y ⊗ x.\n\nMonoids are a really nice setting for computation, and their symmetries (associativity, and left and right identities) ensure that we can focus on structure that matters while systematically ignoring the structure that does not matter.\n\nHaving the categorical structure also enables us to separate data that the program manipulates from the structure of the program itself.\nFinally, we consider static analysis as the analogous laws on plain monoids (not functors) via the constant functor (decategorification).\n\nSo we want an explanation of selective applicative functors in terms of monoidal tensors.\n\nTensorfail\n\nMost explanations are missing the point of what should be possible with selective applicatives:\nthey search for a tensor that can explain something about select:: f ( Either u v) -> f (u -> v) -> f v.\n\nOne suggestion Which I have seen a couple times is that it should come from a lax monoidal action on coproducts, of the shape f ( Either u v) -> Either (f u) (f v), but that is just wrong:\nthat is the wrong kind of static analysis for selective applicatives to support, as it would require evaluating the control flow purely statically, meaning that f cannot contain any interesting effects after all.\n\nThat is: Lax actions over coproducts ( f ( Either u v) -> Either (f u) (f v) ) are incredibly rare in programming (or Set-like categories, more formally).\nIt is clear that motivating examples of selective applicative functors do not satisfy this.\nWe will in fact see that selective applicative structures are much more plentiful than monads or applicatives.\n\nAnother approach I looked at is viewing it through the lens of possibly-constant functions, like Either (u -> v) v.\nThese are not nicely behaved either: they do not form a category at least not with nice properties, and even computationally, since we cannot detect constant functions in general, it is not clear why the computationally distinguished constant functions should be treated specially.\nWhich is silly, because the point of these functors is all about computation, and I believe there is a nice story for it in category theory too!\n\nSo we need to change our approach and recognize that select is not a primitive that composes nicely: it does not have a monoidal tensor, so it is not the operation we should be looking at.\n\nWe need monoids if we are to determine what “composes nicely” even means.\nThe failure of select to faithfully reproduce branch (or at least the failure of branch to faithfully reproduce a ternary branch ) is exactly the failure of finding a tensorial explanation for what is happening.\n\nWe need to work harder to find the tensor in selective applicative functors, if we want to see a compositional story.\n\nFinite continuations\n\nLetʼs revisit the idea of adapting (>>=):: f i -> (i -> f r) -> f r into a function with a finite number of cases.\nCan we find a data type that encodes this idea?\n\nNot quite good enough\n\nAn idea that is a non-starter is just requiring that i is a finite type here: that would allow enumerating the cases f r, but there is no way to pass non-finite data from i to r (like branch:: f ( Either x y) -> f (x -> r) -> f (y -> r) -> f r clearly does).\n\nSlightly better would be something like f (idx, dat) -> (idx -> f (dat -> r)) -> f r, where idx is finite, and dat is non-finite data that can be processed to obtain r.\nBut again this is unsatisfactory, since dat cannot depend on the chosen index, which again is something that branch can do (and this really matters if one case results in Void, for example, and you need to use absurd ).\n\nIf we had dependent types, we could do IsFinite idx -> f (Σ (i: idx), dat i) -> (Π (i: idx), f (dat i -> r)) -> f r, which would adequately describe what we want.\nBut we do not have dependent types in Haskell, and it does not feel like dependent types should be necessary, even if they are convenient.\nAnd it does not particularly get us closer to a categorical formulation.\n\nBranching\n\nSo … letʼs revisit the type of branch again.\n\nWe want branch:: f ( Either x y) -> f (x -> r) -> f (y -> r) -> f r to look more like (>>=):: f i -> (i -> f r) -> f r.\nIs there a type Branch f i r that can replace the arrow (i -> f r) to make it happen?\n\nWe can mechanically write a datatype that expresses this: when i is of the form Either x y, then we should have the data f (x -> r) and f (y -> r):\n\nMatching on Branch gives the type equality i ~ ( Either x y), but this is not essential, since we can always map an arbitrary f i on the left into a type that is
