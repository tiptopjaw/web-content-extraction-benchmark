{"title":"Preparing a transition in Debusine","author":null,"date":"2025-12-18T13:21:19+00:00","main_content":"We announced a public beta of\nDebusine repositories\nrecently ( Freexian\nblog,\ndebian-devel-announce ).\nOne thing I’m very keen on is being able to use these to prepare\n“transitions”: changes to multiple packages that need to be prepared\ntogether in order to land in testing. As I said in my DebConf25\ntalk:\n\nWe have distribution-wide CI in unstable, but there’s only one of it and\nit’s shared between all of us. As a result it’s very possible to get into\ntangles when multiple people are working on related things at the same\ntime, and we only avoid that as much as we do by careful coordination such\nas transition bugs. Experimental helps, but again, there’s only one of it\nand setting up another one is far from trivial.\n\nSo, what we want is a system where you can run experiments on possible\nDebian changes at a large scale without a high setup cost and without fear\nof breaking things for other people. And then, if it all works, push the\nwhole lot into Debian.\n\nTime to practice what I preach.\n\nSetup\n\nThe setup process is documented on the Debian\nwiki. You need to\ndecide whether you’re working on a short-lived experiment, in which case\nyou’ll run the create-experiment workflow and your workspace will expire\nafter 60 days of inactivity, or something that you expect to keep around for\nlonger, in which case you’ll run the create-repository workflow. Either\none of those will create a new workspace for you. Then, in that workspace,\nyou run debusine archive suite create for whichever suites you want to\nuse. For the case of a transition that you plan to land in unstable, you’ll\nmost likely use create-experiment and then create a single suite with the\npattern sid-<name>.\n\nThe situation I was dealing with here was moving to\nPylint 4. Tests showed that we\nneeded this as part of adding Python 3.14 as a supported Python version, and\nI knew that I was going to need newer upstream versions of the astroid and\npylint packages. However, I wasn’t quite sure what the fallout of a new\nmajor version of pylint was going to be. Fortunately, the Debian Python\necosystem has pretty good autopkgtest coverage, so I thought I’d see what\nDebusine said about it. I created an experiment called cjwatson-pylint\n(resulting in\nhttps://debusine.debian.net/debian/developers-cjwatson-pylint/ - I’m not\nmaking that a proper link since it will expire in a couple of months) and a\nsid-pylint suite in it.\n\nIteration\n\nFrom this starting point, the basic cycle involved uploading each package\nlike this for each package I’d prepared:\n\n$ dput -O debusine_workspace = developers-cjwatson-pylint \\\n-O debusine_workflow = publish-to-sid-pylint \\\ndebusine.debian.net foo.changes\n\nI could have made a new dput-ng profile to cut down on typing, but it\nwasn’t worth it here.\n\nThen I looked at the workflow results, figured out which other packages I\nneeded to fix based on those, and repeated until the whole set looked\ncoherent. Debusine automatically built each upload against whatever else\nwas currently in the repository, as you’d expect.\n\nI should probably have used version numbers with tilde suffixes (e.g.\n4.0.2-1~test1 ) in case I needed to correct anything, but fortunately that\nwas mostly unnecessary. I did at least run initial test-builds locally of\njust the individual packages I was directly changing to make sure that they\nweren’t too egregiously broken, just because I usually find it quicker to\niterate that way.\n\nI didn’t take screenshots as I was going along, but here’s what the list of\ntop-level\nworkflows\nin my workspace looked like by the end:\n\nYou can see that not all of the workflows are successful. This is because\nwe currently just show everything in every workflow; we don’t consider\nwhether a task was retried and succeeded on the second try, or whether\nthere’s now a newer version of a reverse-dependency so tests of the older\nversion should be disregarded, and so on. More fundamentally, you have to\nlook through each individual workflow, which is a bit of a pain: we plan to\nadd a dashboard that shows you the current state of a suite as a whole\nrather than the current workflow-oriented view, but we haven’t started on\nthat yet.\n\nDrilling down into one of these workflows, it looks something like this:\n\nThis was the first package I uploaded. The first pass of failures told me\nabout pylint (expected), pylint-flask (an obvious consequence), and\npython-sphinx-autodoc2 and sphinx-autoapi (surprises). The slightly odd\npattern of failures and errors is because I retried a few things, and we\nsometimes report retries in a slightly strange way, especially when there\nare workflows involved that might not be able to resolve their input\nparameters any more.\n\nThe next level was:\n\nAgain, there were some retries involved here, and also some cases where\npackages were already failing in unstable so the failures weren’t the fault\nof my change; for now I had to go through and analyze these by hand, but\nwe’ll soon have regression tracking to compare with reference runs and show\nyou where things have got better or worse.\n\nAfter excluding those, that left pytest-pylint (not caused by my changes,\nbut I fixed it anyway in unstable to clear out some noise) and spyder.\nI’d seen people talking about spyder on #debian-python recently, so after\na bit of conversation there I sponsored a rope upload by Aeliton Silva,\nupgraded python-lsp-server, and patched spyder. All those went into my\nrepository too, exposing a couple more tests I’d forgotten in spyder.\n\nOnce I was satisfied with the results, I uploaded everything to unstable.\nThe next day, I looked through the tracker as usual starting from\nastroid, and while there are some\ntest failures showing up right now it looks as though they should all clear\nout as pieces migrate to testing. Success!\n\nConclusions\n\nWe still have some way to go before this is a completely smooth experience\nthat I’d be prepared to say that every developer can and should be using;\nthere are all sorts of fit-and-finish issues that I can easily see here.\nStill, I do think we’re at the point where a tolerant developer can use this\nto deal with the common case of a mid-sized transition, and get more out of\nit than they put in.\n\nWithout Debusine, either I’d have had to put much more effort into searching\nfor and testing reverse-dependencies myself, or (more likely, let’s face it)\nI’d have just dumped things into unstable and sorted them out afterwards,\nresulting in potentially delaying other people’s work. This way, everything\nwas done with as little disruption as possible.\n\nThis works best when the packages likely to be involved have reasonably good\nautopkgtest coverage (even if the\ntests themselves are relatively basic). This is an increasingly good\nbet in Debian, but we have plans to\nadd installability comparisons (similar to how Debian’s testing suite\nworks ) as well as optional rebuild testing.\n\nIf this has got you interested, please try it out for\nyourself and let us know how it\ngoes!"}
