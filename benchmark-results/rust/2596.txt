{"title":"How ByteDance Solved Billion-Scale Vector Search Problem with Apache Doris 4.0","author":null,"date":null,"main_content":"Vector search has dominated conversation in context engineering. Pure similarity searches with vector index work well for proof of concepts but face challenges at scale and in production.\n\nByteDance discovered this while building a search for over one billion vectors with Apache Doris. Their initial pure vector implementation hit three critical problems: 1) Result accuracy due to semantic confusion, 2) Ranking stability due to database optimization, and 3) Very high memory requirements that balloon the cost. These problems make a pure vector search solution unviable.\n\nIn this blog, we explore how Bytedance overcame these three challenges with Apache Doris 4.0 to build a cost-effective and performant system handling 1 billion + vectors while also achieving the following benchmarks\n\n400ms latency (7× faster than pure vector search)\n\n500GB memory (20× less than pure HNSW indexing)\n\n89% accuracy (+7 points over pure keyword or vector alone)\n\nSingle server deployment (vs 20-30 servers for sharded vector search)\n\nThe Three Problems with Pure Vector Search\n\nProblem 1: Result accuracy due to semantic confusion\n\nVector search finds vectors with a similar meaning. This becomes a problem when users need exact matches.\n\nExample: A recruiter searches for \"Python developers in San Francisco with 5+ years of experience.\"\n\nPure vector search returns:\n\nCandidates in Seattle (semantically similar—both are West Coast tech hubs)\n\nMissing resumes that say \"5 years of Python\" (embeddings compress numbers imprecisely)\n\nCandidates with 2 years of experience who worked on \"senior-level projects\"\n\nThe root cause is that embedding models optimize for semantic similarity, not exact matching. In the embedding space, \"San Francisco\" and \"Seattle\" are close neighbors. They share similar contexts in training data—both appear in sentences about tech jobs, startups, and West Coast culture. The number \"5\" in \"5 years\" carries little weight compared to the surrounding words.\n\nThis matters because business requirements often demand precision. A compliance officer searching for \"GDPR Article 17\" cannot accept results for \"GDPR Article 13\"—they are legally different documents. A recruiter filtering by location cannot accept candidates in the wrong city.\n\nImpact: In ByteDance's testing, pure vector search achieved only 58% relevant results for queries with exact-match requirements.\n\nProblem 2: Ranking stability due to database optimization\n\nA recruiter searches \"python developer\" on Monday, finds the perfect candidate at position #1, and bookmarks the profile. On Tuesday, she runs the same search to share it with her manager. The candidate now appears at position #8. No new resumes were added. No data changed. The search is \"broken.\"\n\nThis is not a rare edge case. It happens routinely in production systems.\n\nWhy this happens: Databases split large tables into segments for storage efficiency. A billion-row table becomes 100 segments of 10 million rows each. The problem is that BM25 scoring depends on term statistics—how rare is this word?—and these statistics are calculated per segment.\n\nConsider a concrete example:\n\nSegment Total Docs Docs with \"python\" Term Frequency IDF Score\nSegment A 10 million 50,000 0.5% 2.30\nSegment B 10 million 80,000 0.8% 2.10\n\nThe same resume with identical text receives a score of 8.5 in Segment A but 7.9 in Segment B. Where your document lands during ingestion affects its ranking permanently.\n\nThe problem compounds over time. Databases merge segments during nightly maintenance for storage optimization. When Segment A and Segment B merge:\n\nBefore merge: Resume X scores 8.5 (in Segment A)\n\nAfter merge: Combined segment has different statistics, Resume X now scores 8.1\n\nResult: Resume X drops from position #1 to position #8\n\nNo data changed. The database simply reorganized its internal storage. Users see rankings shuffle and conclude the search is unreliable.\n\nThe business impact is severe:\n\nLost trust: Users stop relying on search and revert to manual browsing\n\nSupport burden: \"Why did my saved search change?\" becomes a top ticket category\n\nCompliance risk: In regulated industries, auditors ask why the same query produces different results\n\nMulti-tenant leakage: In SaaS platforms, one customer's document volume can bias another customer's search scores, a subtle but serious data isolation problem\n\nThis instability makes search results feel arbitrary, even when the underlying algorithm is correct.\n\nProblem 3: High memory requirements, high cost\n\nA billion 768-dimension vectors requires significant storage. The standard approach, using HNSW (Hierarchical Navigable Small World) indexing, builds a multi-layer graph structure for fast nearest-neighbor search:\n\nMemory breakdown for 1 billion vectors:\n\nRaw vectors: 1B × 768 dims × 4 bytes = 2.9TB\nGraph connections: 1B × 16 neighbors × 8 bytes = 128GB\nMulti-layer graph: Navigation structure = ~7TB\n\nTotal: ~10TB\n\nTypical production servers have 128GB to 512GB of RAM. Hosting 10TB requires either sharding across 20-30 machines or renting specialized high-memory instances that cost significantly more.\n\nFor comparison, a keyword inverted index on the same data requires approximately 50-100GB, 100× smaller than the HNSW index.\n\nWhy it matters: Infrastructure cost directly affects whether a feature ships. A 20-server deployment requires budget approval, operational complexity, and cross-team coordination. A single-server deployment ships faster.\n\nHow Hybrid Search in Apache Doris Solves Each Problem\n\nHybrid search with vector search introduced Apache Doris 4.0 addresses these problems through progressive filtering: apply the cheapest operation first, expensive operations last. Each layer reduces the data size before the next step.\n\nSolution to Problem 1: Filter Before Semantic Search\n\nInstead of searching all documents semantically, hybrid search applies structured and keyword filters first.\nSELECT candidate_name, resume_text,\nbm25_score(resume_text) as keyword_score,\ncosine_distance(embedding, query_vec) as vector_score\nFROM resumes\nWHERE\n\nlocation = 'San Francisco' AND seniority = 'Senior'\n\nAND resume_text MATCH_ANY 'Python'\n\nAND cosine_distance(embedding, query_vec) < 0.3\n\nORDER BY (keyword_score * 0.3 + vector_score * 0.7 ) DESC\nLIMIT 10;\n\nHow this solves the problem: The location = 'San Francisco' filter eliminates Seattle candidates before the semantic search runs. The keyword filter ensures \"Python\" appears literally in the document, no semantic guessing required. Vector search then ranks the most relevant matches within this already-precise subset.\n\nResult: Relevant results increased from 58% to 94% in ByteDance's testing.\n\nSolution to Problem 2: Calculate Statistics Globally\n\nThe ranking instability problem occurs because BM25 calculates term frequencies per segment. ByteDance's solution: calculate statistics across the entire table before scoring any document.\n\nThe approach:\n\nBefore processing segments, execute a pre-scan that visits all segments (~50ms)\n\nCount total documents (N) and term frequencies (DF) across the entire table\n\nCache these global statistics\n\nApply the same N and DF values when scoring documents in every segment\n\nHow this solves the problem: When every segment uses identical statistics, the same document text produces identical scores regardless of which segment contains it. Segment merges no longer affect rankings because the global counts remain stable.\n\nThe tradeoff: Each query requires an additional 50ms pre-scan. This overhead is acceptable because ranking stability directly affects user trust. With this approach, users can now reliably find previously viewed results in the same positions.\n\nSolution to Problem 3: IVPQ Compression Reduces Memory 20×\n\nHNSW uses 10TB because it stores full-precision vectors plus a multi-layer navigation graph. IVPQ (Inverted File + Product Quantization) replaces this with two compression techniques.\n\nTechnique 1: Inverted File (Clustering)\n\nInstead of one large graph, IVPQ clusters vectors into groups:\n1 billion vectors → 10,000 clusters (via K-means)\n\nEach cluster centroid: [0.12, 0.87, -0.34,...]\nEach cluster contains: ~100,000 similar vectors\n\nSearch process:\n\nCalculate distance from query to 10,000 centroids (fast)\n\nSelect the closest 10 clusters\n\nSearch only within those clusters (10 × 100K = 1M vectors instead of 1B)\n\nThis reduces the search space by 1,000×.\n\nTechnique 2: Product Quantization (Compression)\n\nEach 768-dimension vector occupies 3,072 bytes (768 × 4 bytes per float). Product quantization compresses this to 8 bytes:\n\nSplit the vector into 8 subvectors (96 dimensions each)\n\nTrain a codebook with 256 entries for each subvector position\n\nReplace each subvector with its nearest codebook entry ID (1 byte)\n\nResult: 3,072 bytes → 8 bytes (384× compression)\n\nMemory comparison:\n\nComponent HNSW IVPQ\nVectors 2.9TB 8GB (compressed)\nIndex structure 7TB 400GB\nTotal ~10TB ~500GB\n\nThe tradeoff: IVPQ achieves 92% recall compared to HNSW's 95% recall. This 3% accuracy loss is acceptable given the 20× memory reduction and ability to run on a single server.\n\nOptimization: Skip the Index for Small Result Sets\n\nAfter filtering, hybrid search often reduces candidates to under 50,000 documents. At this scale, ByteDance discovered that brute-force distance calculation outperforms index lookup.\n\nWhy brute force wins on small sets:\n\n20,000 vectors × 768 dimensions × 4 bytes = 61MB of data\n\nModern CPUs have 30-50MB of L3 cache. Sequential reads through 61MB approach cache efficiency. The CPU prefetches data at 100GB/sec.\n\nIndex lookup, by contrast, requires random memory access (jumping between graph nodes) and decompression overhead. Random access is 10× slower than sequential.\n\nBenchmark comparison:\n\nApproach Steps Time\nIndex lookup Decompress → Navigate clusters → Calculate approximate distances 90ms\nBrute force Load 61MB sequentially → Calculate exact distances 30ms\n\nBrute force is 3× faster and returns exact distances (no compression error).\n\nDecision rule:\nif candidates_remaining < 50,000:\nuse brute_force() # Faster and more accurate\nelse:\nuse_index() # Necessary for large sets\n\nProduction Results\n\nByteDance deployed hybrid search on 1 billion resumes, handling high query volume.\n\nExample query: \"Senior ML engineer in Beijing with NLP experience\"\n\nLatency Breakdown\n\nStage Operation Time\nStructured filter location + seniority 50ms\nKeyword search ML, NLP terms via BM25 200ms\nVector search Semantic matching on filtered set 100ms\nReranking Combine scores 50ms\nTotal 400ms (p95)\n\nComparison to Alternatives\n\nApproach Memory Servers Latency Recall\nPure HNSW 10TB 20-30 2.8s 95%\nHybrid (IVPQ) 500GB 1 400ms 92%\n\nAccuracy by Approach (nDCG@10)\n\nApproach Score Limitation\nPure keyword 0.65-0.70 Misses semantic matches\nPure vector 0.80-0.85 Misses exact keywords\nHybrid 0.85-0.90 Best combination\n\nSummary\n\nFor ByteDance, pure vector search fails at billion-scale due to three problems:\n\nSemantic confusion loses exact matches: \"San Francisco\" returns Seattle results\n\nUnstable rankings change when segments merge: users lose trust\n\nMemory explosion requires 10TB for 1B vectors: needs 20+ servers\n\nByteDance used hybrid search in Apache Doris to solve each problem:\n\nProgressive filtering applies exact-match filters before semantic search → 58% to 94% relevance\n\nTable-level statistics calculate BM25 globally → stable rankings across segment merges\n\nIVPQ compression reduces memory 20× → single-server deployment\n\nThe counter-intuitive insight: hybrid search is not a compromise. It outperforms both keyword and vector approaches individually. Progressive filtering makes vector search faster by shrinking the candidate set. Combining scores makes results more accurate through exact and semantic matching together. Compression makes deployment cheaper with 20× less memory.\n\nFor search at billion-scale, hybrid is the standard approach.\n\nReady to build production-grade search?
