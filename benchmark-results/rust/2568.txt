It was designed for macOS enterprise environments where data security compliance required all removable media to be encrypted.\n\nWhat It Did Well\n\nThe original script handled the basics competently:\n\n– Detected external APFS, HFS+, and ExFAT/FAT volumes\n\n– Prompted users with a swiftDialog interface\n\n– Encrypted APFS volumes directly\n\n– Converted HFS+ volumes to APFS before encryption\n\n– Erased and reformatted ExFAT/FAT volumes as encrypted APFS\n\n– Offered a “mount as read-only” option for users who didn’t want encryption\n\nThe Hidden Limitations\n\nBut as deployments grew, limitations became apparent:\n\n1. Single-Volume Processing**\n\nIf you inserted a disk with three partitions, the script would only process the first one. The other two? Invisible to the system.\n\n“`bash\n\nOriginal approach\n\nif [[ $StorageType =~ “Apple_APFS” ]]; then\n\n    # Process APFS\n\nelif [[ $StorageType =~ “Apple_HFS” ]]; then\n\n    # Process HFS+\n\nelif [[ $StorageType =~ “Microsoft Basic Data” ]]; then\n\n    # Process ExFAT\n\nfi\n\nOnly ONE branch executed, then script exits!\n\n“`\n\n2. No Logging Infrastructure**\n\nTroubleshooting was a nightmare. Basic `echo` statements went nowhere when run as a LaunchDaemon.\n\n3. No Testing Capability\n\nWant to test the script? Hope you don’t mind encrypting real drives, because there was no dry-run mode.\n\n4. Generic Error Messages\n\n“FileVault is disabled on disk4s1” – Great, but which drive is that? What’s it called?\n\n5. The Re-Prompt Problem\n\nUser mounts a drive as read-only. Five minutes later, they insert another drive. The script triggers again and asks about BOTH drives, including the one already mounted read-only. Every. Single. Time.\n\n—\n\nChapter 2: The First Major Overhaul – v2.0 Enhanced (December 2025)\n\n### Modernization for macOS 15+ Sequoia\n\nWhen macOS 15 (Sequoia) shipped, several compatibility issues emerged. This sparked a complete reimagining of the script’s architecture.\n\n### The Two-Phase Architecture\n\nThe biggest change was moving from a linear “process-and-exit” model to a sophisticated two-phase system:\n\n**Phase 1: Discovery**\n\n“`bash\n\n# Scan ALL drives\n\n# Detect ALL partitions\n\n# Check ALL volume types independently\n\n# Build a queue of unencrypted volumes\n\n“`\n\n**Phase 2: Processing**\n\n“`bash\n\n# Process each queued volume sequentially\n\n# Track what was encrypted\n\n# Show comprehensive summary at end\n\n“`\n\nThis solved the multi-volume problem instantly. Now a disk with three APFS volumes, two HFS+ partitions, and an ExFAT partition would have ALL of them discovered and processed.\n\n### The Multi-Volume Victory\n\nHere’s a real example of the difference:\n\n**Original v1.0 (Single partition found):**\n\n“`\n\nProcessing /dev/disk4\n\nFound: disk4s1 (APFS volume “Old”)\n\nEncrypted: disk4s1\n\nExit.\n\n“`\n\n**Enhanced v2.0 (All partitions found):**\n\n“`\n\nDiscovery Phase:\n\n– disk29s1 “Old” (APFS, unencrypted) → Added to queue\n\n– disk29s2 “New” (APFS, unencrypted) → Added to queue\n\n– disk29s3 “Lucky” (APFS, unencrypted) → Added to queue\n\n– disk28s3 “Untitled 2” (HFS+, unencrypted) → Added to queue\n\nProcessing Phase:\n\n[1/4] Encrypting “Old” (disk29s1)… Done\n\n[2/4] Encrypting “New” (disk29s2)… Done\n\n[3/4] Encrypting “Lucky” (disk29s3)… Done\n\n[4/4] Converting and encrypting “Untitled 2” (disk28s3)… Done\n\nSummary: 4 volumes encrypted in this session\n\n“`\n\nVolume Names Everywhere\n\nEvery dialog, every log message, every notification now showed friendly volume names alongside technical IDs:\n\n“`\n\nBefore:”Processing disk4s1″\n\nAfter:”Processing ‘MyBackups’ (disk4s1)”\n\n“`\n\nUsers could finally understand what was happening to which drive.\n\nThe LaunchDaemon Dialog Fix\n\nA critical bug emerged: when run as a LaunchDaemon (root), swiftDialog couldn’t accept keyboard input. Password fields were useless. The fix used `launchctl asuser` to run dialogs in the user’s GUI session context:\n\nNow dialogs appeared correctly with proper focus and keyboard interaction.\n\n—\n\nChapter 3: Production Readiness – v2.1 (December 2025)\n\n### The Logging Revolution\n\nv2.1 introduced a professional four-level logging system:\n\n| Level | Output | Use Case |\n\n|——-|——–|———-|\n\n| **0 – Minimal** | Errors only | Production (silent success) |\n\n| **1 – Normal** | Errors + key operations | Standard deployment |\n\n| **2 – Verbose** | + detailed progress | Troubleshooting |\n\n| **3 – Debug** | Everything | Development/diagnosis |\n\n**Dual-destination logging** sent output to both console and macOS unified logging:\n\n“`bash\n\nlog_info() {\n\n    if [[ $LOG_LEVEL -ge 1 ]]; then\n\n        echo “[$(get_timestamp)] INFO: $*”\n\n        logger -p user.info “DiskEncrypter [INFO]: $*”\n\n    fi\n\n}\n\n“`\n\nNow you could watch logs in real-time:\n\n“`bash\n\nlog stream –predicate ‘eventMessage CONTAINS[c] “DiskEncrypter”‘ –info\n\n“`\n\n### Dry-Run Mode: Test Without Fear\n\n“`bash\n\nsudo./DiskEncrypter_Enhanced.sh –dry-run –log-level 3\n\n“`\n\nThis became invaluable for:\n\n– Testing in production environments\n\n– Training new IT staff\n\n– Validating configuration changes\n\n– Demonstrations\n\nEvery disk operation was logged but not executed:\n\n“`\n\n[DRY RUN] Would execute: diskutil apfs encryptVolume disk4s1 -user disk\n\nINFO: DRY RUN: Encryption would start for disk4s1\n\n“`\n\n### Command-Line Arguments\n\nFor the first time, you could override plist settings from the command line:\n\n“`bash\n\n# Debug logging without changing plist\n\n./DiskEncrypter_Enhanced.sh -l 3\n\n# Dry-run with verbose logging\n\n./DiskEncrypter_Enhanced.sh –dry-run -l 2\n\n“`\n\nPriority: **Command-line → Plist → Default**\n\n—\n\nChapter 4: The Bug Hunt – v2.2 (December 2025)\n\n### The Critical Read-Only Bug\n\nA critical bug was discovered: the script couldn’t detect read-only volumes, causing users to be re-prompted endlessly.\n\n**The Bug:**\n\n“`bash\n\n# Wrong field name!\n\nvolumeMountInfo=$(diskutil info “$VolumeID” | grep “Read-Only Volume:”)\n\nif [[ “$volumeMountInfo” == “Yes” ]]; then\n\n    # This NEVER executed!\n\nfi\n\n“`\n\n**Why It Failed:**\n\n“`bash\n\n$ diskutil info disk4s2 | grep “Read-Only Volume:”\n\n# (no output – field doesn’t exist!)\n\n$ diskutil info disk4s2 | grep “Volume Read-Only:”\n\n   Volume Read-Only:           Yes (read-only mount flag set)\n\n# ^^^ The actual field name!\n\n“`\n\n**The Fix:**\n\n“`bash\n\nvolumeMountInfo=$(diskutil info “$VolumeID” | grep “Volume Read-Only:”)\n\nif [[ “$volumeMountInfo” =~ ^Yes ]]; This created a window where data could accidentally be written to unencrypted media.\n\n### Auto Read-Only Mounting\n\nv2.3’s solution was elegant: automatically mount ALL unencrypted volumes as read-only immediately upon detection, before showing any dialog.\n\n**New Function:**\n\n“`bash\n\nmountReadOnly() {\n\n    local VolumeID=$1\n\n    local volumeName=$2\n\n    log_info “Auto-mounting volume as read-only: $VolumeID ($volumeName)”\n\n    # Unmount first\n\n    diskutil unmountDisk “$VolumeID” 2>/dev/null\n\n    # Mount as read-only\n\n    diskutil mount readOnly “$VolumeID”\n\n    return 0\n\n}\n\n“`\n\n**Updated Workflow:**\n\n“`\n\nv2.2:\n\n1. Back up all data to a secure location\n\n2. Use Disk Utility to erase and format as APFS\n\n3. The v2.4 solution of removing the option entirely was controversial but necessary.\n\n**Key Learning:** When data loss is possible, don’t rely on warnings—remove the dangerous option.\n\n### 2. Edge Cases Are Real Cases\n\nThe read-only field name bug (v2.2) seemed minor until users reported constant re-prompts. The duplicate dialog bug (v2.4.3) only appeared in production when LaunchDaemon unmount events triggered re-execution.\n\n**Key Learning:** Real-world testing reveals issues that synthetic tests miss.\n\n### 3. Good UX Is Iterative\n\nv2.4’s dialogs had too much text. v2.4.1’s infobox solution reduced reading time by 66%. Logging Saves Lives (and Debugging Time)\n\nThe v2.1 four-level logging system with dual output (console + system logger) paid dividends in every subsequent version. Debug output that would have taken hours to add per-bug was already there.\n\n**Key Learning:** Invest in logging infrastructure early. Future you will thank present you.\n\n### 5. Backward Compatibility Matters\n\nEvery version maintained 100% backward compatibility with configuration files and user workflows. APFS/HFS+ encryption worked identically across all versions.\n\n**Key Learning:** Add features, don’t break existing deployments.\n\n### 6. Concurrent Execution Is Harder Than It Looks\n\nThe feedback loop bug (v2.4.3) required two complementary solutions: lock files for mutual exclusion AND processed volume tracking for cooldowns.\n\n**Key Learning:** Concurrent execution requires multiple layers of protection.\n\n—\n\nChapter 11: The Final Product\n\n### What Disk Encrypter Enhanced v2.4.5 Delivers\n\n**For IT
