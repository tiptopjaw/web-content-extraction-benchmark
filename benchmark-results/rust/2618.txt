{"title":"Scaling Go Testing with Contract and Scenario Mocks","author":null,"date":null,"main_content":"In this article\n\nMocks aren't the enemy; bad tests are. We scale Go testing using \"Tactical Pairs\": Contract Tests for truth, Scenario Mocks for logic.\n\nDec 17, 2025\n\n6 min read\n\nBrowsing the /r/golang subreddit, you might get the impression that \"mock\" is a dirty word.\n\nI was recently reading a thread on testing strategies and was surprised by the intensity of the anti-mock sentiment. The arguments were visceral:\n\nAvoid mocks like the plague.\n\nMocks are like payday loans. Convenient. You say “expect this method to be called with args X and Y, return Z”. Seems cheap. But the loan shark quickly starts charging interest.\n\nAt FunnelStory, we see it differently. We’ve learned to love mocks—not as a shortcut, but as the only way to meaningfully build and maintain a complex application as a small team (with no dedicated QA).\n\nWe manage a complex distributed system that orchestrates LLMs, syncs CRM data, and handles long-running workflows. In this environment, relying solely on \"real\" integration tests left us with blind spots.\n\nWe couldn't reliably test what happens when an API sends a 429 Rate Limit, or when a workflow needs to pause for exactly 24 hours. More importantly, we realized that the majority of our edge cases depend on specific combinations of integrations that exist in a customer's workspace—states that are impossible to reproduce or recreate in our own environments.\n\nThis reality drove us to define a new goal: 100% meaningful coverage.\n\nWe don't just want green checkmarks for the \"happy paths\" that run when the database is online. We want to cover the \"chaos paths\" that cause production outages. While we are still refactoring our legacy code to fully meet this standard, this disciplined style of mocking has become indispensable to our scale.\n\nHere is the \"Tactical Pair\" strategy we adopted to make mocks safe, reliable, and effective.\n\nThe Problem: Mock Drift vs. Integration Complexity\n\nWhen testing systems, you face two opposing risks:\n\nMock Drift: Mocks are static, but reality evolves. If your database schema changes but your mock doesn't, your test suite stays green while production breaks. The mock becomes a lie.\n\nThe Integration Trap: Testing logic that depends on external volatility (like API rate limits or long-running workflows) against real dependencies is slow, flaky, and non-deterministic.\n\nThe solution isn't to pick a side; it's to sequence them. You should never mock a dependency until you have verified its behavior with a real, isolated Contract Test.\n\nThe Contract Test (Real Infrastructure)\n\nA Contract Test verifies the Data Layer against real infrastructure. It does not test business logic; it tests the Round Trip.\n\nThe Database Contract (Postgres)\n\nBy writing data and immediately reading it back, we prove that the code speaks the correct SQL dialect and that the struct tags match the schema.\n// internal/store/user_repo_test.go\n// RUNS AGAINST: Real Postgres Container\nfunc TestUserRepo_RoundTrip(t *testing.T) {\ndb:= testdb.New(t)\nrepo:= NewPostgresUserRepo(db)\n\n// 1. Write Data (Prove the schema accepts it)\nerr:= repo.UpsertIdentity(ctx, &Identity{UserID: \"u1\", Token: \"xoxb-123\"})\nrequire.NoError(t, err)\n\n// 2. Read Data (Prove we map it back correctly)\nidentity, err:= repo.GetIdentity(ctx, \"u1\")\nassert.NoError(t, err)\nassert.Equal(t, \"xoxb-123\", identity.Token)\n}\n\nThe API Contract (Salesforce)\n\nFor external APIs, we can't always hit the real endpoint (due to rate limits or auth). Instead, we verify our client can parse Real JSON Responses. We use httpmock with a fixture captured from a real Salesforce call.\n// internal/integrations/crm/sfdc_test.go\n// RUNS AGAINST: HttpMock with Real JSON Fixture\nfunc TestSFDC_FindAccountByDomain_ParsesResponse(t *testing.T) {\n// 1. Load a real, messy JSON response captured from Salesforce\n// This proves our struct tags match the actual API field names\nfixture, _:= os.ReadFile(\"testdata/sfdc_account_response.json\")\n\nhttpmock.RegisterResponder(\"GET\", \"https://na1.salesforce.com/services/data/v53.0/query\",\nhttpmock.NewBytesResponder(200, fixture),\n)\n\nclient:= NewSFDCClient(\"valid-token\")\n\n// 2. Execute the Client Method\naccount, err:= client.FindAccountByDomain(ctx, \"acme.com\")\n\n// 3. Verify Parsing\nassert.NoError(t, err)\nassert.Equal(t, \"001xx000003Dgsd\", account.ID)\nassert.Equal(t, \"Acme Corp\", account.Name)\n}\n\nOnce these tests pass, we trust that GetIdentity speaks SQL and FindAccountByDomain speaks Salesforce JSON. We can now mock these contracts in higher-level tests without fear of drift.\n\nYou Still Need E2E Tests\n\nIt is important to note that Contract Tests are not a replacement for End-to-End (E2E) tests.\n\nA Contract Test proves that \"If the API returns X, my code does Y.\" It does not prove that the API still returns X. If Salesforce changes their API schema tomorrow (which they ideally shouldn't, without a version change), our Contract Test will pass (because it uses a cached fixture), but production will fail.\n\nWe still rely on a lightweight layer of true E2E smoke tests (running on a schedule, not on every PR) to verify that our fixtures are still valid and that authentication flows haven't changed.\n\nOnce these Contract tests pass, however, we trust the boundary. We can now mock these interfaces in our complex logic tests without fear of drift.\n\nDesigning for Testability\n\nYou cannot simply add mocks to coupled code. The application must define dependencies via interfaces rather than concrete structs.\n\nConsider a worker that fetches a calendar event, checks a CRM, and sends a Slack notification.\n\nBad: Coupled Dependencies\ntype MeetingWorker struct {\ndb *sql.DB // Concrete SQL connection\nnylasClient *nylas.Client // Concrete HTTP client\n}\n\nGood: Interface-Based Design\n\nBy defining contracts, we separate infrastructure (SQL, HTTP) from business value (filtering, alerting).\n// internal/worker/meeting_prep.go\n\n// The Contracts\ntype CalendarClient interface {\nGetUpcomingEvents(ctx context.Context, token string) ([]Event, error)\n}\ntype CRMClient interface {\nFindAccountByDomain(ctx context.Context, domain string) (*Account, error)\n}\ntype SlackNotifier interface {\nPostMessage(ctx context.Context, userID string, msg string) error\n}\n\n// The Logic\ntype Runner struct {\ncalendar CalendarClient\ncrm CRMClient\nslack SlackNotifier\n}\n\nfunc NewRunner(cal CalendarClient, crm CRMClient, slack SlackNotifier) *Runner {\nreturn &Runner{ calendar: cal, crm: crm, slack: slack }\n}\n\nThe Scenario Test (Mocked Logic)\n\nWith trusted contracts in place, we can use mocks to test \"Impossible\" scenarios—edge cases that are difficult or destructive to reproduce in a real environment.\n\nLogic Filtering\n\nTesting that nothing happens is notoriously flaky in integration tests. Mocks make this deterministic.\nfunc TestProcess_Scenario_InternalMeeting(t *testing.T) {\n// Simulate: CRM returns no match\nmockCRM.On(\"FindAccountByDomain\", ctx, \"funnelstory.ai\").Return(nil, nil)\n\n// Expectation: DO NOT call Slack\nmockSlack.AssertNotCalled(t, \"PostMessage\")\n\nrunner.ProcessMeeting(ctx, \"user_1\")\n}\n\nError Handling\n\nWe can simulate specific downstream failures, such as a Rate Limit error, to ensure the worker re-queues the job correctly.\nfunc TestProcess_Scenario_SlackRateLimit(t *testing.T) {\n// Simulate: CRM Match\nmockCRM.On(\"FindAccountByDomain\", ctx, \"acme.com\").Return(&Account{ID: \"001\"}, nil)\n\n// Simulate: Slack returns a specific error\nmockSlack.On(\"PostMessage\", ctx, \"user_1\", mock.Anything).\nReturn(errors.New(\"slack: rate limit_exceeded\"))\n\nerr:= runner.ProcessMeeting(ctx, \"user_1\")\n\n// Verify the error bubbles up for the Job Queue to handle\nassert.ErrorContains(t, err, \"rate limit\")\n}\n\nThe Toolkit\n\nTo make this practical, we rely on a few battle-tested Go packages:\n\nhttps://pkg.go.dev/github.com/stretchr/testify/mock: The industry standard for assertions and mocking ( testify/mock ). It allows us to generate mocks for our interfaces easily.\n\nhttps://pkg.go.dev/github.com/jarcoal/httpmock: Essential for testing our API client contracts. It lets us intercept HTTP requests at the transport layer to verify URL structure and simulate JSON responses without hitting the network.\n\nhttps://pkg.go.dev/github.com/DATA-DOG/go-sqlmock: Used sparingly. We prefer real Postgres containers for Contract Tests, but sqlmock is occasionally useful for testing complex transaction logic or driver-specific edge cases where a container is overkill.\n\nSummary\n\nThis approach uses Contract Tests to prove the code interacts with the database and external APIs correctly, and Scenario Tests to prove the code handles business logic and errors correctly. This separation allows for high confidence in the system's behavior without the overhead of maintaining a full end-to-end test suite for every edge case."}
