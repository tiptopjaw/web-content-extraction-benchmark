One was a\nan old school C64/Amiga intro type of thing, and the other was a\nnyan cat.\n\n\"TT08\" intro\n\nThe intro features a background starfield, a panning 3D checkerboard with\ncolorful tiles, and wavy scrolling text which casts a shadow onto the\ncheckerboard. I was inspired by lft's first AVR demo\nCraft to add a little cyan\noscilloscope to the side of the screen as well.\n\nThe size limit was two Tiny Tapeout \"tiles\" which is not a lot of space for\nmusic, effects, and data; this is a very constrained platform for a demo: you\nget no ROM, no RAM, and every bit of state is a flipflop which takes up a decent\nchunk of space compared to combinational logic. Traditional \"sizecoding\" tricks\ndon't apply here -- data compression doesn't help you if you can't actually\nstore a ROM; you have no frame buffer so you must produce a pixel every clock\ncycle. There's no CPU to exploit, just your own state machine.\n\nPrototyping\n\nLike the VGA donut, this one used the weird custom video mode of 1220x480 and a\n48MHz clock as it was easiest to prototype with given the only FPGA hardware I\nhad on hand, but in\nretrospect I very much regret using this video mode. Most of the digital\ncaptures of the demo look bad thanks to dithering and aliasing artifacts. It\ndoes look great on a real CRT though!\n\nThe above video was made with a Verilator simulation, which compiles the Verilog\nsource down to C++ and allowed me to do cycle-by-cycle simulation of the design,\nrendering to a SDL window / audio stream. This also allowed me to dump out\nsimulated frames to create the above video. I would later use this SDL VGA\nverilator\ntestbench\nfor the other projects.\n\nThe C++ simulation runs about half-realtime on my system, though, and most of\nthe time I synthesized it for the OrangeCrab's Lattice ECP5 FPGA strapped to an\nugly R-2R DAC (it's two bits per channel, so literally R and 2R, no ladder, for\nRGB) hacked together on a perfboard.\n\nSound is output on a single pin through an RC filter; most people used PWM of\nsome kind, but I implemented sigma-delta conversion which is actually simpler\n(with some tradeoffs; more below ). Either there would be too many cells to fit the area, or they would\nfit but routing wires between them would fail (sometimes after an hour or\nmore!). Being able to dry run several experiments on my own PC was invaluable.\n\nThis design ended up using 3374 total cells ( breakdown of cell use and 3D\nviewer here )\nand filling just about every square nanometer. I still ended up with 293 total flipflops.\n\nSine scroller\n\nThe invitation intro (which is\nnot a real ASIC demo, just a mockup!) that Matt Venn created for the competition\ninspired me to do something in a similar vein, and for some reason, I was\nfixated on using one of the fonts for my demo. In fact part of what the synthesis pipeline 1 does is to take an arbitrarily\nlarge truth table and match it to the set of available logic cells (on an FPGA\nthis tends to be a bunch of 4-input lookup tables but Sky130 has all kinds of\nassorted gates and combined logic\ncells - random example ).\nSo I provided a giant lookup\ntable as a\nhex file and the yosys synthesis pipeline\nfigured out a way to implement it with the standard cell library, effectively by\nfinding patterns in the data that match up with patterns in the addresses, and\nthen, instead of fitting them in a nice rectangle, sprayed those standard logic\ncells all over my available die area in whatever shape made sense for the other\nstandard cells connected to them. This makes is very hard to predict how much\n\"data\" will fit.\n\nWhat this means for a limited gate area demo is that instead of minimizing the\nsize of data to store as lookup tables, it's better to limit the\nalgorithmic complexity of the data as a function of its address. Repeating the\nsame power-of-two-sized pattern is free; only variations need extra logic.\nThis is helpful to keep in mind when composing music, for example.
