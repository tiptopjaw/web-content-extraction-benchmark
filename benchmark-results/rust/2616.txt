Ideally you will have multiple agents working on multiple things at\nthe same time. For instance, you may have one agent fixing a bug in a bugfix-branch while another implements a feature\nin feature-branch. Because of this git worktrees will be essential.\n\nTmux\n\ntmux is a\nterminal multiplexer that gives you a structured way to manage\nmultiple terminals within a series of sessions, windows (tabs), and panes. Just as git worktrees give you multiple\nworkspaces in the codebase within which to work, tmux gives you multiple workspaces for the agents, shells, IDE\ninstances, and various tools that you will be using during development.\n\ntmux also provides a server in the container that can persist beyond your initial connection. If you are accidentally\ndisconnected from your container you can reconnect and re-attach to the existing tmux session to continue exactly where\nyou left off. Your agents, as children of the tmux process, can also continue to run without you being connected to the\ncontainer.\n\nLastly, tmux also has the ability to listen to terminal bells and\nhighlight the appropriate tab/window when an agent has finished a task or is asking you questions. This is an important\nfeature for being truly efficient with your concurrently running agents.\n\nNote: In Claude Code, sending notifications as terminal bells needs to be configured in /config.\n\nNeovim\n\nWhile I have been an ardent user of IntelliJ for over a decade now, it unfortunately\ndoesn’t lend itself very well to my new way of working. This is the ideal structure I’ve found for working with git worktree. The default directory being the\nactual repository usually checked out to main / master (but it can be checked out to anything, think of it as a\ntemporary scratch workspace) with all the additional worktrees being siblings to default.\n\nModeling the repository like this allows you to easily locate and work within a worktree or even multiple worktrees with\nyour agents (e.g. /add-dir../PR-124-fix-thing in Claude Code). This can be done\nmanually pretty easily but as a convenience I created a function in bash that I use for managing worktrees and tmux\nwindows called gwtmux. If using my\ndev container, gwtmux will be added and automatically sourced\nat the start of every shell session making it available for use.\n\ngwtmux is entirely vibe-coded… but have no fear,\nit has tests.\n\nIts usage is pretty simple and the commands can be run from any tmux window. # create tmux window and worktree named after github PR (requires authenticated `gh` cli)\ngwtmux 123\n# create tmux window, worktree, and branch (will create branch if doesn't exist, or checkout if exists locally or remotely)\ngwtmux my-local-or-remote-branch\n# close current tmux window (but leave worktree folder and branch)\ngwtmux -d\n# close current tmux window and delete worktree folder (but leave branch)\ngwtmux -dw\n# close current tmux window, delete worktree folder, safely delete branch\ngwtmux -dwb # or gwtmux -dwB to force delete unmerged branch\n# close current tmux window, delete worktree folder, delete local and remote branch\ngwtmux -dwBr\n\nThat’s the gist of it. Devcontainers are usually meant to be coupled to a single IDE instance on\nthe host machine which then runs a single container for that instance. It makes the code workspace available to the\ncontainer through a bind mount. Meanwhile I’m suggesting\nrunning everything in the container, including the IDE and the code. I avoid bind mounts wherever possible due to\nsecurity and performance related reasons.\n\nI could potentially run a devcontainer while disabling all the things I don’t want it to do… Or I can just create\nmy own container and only add the things I specifically want it to do. In other words, for me it is an abstraction\nthat doesn’t add any value over what I currently have.\n\nWhy go through all the hassle of forwarding $SSH_AUTH_SOCK socket from host instead of just using an SSH\nForwardAgent per session?\n\nThe SSH session lifecycle and keys are meant to be tied to the host’s lifecycle and keys rather than the lifecycle\nand keys associated with a single connection. For example, if you forwarded keys on connecting and started a tmux\nprocess that persisted beyond the lifespan of your connection it would lose access to the ssh keys once you disconnected. Can I run tmux on the host instead of in the container?\n\nYou can but probably shouldn’t. One of the main benefits from running tmux in the container is having a parent tmux\nprocess that adopts your child processes if you get unintentionally disconnected. You could run tmux on both the host,\nand in the container but you should probably avoid nesting tmux sessions if you can.
