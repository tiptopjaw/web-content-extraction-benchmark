A natural way of achieving this is through a state transformation function that takes the current state of the generator as input and produces a new state. As long as the function is deterministic, it is guaranteed that we can reproduce the exact same sequence of numbers starting from the same initial state. Mathematically, it can be expressed like this:\n\n\\(\\text{s}_{n+1} = \\text{f} \\text{(s}_n)\\)\n\nHere, the next state is derived by applying the function f on the current state s_n. As you can see, this is a sequential model where you can’t jump ahead arbitrarily without computing all the previous states, and you can’t shard the generation of the random numbers by distributing the work across threads.\n\nTo parallelize the generation of random numbers, we need a different model where we can directly generate the nth random number without having to go through the generation of all the previous n-1 numbers. Mathematically, it should look like this:\n\n\\(x_{n} = b(n)\\)\n\nWhere x_n is the nth random number we wish to generate by applying a function b. Here, we can think of the input n as an integer counter and as such the PRNGs that follow this model are called counter-based random number generators. One such counter-based PRNG is the Philox PRNG, used widely in frameworks such as PyTorch for parallel random number generation on GPUs.\n\nLet’s understand how Philox works.\n\nHow Philox Works\n\nThe Philox algorithm, short for “Product HI, LOw, with XOR”, is a counter-based PRNG that was designed specifically for parallel computation. It was introduced by Salmon et al. in 2011 as part of the Random123 library. The key insight behind Philox is that we can use a cryptographic-like construction to transform a counter into a pseudorandom number.\n\nThe Core Idea: Treating RNG as Encryption\n\nWe can think of the counter-based RNG problem this way: we want to take a sequence of integers (0, 1, 2, 3, …) and scramble them so thoroughly that they appear random. This is conceptually similar to what a block cipher does in cryptography, it takes a plaintext message and a key, then produces a ciphertext that looks random.\n\nIn Philox’s case:\n\nThe counter (n) acts like the plaintext\n\nThe seed acts like the encryption key\n\nThe output is our pseudorandom number\n\nThe beauty of this approach is that any thread can independently compute its random number by knowing just two things: which counter value it needs (its position in the sequence) and the seed. No synchronization or communication with other threads is needed.\n\nThe Philox Construction\n\nPhilox operates on fixed-size inputs and outputs. The most common variant is Philox-4x32, which means:\n\n4: Works with 4 32-bit integers at a time\n\n32: Each integer is 32 bits wide\n\nSo Philox-4x32 takes a 128-bit counter (represented as four 32-bit integers) and produces a 128-bit output (four 32-bit random numbers). This is perfect for generating multiple random numbers at once, which is common in GPU workloads.\n\nThe algorithm consists of applying multiple rounds of a transformation function. Each round performs these operations:\n\nMultiplication and splitting: Multiply pairs of the input integers and split the results into high and low parts\n\nXOR with keys: XOR certain parts with key-derived values\n\nPermutation: Shuffle the positions of the integers\n\nLet’s break down a single round in detail. Philox-4x32 works with four 32-bit values, which we’ll call ( c 0​, c 1​, c 2​, c 3​). Each round transforms these values through the following steps:\n\nStep 1: Multiply and Split\n\nTake the first pair ( c 0​, c 1​) and the second pair ( c 2​, c 3​). Multiply each by a carefully chosen constant:\n\n\\(\\begin{align}\n\\text{prod}_0 &= M_0 \\times c_0 \\\\\n\\text{prod}_1 &= M_1 \\times c_2\n\\end{align}\n\\)\n\nFor Philox-4x32, these constants are:\n\nM 0​=0xD2511F53\n\nM 1​=0xCD9E8D57\n\nThese constants were chosen through careful analysis to ensure good statistical properties. When we multiply two 32-bit numbers, we get a 64-bit result. We split this into:\n\nHigh 32 bits: hi(prod)\n\nLow 32 bits: lo(prod)\n\nStep 2: XOR with Keys\n\nThe high parts are XORed with round-specific keys derived from the seed, and with the other input values:\n\n\\(\\begin{align}\nh_0 = \\text{hi}(\\text{prod}_0) \\oplus c_1 \\oplus k_0 \\\\\n\nh_1 = \\text{hi}(\\text{prod}_1) \\oplus c_3 \\oplus k_1\n\\end{align}\n\\)\n\nHere, k 0​ and k 1​ are the key values (derived from the seed), and ⊕ represents the XOR operation.\n\nStep 3: Permutation\n\nFinally, we rearrange the values for the next round. The output of one round becomes:\n\n\\((c_0’, c_1’, c_2’, c_3’) = (\\text{lo}(\\text{prod}_0), h_1, \\text{lo}(\\text{prod}_1), h_0)\\)\n\nNotice how the values are shuffled: the low parts of the products go to positions 0 and 2, while the XORed high parts are swapped and go to positions 1 and 3.\n\nMultiple Rounds\n\nTo achieve good randomness, Philox-4x32 typically applies 10 rounds of this transformation. After each round except the last, the keys are also updated:\n\n\\(\\begin{align}\nk_0’ = k_0 + w_0 \\\\\n\nk_1’ = k_1 + w_1\n\n\\end{align}\\)\n\nWhere w 0​=0x9E3779B9 and w 1​=0xBB67AE85 are the “ Weyl sequence ” constants derived from the golden ratio. This ensures that each round uses different key material, increasing the mixing of the input bits.\n\nVisualizing a Complete Philox Transformation\n\nThe following diagram shows the complete flow through multiple rounds:\n\nWhy This Works\n\nThe Philox algorithm achieves good randomness through several mechanisms:\n\nMultiplication is a non-linear operation that mixes bits effectively. Small changes in input lead to large changes in output.\n\nHigh-low splitting ensures we use all 64 bits of the multiplication result, not just the lower 32 bits.\n\nXOR operations combine different data streams (keys, previous values) in a way that’s invertible but unpredictable without knowing the key.\n\nPermutation ensures that the mixing effect propagates to all output positions across rounds.\n\nMultiple rounds compound these effects, ensuring that every output bit depends on every input bit in a complex way.\n\nThe algorithm has been extensively tested and passes standard statistical tests for randomness like the TestU01 suite, making it suitable for scientific computing and machine learning applications.\n\nProperties of Philox\n\nBefore we dive into PyTorch’s implementation, let’s summarize the key properties that make Philox attractive:\n\nParallel-friendly: A GPU with thousands of cores can generate thousands of random numbers simultaneously, each using a different counter value.\n\nDeterministic: Given the same seed and counter, you always get the same output.\n\nLong period: With a 128-bit counter, you can generate 2^128 random numbers before the sequence repeats numbers, more than enough for any practical application.\n\nFast: The operations (multiplication, XOR, bit shifting) are primitive operations that run very efficiently on modern CPUs and GPUs.\n\nMemory efficient: The generator state is just the counter and key, requiring minimal storage per thread.\n\nNext, let’s understand how Philox can be parallelized.\n\nParallelizing Philox: Subsequences and Offsets\n\nNow that we understand how the Philox algorithm works, let’s explore what makes it particularly powerful for parallel computing: the ability to generate random numbers across thousands of threads simultaneously without any coordination.\n\nThe Random Number Space\n\nRecall that Philox is a counter-based PRNG. At its core, it’s a function that maps a 128-bit counter to a 128-bit random output:\n\n\\(\\text{Philox}(\\text{counter}, \\text{key}) \\rightarrow \\text{random_output}\\)\n\nGiven a fixed key (derived from the seed), each unique counter value produces a unique set of random numbers. Since we have a 128-bit counter, we have:\n\n\\(2^{128} \\approx 3.4 \\times 10^{38} \\text{ possible counter values}\\)\n\nEach counter value produces 4 random 32-bit numbers (since 128 bits = 4 × 32 bits), giving us an enormous space of random numbers. We can visualize this as a huge one-dimensional array:\nCounter: 0 1 2 3... 2^128-1\n\n↓ ↓ ↓ ↓ ↓\n\nOutput: [r₀,r₁,r₂,r₃][r₄,r₅,r₆,r₇][r₈,r₉,r₁₀,r₁₁][r₁₂,...]...[...]\n\nHow do we partition this massive space across parallel threads? One approach is to split the counter space between the threads.\n\nPartitioning the Counter Space\n\nThe key insight is that we can split the 128-bit counter into two parts and use them to create a 2D address space. Think of the counter as having 4 components of 32 bits each: ( c 0​, c 1​, c 2​, c 3​).\n\nWe can partition this as:\n\nUpper 64 bits: Which thread’s region we’re in\n\nLower 64 bits: The position within a thread’s assigned region\n\nThis partitioning scheme gives each thread its own “slice” of the random number space:\n\nThread 0 gets counters: (∗,∗,0,0) where ∗∗ can be any value\n\ncounter = (0,0,0,0) → first 4 random numbers for thread 0\n\ncounter = (1,0,0,0) → next 4 random numbers for thread 0\n\ncounter = (2,0,0,0) → next 4 random numbers for thread 0\n\n…\n\nThread 1 gets counters: (∗,∗,1,0)\n\ncounter = (0,0,1,0) → first 4 random numbers for thread 1\n\ncounter = (1,0,1,0) → next 4 random numbers for thread 1\n\ncounter = (2,0,1,0) → next 4 random numbers for thread 1\n\n…\n\nThread 2 gets counters: (∗,∗,2,0)\n\ncounter = (0,0,2,0) → first 4 random numbers for thread 2\n\nAnd so on…\n\nTerminology: Subsequence and Offset\n\nWe now give names to these two parts:\n\nSubsequence: The upper 64 bits of the counter. This identifies which parallel thread or stream we’re referring to. We can have up to 2^64 different subsequences running in parallel.\n\nOffset: The lower 64 bits of the counter. This identifies the position within a subsequence. Each subsequence can generate up to 2^64 sets of random numbers.\n\nTogether, they form a coordinate system ( s, o ) where:\n\ns is the subsequence (which parallel stream)\n\no is the offset (position in that stream)\n\nThe total capacity is:\n\n\\(2^{64} \\text{ subsequences} \\times 2^{64} \\text{ offsets per subsequence} = 2^{128} \\text{ total positions}\n\n\\)\n\nThis matches exactly the size of our original counter space, we’ve simply reorganized it into a 2D structure that’s easy to partition across threads.\n\nHow Offsets Increment\n\nWhen a thread generates more random numbers, it increments the offset portion of the counter. Since Philox generates 4 random numbers at once, we typically increment by 1 each time (remembering that each offset value produces 4 numbers):\nThread 0 subsequence = 0:\n\noffset=0: counter=[0,0,0,0] → Philox → [rand₀, rand₁, rand₂, rand₃]\n\noffset=1: counter=[1,0,0,0] → Philox → [rand₄, rand₅, rand₆, rand₇]\n\noffset=2: counter=[2,0,0,0] → Philox → [rand₈, rand₉, rand₁₀, rand₁₁]\n\n...\n\nThe offset is really tracking “which batch of 4” we’re on. If we need the 10th random number (index 9, counting from 0):\n\nOffset = ⌊9/4⌋=2\n\nPosition within batch = 19mod4=1\n\nSo we use counter [2,0,0,0] and take the second output (index 1)\n\nThe Power of Skip-Ahead\n\nOne powerful consequence of this design is skip-ahead: a thread can jump directly to any offset without computing intermediate values.\nThread 0:\n\n- Jump to offset 1,000,000: counter = [1000000, 0, 0, 0]\n\n- Generate random numbers at this position\n\n- Jump to offset 5,000,000: counter = [5000000, 0, 0, 0]\n\n- No need to compute offsets 1 through 4,999,999!\n\nThis is impossible with traditional sequential PRNGs where state n+1 n +1 depends on state n n.\n\nSetting Up for PyTorch\n\nNow that we understand how the counter space is partitioned, we can see how PyTorch uses this:\n\nWhen PyTorch generates random numbers on a GPU:\n\nIt launches many threads (e.g., 1024 threads)\n\nEach thread is assigned a unique subsequence number (typically its thread ID)\n\nEach thread starts at offset 0 within its subsequence\n\nAs each thread generates random numbers, it increments its offset\n\nPyTorch tracks the global offset to ensure future operations don’t reuse the same counters\n\nWith this foundation, let’s now explore how PyTorch implements these concepts in its Philox engine.\n\nPhilox Implementation in PyTorch\n\nPyTorch uses Philox-4x32-10 (4 values of 32 bits, 10 rounds) as its primary PRNG for CUDA operations. The implementation lives in aten/src/ATen/core/PhiloxRNGEngine.h and is designed to work on both CPU and GPU (via CUDA). Let’s dissect this implementation to understand how the theoretical concepts we discussed earlier translate into actual code.\n\nCore Data Structures\n\nThe implementation starts by defining some type aliases for clarity:\ntypedef std::array<uint32_t, 4> UINT4; // Four 32-bit integers\n\ntypedef std::array<uint32_t, 2> UINT2; // Two 32-bit integers\n\ntypedef std::array<double, 2> DOUBLE2; // Two doubles\n\ntypedef std::array<float, 2> FLOAT2; // Two floats\n\nThese typedefs make the code more readable. UINT4 represents the 128-bit counter or output (4 × 32 bits = 128 bits), while UINT2 represents the 64-bit key (2 × 32 bits = 64 bits).\n\nThe PhiloxEngine Class Structure\n\nThe philox_engine class maintains four critical pieces of state:\nprivate:\n\ndetail::UINT4 counter_; // 128-bit counter (c₀, c₁, c₂, c₃)\ndetail::UINT4 output_; // Cached output from last round\ndetail::UINT2 key_; // 64-bit key derived from seed (k₀, k₁)\nuint32_t STATE; // Position in current output (0-3)\n\nLet’s understand each field:\n\ncounter_: This is the 128-bit counter that gets incremented and transformed through the Philox rounds. It’s divided into four 32-bit components:\n\ncounter_[0] and counter_[1]: Lower 64 bits represent the offset (which random number in the subsequence)\n\ncounter_[2] and counter_[3]: Upper 64 bits represent the subsequence (which parallel stream)\n\nkey_: The 64-bit key derived from the seed. This remains constant for a given seed and is used in the XOR operations during each round.\n\noutput_: Philox generates 4 random 32-bit numbers at once. This field caches those numbers so we don’t have to recompute them for every call.\n\nSTATE: A simple counter (0-3) that tracks which of the four cached output values to return next. This is an optimization to avoid regenerating when we have unused random numbers.\n\nInitialization and State Management\n\nThe constructor initializes the engine with a seed, subsequence, and offset:\n\nThe C10_HOST_DEVICE macro is crucial here, it tells the compiler that this function can run on both the CPU (host) and GPU (device). This allows the same code to be used in both contexts.\n\nLet’s look at how reset_state sets up the initial state:\n\nThis initialization strategy is clever:\n\nThe seed is split into the two key components key_[0] and key_[1]\n\nThe subsequence goes into the upper half of the counter ( counter_[2] and counter_[3] )\n\nThe offset (lower half of counter) starts at zero but can be set later via incr_n(offset)\n\nThis design allows for massive parallelism. Imagine running 1024 CUDA threads simultaneously:\nThread 0: subsequence=0, offset=0 → counter = [0, 0, 0, 0]\n\nThread 1: subsequence=1, offset=0 → counter = [0, 0, 1, 0]\n\nThread 2: subsequence=2, offset=0 → counter = [0, 0, 2, 0]\n\n...\n\nThread 1023: subsequence=1023, offset=0 → counter = [0, 0, 1023, 0]\n\nEach thread has a unique counter value from the start, so they all generate independent random sequences without any coordination.\n\nThe Core Algorithm: Single Round\n\nNow let’s examine the heart of the Philox algorithm—the single_round function:\n\nLet’s break this down step by step, mapping it to our earlier theoretical description:\n\nStep 1: Multiply and Split uint32_t lo0 = mulhilo32(kPhiloxSA, ctr[0], &hi0);\nuint32_t lo1 = mulhilo32(kPhiloxSB, ctr[2], &hi1);\n\nHere we multiply:\n\nctr[0] by kPhiloxSA (the constant 0xD2511F53)\n\nctr[2] by kPhiloxSB (the constant 0xCD9E8D57)\n\nThe mulhilo32 function performs the multiplication and splits the 64-bit result:\n\nReturns the low 32 bits ( lo0 or lo1 )\n\nStores the high 32 bits in the passed pointer ( hi0 or hi1 )\n\nLet’s look at mulhilo32 itself:\n\nThis function has two implementations:\n\nOn CUDA (GPU): Uses the intrinsic __umulhi which directly computes the high 32 bits of a multiplication. This is extremely fast on GPU hardware.\n\nOn CPU: Promotes both operands to 64 bits, multiplies them, then extracts high and low parts manually via shifting and casting.\n\nHere’s what happens mathematically:\n\n\\(\\begin{align}\n\n\\text{prod}_0 &= \\text{kPhiloxSA} \\times \\text{ctr}[0] = \\text{0xD2511F53} \\times \\text{ctr}[0] \\\\\n\n\\text{lo}_0 &= \\text{prod}_0\\text{ } \\And \\text{ 0xFFFFFFFF} \\quad \\text{(lower 32 bits)} \\\\\n\n\\text{hi}_0 &= \\text{prod}_0 \\gg 32 \\quad \\text{(upper 32 bits)}\n\n\\end{align}\n\n\\)\n\nStep 2: XOR and Permute ret[0] = hi1 ^ ctr[1] ^ in_key[0];\nret[1] = lo1;\nret[2] = hi0 ^ ctr[3] ^ in_key[1];\nret[3] = lo0;\n\nNotice the pattern:\n\nret[0]: Takes hi1 (high bits from second multiplication), XORs with ctr[1] and in_key[0]\n\nret[1]: Simply uses lo1 (low bits from second multiplication)\n\nret[2]: Takes hi0 (high bits from first multiplication), XORs with ctr[3] and in_key[1]\n\nret[3]: Simply uses lo0 (low bits from first multiplication)\n\nLet us visualize this transformation:\n\nThis permutation ensures that bits from different positions get mixed together in subsequent rounds.\n\nConstants: The Magic Numbers\n\nYou might wonder where these constants come from:\nstatic const uint32_t kPhilox10A = 0x9E3779B9; // Weyl sequence\nstatic const uint32_t kPhilox10B = 0xBB67AE85; // Weyl sequence\nstatic const uint32_t kPhiloxSA = 0xD2511F53; // Multiplier\nstatic const uint32_t kPhiloxSB = 0xCD9E8D57; // Multiplier\n\nWeyl sequence constants ( kPhilox10A and kPhilox10B ): These are derived from the golden ratio. The constants are:\n\n\\(\\begin{align}\n\n\\text{kPhilox10A} &= \\lfloor 2^{32} / \\phi \\rfloor = \\text{0x9E3779B9} \\\\\n\n\\text{kPhilox10B} &= \\lfloor 2^{32} / \\phi^2 \\rfloor = \\text{0xBB67AE85}\n\n\\end{align}\n\n\\)\n\nThe golden ratio has special properties that make it useful for distributing values uniformly. These constants are added to the key after each round to ensure different key material is used.\n\nMultiplier constants ( kPhiloxSA and kPhiloxSB ): These were carefully chosen through empirical testing to maximize statistical quality. They need to have good bit-mixing properties when multiplied with typical counter values.\n\nRunning Multiple Rounds\n\nThe rand function orchestrates running all rounds:\n\nThis is straightforward:\n\nRun n_rounds - 1 iterations where we:\n\nApply single_round to transform the counter\n\nUpdate the key by adding the Weyl constants\n\nApply one final round without updating the key\n\nBy default, PyTorch uses 10 rounds ( n_rounds = 10 ), which provides a good balance between performance and statistical quality.\n\nGenerating Random Numbers: The Operator\n\nThe operator () is what users call to get random numbers:\n\nThis function is clever in its efficiency:\n\nCheck if we need new random numbers: if(STATE == 0) checks if we’ve exhausted the previous batch. Remember, STATE cycles through 0, 1, 2, 3.\n\nGenerate a batch: When needed, it:\n\nRuns the full Philox algorithm via rand(counter, key, n_rounds)\n\nStores the result in output_ (four 32-bit random numbers)\n\nIncrements the counter for next time via incr()\n\nReturn next value: Grab the current position from output_, then advance STATE.\n\nThe line STATE = (STATE + 1) & 3 is a bit trick equivalent to STATE = (STATE + 1) % 4, using bitwise AND since 3 is binary 11.\n\nThis batching strategy is a significant performance optimization. Instead of running Philox for every random number, we run it once per four random numbers.\n\nCounter Increment Logic\n\nThe counter increment operations deserve special attention because they handle the 128-bit arithmetic correctly. Let’s start with the simple case:\n\nThis increments the 128-bit counter by 1. The logic is:\n\nIncrement counter_[0] (least significant 32 bits)\n\nIf it’s non-zero after increment, we’re done (no overflow)\n\nIf it overflowed to zero, carry to counter_[1]\n\nContinue propagating carries until we find a non-zero result\n\nThe more complex function is incr_n, which increments by an arbitrary 64-bit value:\n\nThis function is more intricate because it needs to:\n\nSplit the 64-bit increment n into nlo and nhi\n\nAdd nlo to counter_[0]\n\nDetect overflow by checking if counter_[0] < nlo (if the result is less than what we added, overflow occurred)\n\nIf overflow, increment nhi to carry over\n\nAdd nhi to counter_[1] and check for overflow again\n\nIf still overflowing, propagate to the upper 64 bits\n\nThe overflow detection counter_[0] < nlo is a standard technique in multi-precision arithmetic. After adding, if the result is less than one of the operands, an overflow must have occurred since we’re working with unsigned integers.\n\nConverting to Floating Point\n\nFor machine learning applications, we often need floating-point random numbers in the range [0, 1), while Philox gives us integers. So, PyTorch applies a conversion function:\n\nThis function is carefully designed:\n\nMask off sign bit: value & 0x7FFFFFFF clears the highest bit, giving us values from 0 to 2^31−1\n\nScale down: Multiplying by scale = 4.6566127342e-10 maps these integers to floats in [0, 1).\n\nThe scale factor is calculated as:\n\n\\(\\text{scale} = \\frac{1}{2^{31}} \\approx 4.6566127342 \\times 10^{-10}\\)\n\nWhy use only 31 bits instead of all 32? Because:\n\nWe want only positive values (for [0, 1) range)\n\nThe highest representable float less than 1.0 needs careful handling\n\nUsing 31 bits avoids potential rounding issues near 1.0\n\nNormal Distribution Generation\n\nThe randn function generates normally distributed random numbers using the Box-Muller transform:\n\nThe Box-Muller transform converts two uniform random variables U 1​, U 2​∼Uniform(0,1) into a normal random variable Z ∼N(0,1):\n\n\\(Z = \\sqrt{-2 \\ln U_1} \\cos(2\\pi U_2)\\)\n\nMemory Layout and Efficiency\n\nOne of the beauties of this implementation is how compact the state is. Each philox_engine instance requires:\ncounter_: 4 × 4 bytes = 16 bytes\n\noutput_: 4 × 4 bytes = 16 bytes\n\nkey_: 2 × 4 bytes = 8 bytes\n\nSTATE: 4 bytes = 4 bytes\n\nTotal = 44 bytes\n\nThis is tiny! On a GPU, you could have millions of these generators running in parallel, each consuming only 44 bytes. In comparision, traditional RNGs can take kilobytes of state per instance.\n\nSummary\n\nIn this article, we explored Philox, a counter-based PRNG designed for parallel computing environments.
