Execline scripts are often used as an alternative to shellscripts ( /bin/sh or /bin/bash ), especially but not exclusively when used in run scripts of process supervisors.\n\nExecline scripts distinguishes themselves from /bin/bash or /bin/sh on smallish, straightforward scripts, by using less RAM, not staying resident in memory while execution proceeds, and having a more consistent syntax. Like /bin/bash and /bin/sh, execline delivers unacceptable performance when executing tight loops with over 100 iterations. Use Perl/Python/Ruby/Lua/C, or other \"real languages\", in those situations.\n\nLikewise, if the logic gets too hairy, or if you find yourself needing data structures, use \"real languages\". But for relatively straightforward stuff, execline is in my opinion a more straightforward syntax than Bash or /bin/sh, and might save you some RAM.\n\nExecline is a different breed. It's not natively procedural, but instead tends to use the first word of the command line as a program, and use the rest of the words as arguments to that program. It's important to realize that execline uses the exec function a lot.\n\nNOTE:\n\nIt actually uses the execve function, but I'm speaking of it as a concept, so I use the word \"exec\".\n\nThe exec function works kinda sorta like the following, as a gross generalization not applicable to real life:\n\nProgram A calls exec on program b, with args c, d and e\n\nProgram B starts running, with args c, d and e, with the PID that program A used to have. Program A has given up all its RAM.\n\nYou can have a whole chain of execs, where A execs B, which execs C, which execs D, and when all is said and done, D is running with the PID of A. Note that A, B, C and D can do some work before execing their successor.\n\nThis type of behavior is used quite a bit in execline. Some people call it \"chaining\". Some call it \"Bernstein Chaining\". Whatever it's called, the principle is that each program execs the next, giving up all its RAMin the process, which in many cases saves on RAM.\n\nIt would be counterproductive to discuss such chaining further here: Instead use this document as a tutorial. Just be aware that this is not procedural programming, but instead it's based on chaining.\n\nWARNING!\n\nMany portions of this document describe program processes with their use of exec and fork, etc. These descriptions are approximations for the purpose of introductory understanding: They are not completely accurate. For the absolutely accurate story, you'll need to look at the source code at https://github.com/skarnet/skalibs and https://github.com/skarnet/execline, and if you want to go really deep,the posix source code implementing the functions in /usr/include/spawn.h.\n\nIn addition, there are elements of the execlineb program that I don't even touch: Parsing, substitution, and argv assembly. I don't think you need intimate knowledge of these things to write casual execline code. Beware, however, if you're speaking to an execline/execlineb expert, you'll rightfully be called wrong for restating some of these descriptions, even though these descriptions helped you learn to code execline.\n\nHello World\n\nHere's the simplest Hello World execline program:\n\n#!/bin/execlineb\necho Hello World!\n\nPermission it executable, run it, and you'll see exactly what you expect. This is the simplest possible execline program: It's deceptively simple, as you'll soon see.\n\nIf you look at the shebang line (the line starting with #! ), it references execlineb, not execline. The explanation follows:\n\nexecline is the language.\n\nexeclineb is the interpreter to run execline programs.\n\nWhat happened in this program is that the execline program exec'ed the echo program with the arguments Hello and World!.\n\nHello World Fail\n\nLet's try to add a second line of output, the obvious way, to this Hello World program, and watch it fail:\n\n#!/bin/execlineb\necho Hello World!\necho 2\n\nThe preceding should print \"Hello World!\" on one line, and \"2\" on the next line, right? Well, ummm, here's what really happens:\n\n[slitt@mydesk test]$./hello.e\nHello World! echo 2\n[slitt@mydesk test]$\n\nIn the preceding output, we see that not only did it print just one line, but the second instance of the word \"echo\" was just taken as one more piece of data to be printed. The reasons are as follows:\n\nNewlines in execline code are no different from spaces.\n\nThe first word (first echo ) is considered\nby execlineb a program,\nand every word to its right ( Hello, World, echo and 2 ) is considered an argument to the program represented by the first word.\n\nSo what happened is that execline exec'ed echo with arguments Hello, World, echo and 2. Which explains the exact results obtained.\n\nThe next section lays out the right way to print these two lines.\n\nUsing the foreground Command\n\nThe following execline program prints the two lines:\n#!/bin/execlineb\nforeground { echo Hello World! }\necho 2\n\nThe preceding code produces the following output:\n\n[slitt@mydesk test]$./hello.e\nHello World!\n2\n[slitt@mydesk test]$\n\nAbout the foreground command\n\nThe foreground command is actually a special executable program that ships with execlineb. When run within execlineb, foreground runs exactly two programs, one at a time. It forks the first program, and when that program exits, it execs into the second program. The following is a stepwise description of what the foreground program does:\n\nForks the words in the curly braces as a command with args, with the first word in the curly braces being the program and the rest of the words being command line arguments.\n\nWaits for the program just run to exit.\n\nExecs into the program after closing brace, with the first word after the curly brace used as a program, and the rest of the words used as arguments.\n\nIn the execline language, those curly braces are not punctuation, they're separate words. In the execline language words are separated by whitespace, any kind of whitespace: One space, two spaces, one or more tabs, or one or more newlines: All these kinds of space are the same to execline: They separate words.\n\nThe curly brace words are used to delineate a block. The typical use of a block is to group a program with all its command line arguments.\n\nLet's say you didn't use braces to define a block:\nforeground a b c d e f\n\nIn the preceding, you know for a fact that a is a program to be executed. But what is b? It could be the second program, or it could be an argument to the first. The exact same thing could be said for c d and e. In order to discern which args go with the first program, you need a block.\n\nLet's briefly discuss formatting, using shorter echo commands. I formatted mine as:\n\n#!/bin/execlineb\nforeground { echo 1 }\necho 2\n\nBecause newlines are just space in execline, the preceding could be written:\n\n#!/bin/execlineb\nforeground { echo 1 } echo 2\n\nIn fact, it is written that way in much existing documentation. For reasons pointed out in the next section, I prefer putting a newline in the execline source code between the first and second program to be run by foreground.\n\nRunning Three Programs Consecutively\n\nThe preceding section illustrated how to use foreground to run two programs consecutively. But how do you run three programs consecutively? It turns out quite simple if you format it right. The following is an example, this time using the short echo commands:\n\n#!/bin/execlineb\nforeground { echo 1 }\nforeground { echo 2 }\necho 3\n\nThe preceding code outputs the following:\n\n[slitt@mydesk test]$./test.e\n1\n2\n3\n[slitt@mydesk test]$\n\nThis code could have been formatted on one line:\n\n#!/bin/execlineb\nforeground { echo 1 } foreground { echo 2 } echo 3\n\nYour mileage may vary, but to me, the 1 line approach makes a mistake with quote inclusion more likely. I prefer the line by line:\n\n#!/bin/execlineb\nforeground { echo 1 }\nforeground { echo 2 }\necho 3\n\nThe preceding program runs as follows: Execline forks foreground, which forks echo 1, after the termination of which it execs the second foreground, which forks echo 2, and after the termination of echo 2, the second foreground execs echo 3. To really study what forks what, what execs what, and what stays in memory and what doesn't, I suggest studying the following program in the ps axjf | less command:\n\n#!/bin/execlineb\nforeground { gnumeric }\ninkscape\n\nIf you don't have either of the preceding two GUI programs, use almost any GUI program. Don't use gvim, because it double-forks itself and instantly returns control to whatever called it.\n\nVariables\n\nFirst try the following:\n\n#!/bin/execlineb\n\ndefine myvar Hello\necho $myvar\n\nWhen you run the preceding, you get exactly what you expect. The define statement assigns \"Hello\" to name myvar. Very much like Bash, you use a dollar sign when using a variable, but not when assigning a value to it.\n\nThe define command/program works as follows:\n\nIt replaces the following word with the word or group that follows that (sort of like assigning the second word to the first)\n\nIt execs the third word or group\n\nNow try the following:\n#!/bin/execlineb\n\ndefine myvar Hello World!\necho $myvar\n\nThe preceding errors out, producing the following output:\n[slitt@mydesk ~]$./junk.e\ndefine: fatal: unable to exec World!: No such file or directory\n[slitt@mydesk ~]$\n\nThe explanation of the preceding error is that define assigned \"Hello\" to myvar, and then ran the next word, \"World!\".\n\nSo try to fix it as follows:\n#!/bin/execlineb\n\ndefine myvar { Hello World! }\necho $myvar\n\nRunning the preceding produces the same error message. With both \"Hello\" and \"World!\" in a group, one would think it would be taken as a single element, but for some reason that doesn't happen.\n\nThe correct technique is as follows:\n#!/bin/execlineb\n\ndefine myvar \"Hello World!\"\necho $myvar\n\nThe preceding produces a \"Hello World!\" output.\n\nNow watch the following fail:\n#!/bin/execlineb\n\ndefine myvar \"Hello World!\"\necho \"$myvar to everyone!\"\n\nThe preceding code produces the following output:\n[slitt@mydesk ~]$./test.e\n$myvar to everyone!\n[slitt@mydesk ~]$\n\nFix it by putting curly braces directly around the variable name, as follows:\n\n#!/bin/execlineb\n\ndefine myvar \"Hello World!\"\necho \"${myvar} to everyone!\"\n\nThe preceding code produces the desired output:\n\n[slitt@mydesk ~]$./test.e\nHello World! to everyone!\n[slitt@mydesk ~]$\n\nWhenever you want to put a variable substitution, like $myvar or $1 in a doublequoted string, it's important to protect that variable substitution with curly braces, like ${myvar} or ${1}. Otherwise the output will contain only the variable name preceded by a dollar sign.\n\nIt's important to differentiate, in your mind, the difference between using curly braces to protect a variable name, in which case the braces are flush up against the name, as opposed to use of curly braces to form a group, in which case the curly braces are standalone words, surrounded on both sides by space.\n\nRemember, when a variable prints as its name rather than its value, the two things to check are:\n\nHave I protected the variable name with braces?\n\nHave I preceded the value's use with the necessary importas commands?\n\nAs far as I know, it's never a problem to brace-protect a variable, so it might be a good idea to always protect as a habit.\n\nDeliberate Directory Fail\n\nCreate the following test.e:\n\n#!/bin/execlineb\nls /etc/f*\n\nLook what happens when you run it:\n\n[slitt@mydesk test]$./test.e\nls: cannot access '/etc/f*': No such file or directory\n[slitt@mydesk test]$\n\nIt turns out that execline cannot directly handle the stars and question marks used in file globbing. Execline itself could have found /etc/fstab, but not /etc/f* To do file globbing within execline, you need to use the elglob program.\n\nFile Globbing\n\nThe preceding section demonstrated that yoou can't directly use file wildcards in execline. So you need to use the elglob program, as follows:\n\n#!/bin/execlineb\nelglob fil /etc/f* echo ${fil}\n\nHere's the explanation. elglob is an execline program, as is foreground. elglob requires a variable name as its first arg, the file pattern as its second arg, and a command as its third arg:\n\nelglob is arg0\n\nfil is arg1, the name of a variable whose purpose is to hold the result of the file globbing, in other words, to hold a list of files.\n\n/etc/f* is arg2, the file pattern.\n\nAll the rest is the program to be run. Notice that the value of fil, ${fil}, is available to the program to be run.\n\nI won't show the output because it all comes on one line, but the output is every file in /etc that begins with f, separated by spaces.\n\nAn equivalent but easier to handle format for the preceding program is the following:\n\n#!/bin/execlineb\nelglob fil /etc/f*\necho ${fil}\n\nSomething handy is to put the variable value in quotes, and put a newline at the end:\n#!/bin/execlineb\n#ls /etc/a*\nelglob fil /etc/f*\necho \"${fil}\\n\"\n\nNotice the only change in the immediately preceding version was putting a newline behind the variable value, and putting the variable and value combination in quotes. This changes the output from a one liner to the following:\n\n[slitt@mydesk test]$./test.e\n/etc/freepats\n/etc/fppkg\n/etc/fstab.bup\n/etc/fstab.org\n/etc/fonts\n/etc/fstab\n/etc/fpc.cfg\n/etc/fail2ban\n/etc/fuse.conf\n/etc/fppkg.cfg\n\n[slitt@mydesk test]$\n\nEverything seems cool except for the single space prepended to each but the first line, and the extra newline at the end of the list. Ways to fix that are described later in this document, but for the time being, this is a great way to get quick info.\n\nOne other problem is what happens if no matching files are found. My /etc has no files beginning with \"y\", so let's change the pattern to /etc/y*:\n\n#!/bin/execlineb\n#ls /etc/a*\nelglob fil /etc/y*\necho \"${fil}\\n\"\n\nThe output follows:\n\n[slitt@mydesk test]$./test.e\n/etc/y*\n\n[slitt@mydesk test]$\n\nThe preceding is just plain inaccurate. I have no file in /etc named y*. To fix this problem, you need to add -0 to the elglob command, as shown in the following version:\n\n#!/bin/execlineb\nelglob -0 fil /etc/y*\necho \"${fil}\\n\"\n\nThe preceding does the right thing whether there are any matches or not.\n\nThere are many other command line options for elglob. You can find those in the official documentation at https://skarnet.org/software/execline/.\n\nIntro to Loops\n\nThis section barely scratches the surface of loops. It's a loop Hello World, if you will. Some things, such as the necessity of importas inside a loop, aren't readily evident in most execline documentation.\n\nCreate the following test.e:\n#!/bin/execlineb\nforx myvar { a b c d e }\nimportas myvar_inside myvar\necho \"${myvar_inside}\"\n\nThe preceding code produces the following output:\n\n[slitt@mydesk test]$./test.e\na\nb\nc\nd\ne\n[slitt@mydesk test]$\n\nRe-showing the code, so it can be discussed line by line...\n\n#!/bin/execlineb\nforx myvar { a b c d e }\nimportas myvar_inside myvar\necho \"${myvar_inside}\"\n\nforx is a simple execline loop to iterate over a list. At its simplest it takes three args:\n\nThe variable to hold the current iteration's list element\n\nThe list itself, in a group\n\nThe program to be exec'ed on each iteration\n\nmyvar is a variable that holds a copy of each iteration's element of the list.\n\n{ a b c d e } is the list, contained in a group.\n\nVariable myvar holds the list element, but it is not recognized inside the iteration. The importas program makes it available, under the same or a different name (in this case different). The importas program takes exactly three arguments:\n\nThe variable recognized the iteration\n\nThe variable recognized outside the iteration\n\nThe program to be exec'ed, in this case echo ${\"myvar_inside}\n\nNOTE:\n\nThe indentation of the importas and echo lines isn't necessary for function: I did it to make it clear that those two lines were exec'ed by the forx line on each iteration.\n\nCommand Line Arguments\n\nFirst, create the following test jig shellscript, called testjig.sh, to display the executable path, as one directory per line, and then all arguments sent to it:\n\n#!/bin/sh\necho ==================\necho $PATH | sed -e\"s/:/:\\\\n/g\"\necho ==================\necho $@\necho ==================\n\nNext, create the following path prepender execline script, called prepath.e, whose arg1 is the directory or series of directories to put on the front of the path, and whose arg2 is the program for the prepender to execute (after prepending the path, and all additional arguments are arguments for the program executed by the prepender:\n#!/bin/execlineb -s2\n\nimportas OLDPATH PATH\nexport PATH \"${1}:${OLDPATH}\"\nimportas PATH PATH\n${2} ${@}\n\nExamine the preceding execline script:\n\nThe -s2 argument to execlineb means to take incoming arguments arg1 and arg 2, assign them to $1 and $2, and take the rest of the arguments and assign them to $@. Without this execlineb argument, or a similar one beyond the scope of this document, it would be impossible to access the incoming command line arguments.\n\nIncoming $1 is the directory(s) to be prepended to the path, while $2 is the program to be run under the new path.\n\nBe sure to use curly braces to protect variables $1 and OLDPATH, which should be written $1 and $2 respectively.\n\nThe line with importas PATH PATH makes\nthe new path available to the rest of the execline program. Without\nit, the rest of the program would still be operating with the old\npath.\n\n$2 $@ runs the program described by incoming $2, and all further arguments are arguments to that program.\n\nIntro to Branching\n\nThe execline language has four command programs for branching:\n\nif\n\nifelse\n\nifthenelse\n\nifte\n\nThis section discusses the first three. Start with an example showcasing if:\n\nif\n\nThe best use of if is when you want the script to continue only if the test is true. Here's a basic example:\n\n#!/bin/execlineb -s1\nif { test $1 -gt 10 }\necho ${1} is Greater than ten\n\nRun the preceding with an argument of 4, then an argument of 14, and notice it prints on the latter but not on the former. If the program run as a test delivers up something other than 0, all execution of the script stops. Observe also that with the use of foreground and other things, this script could be modified to do five or ten more things after the program being tested returns a zero.\n\nifelse\n\nBest use of ifelse is when you want to do one thing if true, another thing if false, but either way, execution ends there. Nothing following the ifelse is executed. Or if you want the script to stop after executing the program to be run if true, but to continue if false. A simple example follows:\n\n#!/bin/execlineb -s1\nifelse { test $1 -gt 10 }\n{ echo ${1} is Greater than ten }\necho ${1} is Not greater than ten\n\nIn the preceding code, the program to be tested is still the first one, but now it is followed by two programs: One in a brace block to be executed if the first one returned 0, and one without a brace block to be executed if the first one returned something other than 0. Try it with command line argument 4, and then 14, and see what happens.\n\nifthenelse\n\nUse ifthenelse when you want execution to continue beyond the test and whatever programs the test runs. One typical use is when you want to set some variables whose values depend on the test, but then the rest of the script will use those variables. The following is an example:\n\n#!/bin/execlineb -s1\nifthenelse { test $1 -gt 10 }\n{ echo ${1} is Greater than ten }\n{ echo ${1} is Not greater than ten }\necho Either way, life is good.\n\nOnce again, try the preceding first with an arg of 4, then an arg of 14, and you'll get the picture.\n\nJust for fun, check this out:\n\n#!/bin/execlineb -s1\nifelse { test $1 -gt 10 }\n{ echo ${1} is Greater than ten }\nifthenelse { test $1 -eq 10 }\n{ echo ${1} is Equal to ten }\n{ echo ${1} is Less than ten }\necho Whichever way, life is good.\n\nImitation Case Statement\n\nThis subsection starts with an immitation case statement built with ifthenelse commands, and then presents a possibly simpler version created with one foreground command and a bunch of ifelse commands. Let's start with the ifthenelse version. Just as a review, ifthenelse looks like the following:\n\nifthenelse { testpgm } { truepgm } { falsepgm }\n\nWhich, because newline is just another space in execline, is equivalent to:\n\nifthenelse { testpgm }\n{ truepgm }\n{ falsepgm }\n\nIf testpgm returns 0, that's considered true. If it returns a different number that's considered false. Several ifthenelse commands can be nested to produce a pseudo case statement that's really quite convenient and readable. See the following example:\n\n#!/bin/execlineb -s1\n\nifthenelse -X { test $1 -eq 1 }\n{ echo ${1} equals 1 }\n{ ifthenelse -X { test $1 -eq 2 }\n{ echo ${1} equals 2 }\n{ ifthenelse -X { test $1 -eq 3 }\n{ echo ${1} equals 3 }\n{ ifthenelse -X { test $1 -lt 1 }\n{ echo ${1} less than 1 }\n{ ifthenelse -X { test $1 -gt 3 }\n{ echo ${1} greater than 3 }\n{ echo ${1} internal error } # ending default\n} } } }\necho Continuing on with rest of program.\n\nThe preceding code produces the following first-line results, depending on argument:\n\narg1 1st line\n0 0 less than 1\n1 1 equals 1\n2 2 equals 2\n3 3 equals 3\n4 4 greater than 3\nnan internal error\n\nThe second line is always \"Continuing on with rest of program.\".\n\nKeep the following tips in mind to make these case statements easy:\n\nUse ifthenelse on all the cases choices. Much less confusing that way.\n\nAll but the first ifthenelse line has an opening brace and space before it.\n\nIndent the groups that don't start with ifthenelse. This makes things easier to keep track of, especially when case choices consume more than one line.\n\nThe final ifthenelse line is followed by two complete groups, not just one.\n\nAfter all the lines in this contrived case statement, insert a line with ending braces numbered one ending brace for each ifthenelse line that began with an opening brace and space.\n\nThe -X argument in an ifthenelse makes a program performing a test return false (nonzero) if it crashes. That way, if one crashes, it falls through to the next, and in the preceding example, all the way to the \"internal error\" result.\n\nAn arguably easier way to make a pseudo case statement is to use a foreground command with a bunch of ifelse commands. Such a construction relieves one of counting braces. The following source code produces the exact same result as the preceding execline program:\n\n#!/bin/execlineb -s1\n\nforeground {\nifelse -X { test $1 -eq 1 } { echo ${1} equals 1 }\nifelse -X { test $1 -eq 2 } { echo ${1} equals 2 }\nifelse -X { test $1 -eq 3 } { echo ${1} equals 3 }\nifelse -X { test $1 -lt 1 } { echo ${1} less than 1 }\nifelse -X { test $1 -gt 3 } { echo ${1} greater than 3 }\necho ${1} internal error # ending default\n} # Foreground ending brace\necho Continuing on with rest of program.\n\nNotice you don't have to insert a row of ending braces: You just put one ending brace after the case's default. The preceding looks great if all the \"then clauses\" are short. If some are long enough to require multiple lines, the preceding can be reformatted as follows:\n\n#!/bin/execlineb -s1\n\nforeground {\nifelse -X { test $1 -eq 1 }\n{\necho ${1} equals 1\n}\nifelse -X { test $1 -eq 2 }\n{\necho ${1} equals 2\n}\nifelse -X { test $1 -eq 3 }\n{\necho ${1} equals 3\n}\nifelse -X { test $1 -lt 1 }\n{\necho ${1} less than 1\n}\nifelse -X { test $1 -gt 3 }\n{\necho ${1} greater than 3\n}\necho ${1} internal error # ending default\n} # Foreground ending brace\necho Continuing on with rest of program.\n\nNumerical Looping and forbacktickx\n\nJust like /bin/sh, large scale numerical looping is where you run into time trouble with execline. Consider the following numloop.e:\n\n#!/bin/execlineb -s1\n\nforbacktickx i { seq $1 }\nimportas i i\necho $i\n\nIn the preceding code, the -s1 enables the capture of the first command line argument, which of course is the number to loop until. The importas i i enables the use of $1 within the forbacktickx construct. This code enables time testing, as follows:\n\n[slitt@mydesk ~]$ time./numloop.e 10 > /dev/null\n\nreal 0m0.056s\nuser 0m0.013s\nsys 0m0.024s\n[slitt@mydesk ~]$ time./numloop.e 100 > /dev/null\n\nreal 0m0.417s\nuser 0m0.082s\nsys 0m0.233s\n[slitt@mydesk ~]$ time./numloop.e 1000 > /dev/null\n\nreal 0m4.797s\nuser 0m0.888s\nsys 0m2.338s\n[slitt@mydesk ~]$ time./numloop.e 10000 > /dev/null\n\nreal 0m47.838s\nuser 0m8.159s\nsys 0m23.914s\n[slitt@mydesk ~]$ time seq 10000 > /dev/null\n\nreal 0m0.002s\nuser 0m0.000s\nsys 0m0.001s\n[slitt@mydesk ~]$\n\nThe preceding time tests are telling. The time execline takes to complete the task seems pretty much proportional to how many numbers you loop through. Looping 100 times takes a half second: No problem. Looping 1000 times takes almost 5 seconds, which is acceptable if it doesn't happen often and if a prompt warns the user to expect a delay. At this point you might consider trading execline for a language like Python, Lua, Ruby or Perl. Looping 10000 times takes almost 50 seconds: Completely unacceptable: Use a different language.\n\nHandling File Directories\n\nIf you don't need file globbing, execline is trivial:\n\n#!/bin/execlineb\nls /etc\n\nAs mentioned before, when globbing is necessary, the elglob program is necessary. The following is an elglob enabled script that's sophisticated enough to do work on each file found:\n\n#!/bin/execlineb\nelglob -0 files /etc/f*\nforx file { ${files} }\nimportas file file\necho \"File: ${file}\"\n\nThe preceding code produces the following output:\n\n[slitt@mydesk test]$./test.e\nFile: /etc/freepats\nFile: /etc/fppkg\nFile: /etc/fstab.bup\nFile: /etc/fstab.org\nFile: /etc/fonts\nFile: /etc/fstab\nFile: /etc/fpc.cfg\nFile: /etc/fail2ban\nFile: /etc/fuse.conf\nFile: /etc/fppkg.cfg\n[slitt@mydesk test]$\n\nPipelining\n\nThe following works in bash type shellscripts, but not in execline:\nls -1t | head -n5\n\nInstead, you need to use execline's pipeline command program. The following execline script prints the top 20 biggest files in the /etc directory (not tree):\n\n#!/usr/bin/execlineb\n\npipeline { ls -l /etc } # all entries in /etc\npipeline { grep -v \"^d\" } # no directories\npipeline { sed -re \"s|\\\\s+| |g\" } # turn multiple space\n# into single\npipeline { cut -d \" \" -f 5,9- } # Field 5, and 9 and above\n# fld 5 is size, 9- fname\npipeline { sort -rn } # Sort numerically reverse\npipeline { head -n 20 } # Only the top 20\ncat -n # Add rank numbers on left\n\nBy the way, that double backslash in the sed filter is necessary so that sed reads it as a single backslash. The \"9-\" in the cut filter assures whole filenames printed if somebody unwisely puts spaces in filenames.\n\nPerhaps you need to do other tasks after all this pipelining. You can use a foreground command program to accomplish this, making the whole bunch of pipelines a single group, as follows:\n#!/usr/bin/execlineb\n\nforeground {\npipeline { ls -l /etc } # all files in /etc\npipeline { grep -v \"^d\" } # no directories\npipeline { sed -re \"s|\\\\s+| |g\" } # turn multiple\n# space into single\npipeline { cut -d \" \" -f 5,9- } # Field 5, and 9 and above\npipeline { sort -rn } # Sort numerically reverse\npipeline { head -n 20 } # Only the top 20\ncat -n # Add rank numbers on left\n}\necho \"I finished\"\n\nA Real Life Example\n\nMy UMENU2 software is in a directory tree of its own, not on the executable path. My UMENU2 software takes an argument telling it which of myh many menus to display, and it has a -t option that makes it terminate when it runs a program, instead of doing more menuing. The shellscript to run UMENU2, in the terminating mode, on a menu defined by arg1, looks like the following:\n\n#!/bin/sh\n\ncd ~/umenu2/prog\n./umenu.py -t $1\n\nThe execline script that does exactly the same thing, if called from bash or /bin/sh follows:\n\n#!/bin/execlineb -s1\n\nimportas HOME HOME\ncd ${HOME}/umenu2/prog\n./umenu.py -t $1\n\nThe reason the execline script is one line longer is that execline has no shorthand for the home directory similar to bash's ~, so it has to import the calling shell's $HOME. And if it's expected to be called from a shell-less situation (boot, for instance), or with a shell substantially different from bash (csh, for instance), then you can't even count on the environment variable $HOME, so it gets a little longer:\n\n#!/bin/execlineb -s1\n\nbacktick -n username { whoami }\nimportas username username\nbacktick -n myhome { homeof ${username} }\nimportas myhome myhome\ncd ${myhome}/umenu2/prog\n./umenu.py -t $1\n\nThe reason the preceding execline script is longer than the one before is that one can't count on bash's $HOME environment variable, so it must be deduced by Linux command whoami followed by execline command homeof.\n\nPointers to Execline Documentation\n\nThe document you're now reading can take you only so far.
