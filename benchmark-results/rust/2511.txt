We wrote this document as a way of identifying\nsome general principles and specific techniques that we use when doing this sort\nof work, and tried to pick illustrative source code changes (change lists, or\nCLs) that provide examples of the various approaches and techniques. Most of the\nconcrete suggestions below reference C++ types and CLs, but the general\nprinciples apply to other languages. The document focuses on general performance\ntuning in the context of a single binary, and does not cover distributed systems\nor machine learning (ML) hardware performance tuning (huge areas unto\nthemselves). We hope others will find this useful.\n\nMany of the examples in the document have code fragments that demonstrate the\ntechniques (click the little triangles!). Note that some of these\ncode fragments mention various internal Google codebase abstractions. We have\nincluded these anyway if we felt like the examples were self-contained enough to\nbe understandable to those unfamiliar with the details of those abstractions.\n\nThe importance of thinking about performance\n\nKnuth is often quoted out of context as saying premature optimization is the\nroot of all evil. The\nfull quote reads: “We\nshould forget about small efficiencies, say about 97% of the time: premature\noptimization is the root of all evil. Yet we should not pass up our\nopportunities in that critical 3%.” This document is about that critical\n3%, and a more compelling quote,\nagain from Knuth, reads:\n\nThe improvement in speed from Example 2 to Example 2a is only about 12%, and\nmany people would pronounce that insignificant. The conventional wisdom shared\nby many of today’s software engineers calls for ignoring efficiency in the\nsmall; but I believe this is simply an overreaction to the abuses they see\nbeing practiced by penny-wise-and-pound-foolish programmers, who can’t debug\nor maintain their “optimized” programs. In established engineering disciplines\na 12% improvement, easily obtained, is never considered marginal; and I\nbelieve the same viewpoint should prevail in software engineering. Of course I\nwouldn’t bother making such optimizations on a one-shot job, but when it’s a\nquestion of preparing quality programs, I don’t want to restrict myself to\ntools that deny me such efficiencies.\n\nMany people will say “let’s write down the code in as simple a way as possible\nand deal with performance later when we can profile”. However, this approach is\noften wrong:\n\nIf you disregard all performance concerns when developing a large system,\nyou will end up with a flat profile where there are no obvious hotspots\nbecause performance is lost all over the place. It will be difficult to\nfigure out how to get started on performance improvements.\n\nIf you are developing a library that will be used by other people, the\npeople who will run into performance problems will be likely to be people\nwho cannot easily make performance improvements (they will have to\nunderstand the details of code written by other people/teams, and have to\nnegotiate with them about the importance of performance optimizations).\n\nIt is harder to make significant changes to a system when it is in heavy\nuse.\n\nIt is also hard to tell if there are performance problems that can be solved\neasily and so we end up with potentially expensive solutions like\nover-replication or severe overprovisioning of a service to handle load\nproblems.\n\nInstead, we suggest that when writing code, try to choose the faster alternative\nif it does not impact readability/complexity of the code significantly.\n\nEstimation\n\nIf you can develop an intuition for how much performance might matter in the\ncode you are writing, you can make a more informed decision (e.g., how much\nextra complexity is warranted in the name of performance). Some tips on\nestimating performance while you are writing code:\n\nIs it test code? If so, you need to worry mostly about the asymptotic\ncomplexity of your algorithms and data structures. (Aside: development cycle\ntime matters, so avoid writing tests that take a long time to run.)\n\nIs it code specific to an application? If so, try to figure out how much\nperformance matters for this piece of code. This is typically not very hard:\njust figuring out whether code is initialization/setup code vs. code that\nwill end up on hot paths (e.g., processing every request in a service) is\noften sufficient\n\nIs it library code that will be used by many applications? In this case it\nis hard to tell how sensitive it might become. This is where it becomes\nespecially important to follow some of the simple techniques described in\nthis document. For example, if you need to store a vector that usually has a\nsmall number of elements, use an absl::InlinedVector instead of std::vector.\nSuch techniques are not very hard to follow and don’t add any non-local\ncomplexity to the system. And if it turns out that the code you are writing\ndoes end up using significant resources, it will be higher performance from\nthe start. And it will be easier to find the next thing to focus on when\nlooking at a profile.\n\nYou can do a slightly deeper analysis when picking between options with\npotentially different performance characteristics by relying on\nback of the envelope calculations.\nSuch calculations can quickly give a very rough estimate of the performance of\ndifferent alternatives, and the results can be used to discard some of the\nalternatives without having to implement them.\n\nHere is how such an estimation might work:\n\nEstimate how many low-level operations of various kinds are required, e.g.,\nnumber of disk seeks, number of network round-trips, bytes transmitted etc.\n\nMultiply each kind of expensive operation with its rough cost, and add the\nresults together.\n\nThe preceding gives the cost of the system in terms of resource usage. If\nyou are interested in latency, and if the system has any concurrency, some\nof the costs may overlap and you may have to do slightly more complicated\nanalysis to estimate the latency.\n\nThe following table, which is an updated version of a table from a\n2007 talk at Stanford University\n(video of the 2007 talk no longer exists, but there is a\nvideo of a related 2011 Stanford talk that covers some of the same content )\nmay be useful since it lists the types of operations to consider, and their\nrough cost:\n\nL1 cache reference 0.5 ns\nL2 cache reference 3 ns\nBranch mispredict 5 ns\nMutex lock/unlock (uncontended) 15 ns\nMain memory reference 50 ns\nCompress 1K bytes with Snappy 1,000 ns\nRead 4KB from SSD 20,000 ns\nRound trip within same datacenter 50,000 ns\nRead 1MB sequentially from memory 64,000 ns\nRead 1MB over 100 Gbps network 100,000 ns\nRead 1MB from SSD 1,000,000 ns\nDisk seek 5,000,000 ns\nRead 1MB sequentially from disk 10,000,000 ns\nSend packet CA->Netherlands->CA 150,000,000 ns\n\nThe preceding table contains rough costs for some basic low-level operations.\nYou may find it useful to also track estimated costs for higher-level operations\nrelevant to your system. E.g., you might want to know the rough cost of a point\nread from your SQL database, the latency of interacting with a Cloud service, or\nthe time to render a simple HTML page. If you don’t know the relevant cost of\ndifferent operations, you can’t do decent back-of-the-envelope calculations!\n\nExample: Time to quicksort a billion 4 byte numbers\n\nAs a rough approximation, a good quicksort algorithm makes log(N) passes over an\narray of size N. On each pass, the array contents will be streamed from memory\ninto the processor cache, and the partition code will compare each element once\nto a pivot element. Let’s add up the dominant costs:\n\nMemory bandwidth: the array occupies 4 GB (4 bytes per number times a\nbillion numbers). Let’s assume ~16GB/s of memory bandwidth per core. That\nmeans each pass will take ~0.25s. N is ~2^30, so we will make ~30 passes, so\nthe total cost of memory transfer will be ~7.5 seconds.\n\nBranch mispredictions: we will do a total of N*log(N) comparisons, i.e., ~30\nbillion comparisons. Let’s assume that half of them (i.e., 15 billion) are\nmispredicted. Multiplying by 5 ns per misprediction, we get a misprediction\ncost of 75 seconds. We assume for this analysis that correctly predicted\nbranches are free.\n\nAdding up the previous numbers, we get an estimate of ~82.5 seconds.\n\nIf necessary, we could refine our analysis to account for processor caches. This\nrefinement is probably not needed since branch mispredictions are the dominant\ncost according to the analysis above, but we include it here anyway as another\nexample. Let’s assume we have a 32MB L3 cache, and that the cost of transferring\ndata from L3 cache to the processor is negligible. The L3 cache can hold 2^23\nnumbers, and therefore the last 22 passes can operate on the data resident in\nthe L3 cache (the 23rd last pass brings data into the L3 cache and the remaining\npasses operate on that data.) That cuts down the memory transfer cost to 2.5\nseconds (10 memory transfers of 4GB at 16GB/s) instead of 7.5 seconds (30 memory\ntransfers).\n\nExample: Time to generate a web page with 30 image thumbnails\n\nLet’s compare two potential designs where the original images are stored on\ndisk, and each image is approximately 1MB in size.\n\nRead the contents of the 30 images serially and generate a thumbnail for\neach one. Each read takes one seek + one transfer, which adds up to 5ms for\nthe seek, and 10ms for the transfer, which adds up to 30 images times 15ms\nper image, i.e., 450ms.\n\nRead in parallel, assuming the images are spread evenly across K disks. The\nprevious resource usage estimate still holds, but latency will drop by\nroughly a factor of K, ignoring variance (e.g, we will sometimes get unlucky\nand one disk will have more than 1/Kth of the images we are reading).\nTherefore if we are running on a distributed filesystem with hundreds of\ndisks, the expected latency will drop to ~15ms.\n\nLet’s consider a variant where all images are on a single SSD. This changes\nthe sequential read performance to 20µs + 1ms per image, which adds up to\n~30 ms overall.\n\nMeasurement\n\nThe preceding section gives some tips about how to think about performance when\nwriting code without worrying too much about how to measure the performance\nimpact of your choices. However, before you actually start making improvements,\nor run into a tradeoff involving various things like performance, simplicity,\netc. you will want to measure or estimate potential performance benefits. Being\nable to measure things effectively is the number one tool you’ll want to have in\nyour arsenal when doing performance-related work.\n\nAs an aside, it’s worth pointing out that profiling code that you’re unfamiliar\nwith can also be a good way of getting a general sense of the structure of the\ncodebase and how it operates. Examining the source code of heavily involved\nroutines in the dynamic call graph of a program can give you a high level sense\nof “what happens” when running the code, which can then build your own\nconfidence in making performance-improving changes in slightly unfamiliar code.\n\nProfiling tools and tips\n\nMany useful profiling tools are available. A useful tool to reach for first is\npprof since it gives\ngood high level performance information and is easy to use both locally and for\ncode running in production. Also try\nperf if you want more\ndetailed insight into performance.\n\nSome tips for profiling:\n\nBuild production binaries with appropriate debugging information and\noptimization flags.\n\nIf you can, write a microbenchmark that covers the code you are\nimproving. Microbenchmarks improve turnaround time when making performance\nimprovements, help verify the impact of performance improvements, and can\nhelp prevent future performance regressions. However microbenchmarks can\nhave pitfalls that make them non-representative of full system\nperformance. Useful libraries for writing microbenchmarks:\nC++ Go Java.\n\nUse a benchmark library to emit performance counter readings both\nfor better precision, and to get more insight into program behavior.\n\nLock contention can often artificially lower CPU usage. Some mutex\nimplementations provide support for profiling lock contention.\n\nUse ML profilers for machine learning performance work.\n\nWhat to do when profiles are flat\n\nYou will often run into situations where your CPU profile is flat (there is no\nobvious big contributor to slowness). This can often happen when all low-hanging\nfruit has been picked. Here are some tips to consider if you find yourself in\nthis situation:\n\nDon’t discount the value of many small optimizations! Making twenty separate\n1% improvements in some subsystem is often eminently possible and\ncollectively mean a pretty sizable improvement (work of this flavor often\nrelies on having stable and high quality microbenchmarks). Some examples of\nthese sorts of changes are in the\nchanges that demonstrate multiple techniques\nsection.\n\nFind loops closer to the top of call stacks (flame graph view of a CPU\nprofile can be helpful here). Potentially, the loop or the code it calls\ncould be restructured to be more efficient. Some code that initially built a\ncomplicated graph structure incrementally by looping over nodes and edges of\nthe input was changed to build the graph structure in one shot by passing it\nthe entire input. This removed a bunch of internal checks that were\nhappening per edge in the initial code.\n\nTake a step back and look for structural changes higher up in the call\nstacks instead of concentrating on micro-optimizations. The techniques\nlisted under algorithmic improvements can be\nuseful when doing this.\n\nLook for overly general code. Replace it with a customized or lower-level\nimplementation. E.g., if an application is repeatedly using a regular\nexpression match where a simple prefix match would suffice, consider\ndropping the use of the regular expression.\n\nAttempt to reduce the number of allocations:\nget an allocation profile, and pick away at the highest\ncontributor to the number of allocations. This will have two effects: (1) It\nwill provide a direct reduction of the amount of time spent in the allocator\n(and garbage collector for GC-ed languages) (2) There will often be a\nreduction in cache misses since in a long running program using tcmalloc,\nevery allocation tends to go to a different cache line.\n\nGather other types of profiles, specially ones based on hardware performance\ncounters. Such profiles may point out functions that are encountering a high\ncache miss rate. Techniques described in the\nprofiling tools and tips section can be\nhelpful.\n\nAPI considerations\n\nSome of the techniques suggested below require changing data structures and\nfunction signatures, which may be disruptive to callers. This helps in\nthree ways:\n\nIt significantly reduces memory usage, since we only use\n\"sizeof(ValuePtr) + value_len\" bytes for each value, rather than\n\"sizeof(SafeCombinerKey) + sizeof(StringPiece) + value_len + new hash\ntable entry overhead\" for each value. This means that we flush the\nreducer buffer less often.\n\nIt's significantly faster, since we avoid extra hash table entries when\nwe're inserting a new value for a key that already exists in the table\n(and instead we just hook the value into the linked list of values for\nthat key).\n\nSince we associate a repetition count with each value in the linked\nlist, we can represent this sequence:\nOutput(key, \"1\");\nOutput(key, \"1\");\nOutput(key, \"1\");\nOutput(key, \"1\");\nOutput(key, \"1\");\n\nas a single entry in the linked list for \"key\" with a repetition count of 5.\nInternally we yield \"1\" five times to the user-level combining function.
