{"title":"\"What is Git and How Does Version Control Work?\"","author":"\"David Kim\"","date":"2025-12-17T13:22:47+00:00","main_content":"If you write code, you need to use Git. Full stop. It’s become so fundamental to software development that it’s hard to imagine working without it. But here’s the thing: a lot of developers use Git without really understanding what’s happening under the hood. Let me fix that.\n\nI’m going to walk you through what Git actually does, why it’s designed the way it is, and how to use it effectively. By the end, you’ll understand not just the commands, but the underlying model that makes Git so powerful.\n\nWhat Problem Does Version Control Solve?\n\nBefore we dive into Git specifically, let’s talk about why version control exists at all. Imagine you’re working on a project and you want to:\nSave your progress at different points in time\nTry an experimental feature without breaking your working code\nCollaborate with others without overwriting each other’s work\nSee what changed between yesterday and today\nRoll back when something breaks (and trust me, something will break)\n\nWithout version control, developers used to save files like this: project_final.py\nproject_final_v2.py\nproject_final_v2_actually_final.py\nproject_final_v2_actually_final_for_real_this_time.py\n\nYeah… not great. Version control systems formalize this chaos into a structured system.\n\nGit vs Other Version Control Systems\n\nGit isn’t the only version control system out there. Before Git became dominant, systems like SVN (Subversion) and CVS were common. Here’s what makes Git different:\n\nDistributed vs Centralized:\nCentralized (SVN): One central server holds the repository. Developers check out files, make changes, commit back to the server.\nDistributed (Git): Every developer has a complete copy of the repository with full history. You can work offline and sync later.\n\nThis might seem like a small difference, but it’s huge in practice. With Git:\nYou can commit changes locally without network access\nThe repository is backed up on every developer’s machine\nBranching and merging are lightning fast (they don’t require server communication)\nYou can work on multiple features simultaneously without conflicts\n\nHow Git Actually Works: The Mental Model\n\nHere’s where it gets interesting. Git doesn’t store your files the way you might think. Understanding this model is key to mastering Git.\n\nThe Three States\n\nEvery file in your Git repository can be in one of three states:\nModified: You’ve changed the file but haven’t committed it yet\nStaged: You’ve marked a modified file to go into your next commit\nCommitted: The file is safely stored in your local repository\n\nThese states correspond to three areas: Working Directory → Staging Area → Git Repository\n(modified) (staged) (committed)\n\nWhen you make changes, they start in your working directory. You explicitly add them to the staging area (git add), then commit them to the repository (git commit).\n\nHow Git Stores Data\n\nHere’s what blew my mind when I first learned it: Git doesn’t store file changes (deltas), it stores snapshots.\n\nWhen you commit, Git takes a snapshot of all your files and stores a reference to that snapshot. If files haven’t changed, Git doesn’t store them again—it just links to the previous version.\n\n# Commit history visualization\nCommit A: [ file1_v1, file2_v1, file3_v1 ]\n↓\nCommit B: [ file1_v2, file2_v1, file3_v1 ] # Only file1 changed\n↓\nCommit C: [ file1_v2, file2_v2, file3_v2 ] # file2 and file3 changed\n\nThis might seem inefficient (storing whole files instead of just changes), but Git uses clever compression and deduplication. Files that are identical across commits are stored only once.\n\nEssential Git Operations: A Practical Guide\n\nLet me walk you through the core Git operations you’ll use daily. I’ll show you what’s happening behind the scenes.\n\nInitializing a Repository\n\n# Create a new Git repository\ngit init my-project\ncd my-project\n# Or initialize in existing directory\ngit init\n\nThis creates a.git directory that contains all of Git’s metadata and object database. Everything Git needs to track your project is in this folder.\n\nMaking Your First Commit\n\n# Create a file\necho \"print('Hello, Git!')\" > hello.py\n# Check status (you'll use this constantly)\ngit status\n# Shows: hello.py is untracked\n# Stage the file\ngit add hello.py\n# Check status again\ngit status\n# Shows: hello.py is staged for commit\n# Commit with a message\ngit commit -m \"Add hello world script\"\n\nWhat just happened?\ngit add created a blob object in.git/objects containing the file contents\ngit commit created a tree object (directory structure) and a commit object (metadata)\nThe commit object points to the tree, which points to the blob\n\nEach commit has:\nA unique SHA-1 hash (like a3f5b7c2... )\nAuthor information\nTimestamp\nCommit message\nPointer to parent commit(s)\nPointer to the tree (file snapshot)\n\nUnderstanding the Commit Graph\n\nCommits form a directed acyclic graph (DAG). Each commit points to its parent(s): A ← B ← C ← D (main)\n↖ E ← F (feature-branch)\n\nCommit D’s parent is C, C’s parent is B, etc. This is how Git knows the history of your project.\n\nBranching: Git’s Killer Feature\n\nBranches in Git are incredibly lightweight—they’re just pointers to commits. This is why branching is so fast in Git (unlike SVN where branches were copies of the entire repository).\n\n# Create a new branch\ngit branch feature-login\n# Switch to the branch\ngit checkout feature-login\n# Or do both at once\ngit checkout -b feature-login\n# Modern Git (v2.23+) uses 'switch' instead of checkout\ngit switch -c feature-login\n\nWhat’s happening?\nGit creates a new pointer called feature-login that points to your current commit. When you make new commits, this pointer moves forward while main stays put: Before:\nmain → A ← B ← C\nfeature-login → (same as main)\n\nAfter commits on feature-login:\nmain → A ← B ← C\nfeature-login → A ← B ← C ← D ← E\n\nMerging: Bringing Changes Together\n\nOnce you’ve finished work on a branch, you merge it back:\n\n# Switch to the branch you want to merge into\ngit checkout main\n# Merge the feature branch\ngit merge feature-login\n\nGit tries to automatically merge changes. There are two types of merges:\n\n1. Fast-Forward Merge (simple case): Before:\nmain → A ← B ← C\nfeature → A ← B ← C ← D ← E\n\nAfter merge:\nmain → A ← B ← C ← D ← E (main caught up)\n\nGit just moves the main pointer forward. No merge commit needed.\n\n2. Three-Way Merge (when both branches have changes): Before:\nmain → A ← B ← C ← D\nfeature → A ← B ← E ← F\n\nAfter merge:\nmain → A ← B ← C ← D ← M\nfeature → A ← B ← E ← F ↗\n\nM = merge commit with two parents (D and F)\n\nGit creates a new commit that combines changes from both branches.\n\nHandling Merge Conflicts\n\nSometimes Git can’t automatically merge changes (e.g., both branches modified the same line). You get a conflict:\n\ngit merge feature-login\n# Auto-merging app.py\n# CONFLICT (content): Merge conflict in app.py\n# Automatic merge failed; fix conflicts and then commit the result.\n\nOpen the conflicted file:\n\n<<<<<<< HEAD\ndef login (username, password):\n# Version in main branch\nreturn authenticate_user(username, password)\n=======\ndef login (email, password):\n# Version in feature branch\nreturn auth_service. verify(email, password)\n>>>>>>> feature - login\n\nYou need to manually resolve this:\n\ndef login (email, password):\n# Resolved: use email parameter with new auth service\nreturn auth_service. verify(email, password)\n\nThen:\n\n# Stage the resolved file\ngit add app.py\n# Complete the merge\ngit commit\n\nWorking with Remote Repositories\n\nGit is distributed, so you can push and pull from remote repositories (like GitHub, GitLab, or your own server).\n\nCloning a Repository\n\n# Clone from remote\ngit clone https://github.com/user/repo.git\ncd repo\n# See configured remotes\ngit remote -v\n# origin https://github.com/user/repo.git (fetch)\n# origin https://github.com/user/repo.git (push)\n\norigin is the conventional name for your primary remote repository.\n\nPushing Changes\n\n# Push commits to remote\ngit push origin main\n# Push a new branch\ngit push -u origin feature-login\n# The -u flag sets up tracking so future pushes can just use: git push\n\nWhat’s happening?\nGit sends your commit objects, trees, and blobs to the remote server. The remote updates its main branch to point to your latest commit.\n\nPulling Changes\n\n# Fetch changes from remote\ngit fetch origin\n# Merge them into your current branch\ngit merge origin/main\n# Or do both at once\ngit pull origin main\n\nPro tip: I prefer git fetch + git merge over git pull because it lets me review changes before merging them into my local branch.\n\nEssential Git Commands Cheat Sheet\n\nHere are the commands you’ll use most often:\n\n# Status and Information\ngit status # Show working directory status\ngit log # Show commit history\ngit log --oneline --graph # Compact, visual history\ngit diff # Show unstaged changes\ngit diff --staged # Show staged changes\n# Basic Operations\ngit add <file> # Stage changes\ngit add. # Stage all changes\ngit commit -m \"message\" # Commit staged changes\ngit commit -am \"message\" # Stage and commit (tracked files only)\n# Branching\ngit branch # List branches\ngit branch <name> # Create branch\ngit checkout <branch> # Switch branch\ngit switch <branch> # Modern way to switch branch\ngit switch -c <branch> # Create and switch to new branch\ngit branch -d <branch> # Delete branch (safe)\ngit branch -D <branch> # Force delete branch\n# Merging\ngit merge <branch> # Merge branch into current\ngit merge --abort # Abort a problematic merge\n# Remote Operations\ngit clone <url> # Clone repository\ngit push origin <branch> # Push to remote\ngit pull origin <branch> # Fetch and merge from remote\ngit fetch # Download objects from remote (no merge)\n# Undoing Changes\ngit checkout -- <file> # Discard changes in working directory\ngit restore <file> # Modern way to discard changes\ngit reset HEAD <file> # Unstage file\ngit reset --hard HEAD # Discard all changes (dangerous!)\ngit revert <commit> # Create new commit that undoes a previous commit\n\nAdvanced Git Concepts\n\nOnce you’re comfortable with the basics, these concepts will level up your Git skills.\n\nRebasing\n\nRebasing rewrites commit history by moving commits to a new base:\n\n# Instead of merging, rebase your feature branch onto main\ngit checkout feature-login\ngit rebase main\n\nBefore rebase: main → A ← B ← C\nfeature → A ← D ← E\n\nAfter rebase: main → A ← B ← C\nfeature → A ← B ← C ← D' ← E'\n\nThe commits D and E are rewritten (hence D’ and E’) to apply on top of C.\n\nWhen to use rebase:\nTo keep a linear project history\nBefore merging a feature branch (clean up your commits)\nTo incorporate upstream changes without merge commits\n\nWhen NOT to use rebase:\nOn commits you’ve already pushed to a shared remote (it rewrites history)\nIf you’re not comfortable with resolving conflicts\n\nCherry-Picking\n\nApply a specific commit from one branch to another:\n\n# Apply commit abc123 to current branch\ngit cherry-pick abc123\n\nThis is super useful when you need just one fix from a feature branch without merging everything.\n\nInteractive Rebase\n\nClean up your commit history before pushing:\n\n# Rebase last 3 commits interactively\ngit rebase -i HEAD~3\n\nThis opens an editor where you can:\nReorder commits\nSquash multiple commits into one\nEdit commit messages\nDrop commits entirely\n\nStashing\n\nTemporarily save changes without committing:\n\n# Stash current changes\ngit stash\n# Do other work, switch branches, etc.\n# Reapply stashed changes\ngit stash pop\n# List all stashes\ngit stash list\n# Apply a specific stash\ngit stash apply stash@ { 2 }\n\nI use this all the time when I need to quickly switch contexts.\n\nBest Practices and Workflow\n\nAfter years of using Git on teams of all sizes, here are the practices that make a difference:\n\n1. Write Good Commit Messages\n\n# Bad\ngit commit -m \"fix\"\ngit commit -m \"updates\"\n# Good\ngit commit -m \"Fix null pointer exception in user login\n- Add null check before accessing user.email\n- Add test case for null user scenario\n- Resolves issue #123\"\n\nA good commit message:\nStarts with a concise summary (50 chars or less)\nExplains what changed and why (not how—the code shows how)\nReferences related issues/tickets\n\n2. Commit Often, Push Deliberately\n\nMake small, focused commits as you work. Each commit should be a logical unit of change that you could revert independently if needed.\n\n# Good: Atomic commits\ngit commit -m \"Add user model\"\ngit commit -m \"Add user validation\"\ngit commit -m \"Add user controller\"\n# Bad: One giant commit\ngit commit -m \"Add entire user system\"\n\n3. Use Branches for Everything\n\nNever commit directly to main. Create a branch for each feature, bug fix, or experiment:\n\ngit checkout -b feature/add-user-auth\ngit checkout -b bugfix/fix-login-timeout\ngit checkout -b experiment/try-new-caching\n\n4. Keep Your Local Main Up to Date\n\n# Regularly update your local main\ngit checkout main\ngit pull origin main\n# Then rebase your feature branch\ngit checkout feature-branch\ngit rebase main\n\nThis prevents massive merge conflicts later.\n\n5. Use.gitignore\n\nDon’t commit generated files, dependencies, or sensitive data: #.gitignore file\n__pycache__/\n*.pyc\nnode_modules/\n.env\n.DS_Store\ndist/\nbuild/\n*.log\n\nConclusion\n\nGit is one of those tools that seems simple on the surface but has incredible depth. The key to mastering it is understanding the underlying model: snapshots, not deltas; branches as pointers; commits as nodes in a graph.\n\nOnce you internalize these concepts, Git’s commands start to make sense. You stop memorizing recipes and start understanding what each command actually does to the repository.\n\nStart with the basics: clone, add, commit, push, pull. Get comfortable with branches and merging. Then gradually add more advanced techniques like rebasing and interactive commits to your toolkit.\n\nAnd remember: Git’s complexity is there for a reason. It’s solving hard problems around distributed collaboration, and it’s doing it remarkably well. The learning curve is steep, but it’s absolutely worth it.\n\nFor deeper dives into Git internals, check out the Pro Git book (free online) and Git’s official documentation. The Git Reference Manual is also invaluable when you need to understand exactly what a command does."}
