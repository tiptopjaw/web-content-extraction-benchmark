{"title":"My Experiments with AI Coding","author":"bibhas2","date":"2025-12-19T16:17:19+00:00","main_content":"I recently shot a roll of EASTMAN DOUBLE-X motion picture film. My developer scanned the images as 16bit greyscale TIFF files. The high bit rate was great for post processing. The downside is that Google Photos doesn’t support those TIFFs. I needed to convert them to PNG. I saw this as a chance to create a challenge for AI assisted coding. Instead of manually saving the photos as PNG, I decided to do the conversion using C++.\n\nTLDR; what could have been a dauting task took me only a day with AI.\n\nInstalling the Dependency Packages\n\nGoogle AI mode steered me towards using libtiff and libpng. While libpng has a few pre-built binaries, libtiff comes only as source code. I brewed over if I should try to compile libtiff or just give up.\n\nAfter more help from Google AI mode I found out that both libraries can be installed as Debian packages. In the end this proved to be an easy and flawless option. AI also suggested that I install a few other commonly used dependencies like zlib1g-dev. Finally, this is what I did.\n\nsudo apt update\n\nsudo apt install build-essential\n\nsudo apt install libtiff-dev libwebp-dev libzstd-dev zlib1g-dev\n\nsudo apt install libpng-dev\n\nCode using these libraries can be compiled like this.\n\ng++ -o tiff-png main.cc -ltiff -lpng\n\nProof of Concept\n\nI prompted Gemini 3 Thinking with:\n\nGenerate C code that uses libtiff to open a tiff file and save it as PNG\n\nThe generated code was fairly easy to understand and told me this is something I can deal with.\n\nIteration 1\n\nI started coding in VS Code with C++ extensions installed. I wasn’t logged into GitHub. So a default AI model was in use by the coding agent.\n\nThe AI agent generated very similar code as Gemini with a similar prompt. The code compiled and worked fine. The converted PNG file looked great. Here’s Iteration 1.\n\nThe VS Code generated code did one thing better than Gemini. Gemini was getting the TIFF pixel values using TIFFReadRGBAImage which gives us the rows from bottom to top. VS Code used TIFFReadRGBAImageOriented which gave us the rows from top to bottom. This made the code simpler.\n\nIteration 2\n\nAs I compared the code generated in VS Code with Gemini, I realized that Gemini was writing PNG one row at a time using png_write_row. The VS Code generated code was accumulating the entire PNG in memory and then writing it to the file using png_write_image. Gemini code was simpler and needed less memory. I wanted to adopt that. Instead of manually making the changes I just asked VS Code to do it for me:\n\nChange the save_tiff_as_png function to use png_write_row to write row by row. This should simplify the code and reuse a single row memory buffer making the code more memory efficient.\n\nHere is Iteration 2.\n\nIteration 3\n\nI did a bunch of other changes, all manually, to clean up the code and process multiple TIFF files in one go. As I was about to declare victory a thought kept gnawing at me. Both Gemini and VS Code assumed the TIFF file to be 4 channel (RGBA) with 8bit per color channel. This didn’t quite sit right. Sure enough, the film developer gave me the TIFF files as single channel (greyscale) with 16bit per channel. The converted PNG was RGBA (even though it looked greyscale) and 8bit. What a bummer.\n\nI learned later that the functions of the ilk TIFFReadRGBAImageOriented normalize all image types as 8bit RGBA. This makes the code easy but you lose a lot of fidelity from the original TIFF.\n\nAnyway, I had to do something about this. I straight up asked Gemini:\n\nThe code appears to be assuming 8bits per color channel and 4 channels (RGBA) to be always present in the TIFF file. Can these be made dynamic?\n\nGemini now produced its masterpiece. The code now set the PNG header with the correct number of channels and bits per channel. TIFF image data was read row by row using TIFFReadScanline which did not do any normalization and gave you the raw pixel values. We wrote the PNG row by row using png_write_row, same as before.\n\nI took the Gemini generated code and manually adopted it in VS Code. The code compiled. But for the first time the converted PNG didn’t look right.\n\nI did old fashioned searches, read StackOverflow, asked AI mode. My gut told me this has something to do with the endianness. You see, when you deal with 16bit pixel values you need to worry about the endianness. I was just vibing with the endianness issue at this point. More research told me the full story.\n\nA TIFF file can be in either little or big endian form. The TIFFReadScanline function converts the pixel values to the machine’s native endianness. That will be little endian for me. The png_write_row function by default doesn’t convert the endianness and writes the data as is. This means, it was writing the pixels as little endian. This is a violation of the PNG spec which requires the pixels to be big endian.\n\nIt turns out that we can make png_write_row swap the bytes before writing by calling png_set_swap. The Gemini generated code was already calling png_set_swap. Why wasn’t it working?\n\nThere was a slight issue. The png_set_swap must be called after png_write_info. Gemini had the ordering wrong. I fixed the issue and got the PNG in its 16bit single channel greyscale glory. No more 8bit and no more 4 channels (RGBA).\n\nHere’s the final version.\n\nConclusions\n\nI used a mix of agents; Gemini, VS Code and Google AI mode (for general research). For batting around an idea, I like using Gemini. When I know what I want I have VS Code agent do it for me. Sometimes, I manually incorporate Gemini’s code when it’s easy to do.\n\nAI helped me with the dependency setup. AI understood all my optimization attempts and kept refining the code.\n\nI was only vibing (doing things without understanding) with the endianness issue. Sure enough I got burned by that. I’m glad I did."}
