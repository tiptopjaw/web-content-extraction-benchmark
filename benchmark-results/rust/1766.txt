Let’s discuss the major practices below:\n\n1. Choose Industry-Specific Coding Standards\n\nCoding best practices and standards vary depending on the industry a specific product is being built for. The standards required for coding software for luxury automobiles will differ from those for gaming software.\n\nFor example, MISRA C and C++ were written for the automotive industry and are considered the de-facto standards for building applications that emphasize safety. They are the absolute best practices for writing code in the industry.\n\nAdhering to industry-specific coding standards in software engineering makes writing correct code that matches product expectations easier. Writing code that will satisfy the end-users and meet business requirements becomes easier.\n\n2. Focus on Code readability\n\nReadable code is easy to follow and optimizes space and time. Here are a few ways to achieve that:\nWrite as few lines as possible.\nUse appropriate naming conventions.\nSegment blocks of code in the same section into paragraphs.\nUse indentation to mark the beginning and end of control structures. Specify the code between them.\nDon’t use lengthy functions. Ideally, a single function should carry out a single task.\nUse the DRY (Don’t Repeat Yourself) principle. Automate repetitive tasks whenever necessary. The same piece of code should not be repeated in the script.\nAvoid Deep Nesting. Too many nesting levels make code harder to read and follow.\nCapitalize SQL special words and function names to distinguish them from table and column names.\nAvoid long lines. It is easier for humans to read blocks of lines that are horizontally short and vertically long.\n\n3. Meaningful Names\n\nChoose meaningful names that convey the purpose of the variable or function. Consistent naming conventions enhance clarity and maintainability. // Bad\n\nconst cust = \"John\"\n\nconst customer = \"Alice\"\n\n// Better\n\nconst customerName = \"John\"\n\nconst customerFullName = \"Alice Johnson\"\n\nDifferent naming conventions used in coding –\nCamel Case – In camel case, you start a name with a lowercase letter. If the name has multiple words, the later words begin with capital letters. Camel case is commonly used in JavaScript for variable and function names.\n\nFor Example: const userName = \"Smith\";\n                     \n\nfunction reverseName(name) {\n\nreturn name.split(\"\").reverse().join(\"\");\n\n}\nSnake Case – In snake case, you start the name with a lowercase letter. If the name has multiple words, the later words are also lowercase, and you use an underscore (_) to separate them.\n\nFor Example: const user_name = \"Smith\";\nKebab Case – Kebab case is similar to snake case, but you use a hyphen (-) instead of an underscore (_) to separate the words.\n\nFor Example: const user-name = \"Smith\";\nPascal Case (Upper Camel Case): – Names in pascal case start with a capital letter. For names with multiple words, all words begin with capital letters. Pascal case is typically used for class names in both Python and JavaScript.\n\nFor Example: class Person {\n\nconstructor(firstName, lastName) {\n\nthis.firstName = firstName;\n\nthis.lastName = lastName;\n\n    }\n\n}\n\n4. Avoid using a Single Identifier for multiple purposes\n\nAscribe a name to each variable that clearly describes its purpose. A single variable can’t be assigned various values or utilized for numerous functions. This would confuse everyone reading the code and make future enhancements more challenging. Always assign unique variable names.\n\nWhen the same variable or function name is used to represent different concepts or purposes within the code, it can lead to confusion, bugs, and unintended behavior.\n\nFor Example: function outerFunction() {\n\n    let count = 10;\n\n    function innerFunction() {\n\n        // Oops! This 'count' shadows the outer one.\n\n        const count = 20;\n\n        console.log(count);\n\n    }\n\n    innerFunction();\n\n    console.log(count);  // Prints 10, not 20\n\n}\n\n5. Add Comments and Prioritize Documentation\n\nComments serve as a form of documentation within the code, explaining the logic, functionality, or purpose of specific sections. Well-placed comments transform complex algorithms or intricate business rules into understandable pieces of information.\n\nFor Example: // TODO: Refactor this function for better performance\n\nfunction processItems(items) {\n\n//... existing logic...\n\n// TODO: Optimize the sorting algorithm\n\nitems.sort((a, b) => a.value - b.value);\n\nif (items.length === 0) {\n\nconsole.warn(\"Empty items array!\");\n\n    }\n\n}\n\nWhen to add comments:\nInclude comments for intricate or non-obvious code segments.\nExplain business rules, domain-specific logic, or regulatory requirements.\nClarify how your code handles edge cases or exceptional scenarios.\nDocument workarounds due to limitations or external dependencies.\nMark areas where improvements or additional features are needed.\n\nWhen Not to add comments:\nAvoid redundant comments that merely repeat what the code already expresses clearly.\nIf the code’s purpose is evident (e.g., simple variable assignments), skip unnecessary comments.\nRemove temporary comments used for debugging once the issue is resolved.\nIncorrect comments can mislead other developers, so ensure accuracy.\n\n6. Efficient Data Processing\n\nDivide code into smaller, self-contained modules or functions for reusability and maintainability. Effective Code Review and Refactoring\n\nEngage QA during refactoring to prevent new bugs. Isolate debugging from refactoring to maintain stability. // Before refactoring\n\nfunction calculateTotal(items) {\n\n    let total = 0;\n\n    for (const item of items) {\n\n        total += item.price;\n\n    }\n\n    return total;\n\n}\n\n// After refactoring\n\nfunction calculateTotal(items) {\n\n    return items.reduce((acc, item) => acc + item.price, 0);\n\n}\n\n9. Try to formalize Exception Handling\n\n‘Exception’ refers to problems, issues, or uncommon events that occur when code is run and disrupt the normal flow of execution. This either pauses or terminates program execution, a scenario that must be avoided.\n\nException handling is a critical aspect of programming, allowing developers to gracefully manage unexpected or erroneous situations. When an error occurs during program execution, the normal flow is disrupted, and an “exception” object containing information about the error is created. Exception handling involves responding to these exceptions effectively.\n\nHowever, when they do occur, use the following techniques to minimize damage to overall execution in terms of both time and dev effort:\nKeep the code in a try-catch block.\nEnsure that auto recovery has been activated and can be used.\nConsider that it might be an issue of software/network slowness. Wait a few seconds for the required elements to show up.\nUse real-time log analysis.\n\nHere are the key components of exception handling:\nTry block: The try block encapsulates code where an error might occur. If an exception occurs within this block, control transfers to the corresponding catch block.\n\nFor Example: try {\n\n    // code that may throw an exception\n\n    const numerator = 10;\n\n    const denominator = 0;\n\n // throws a division by zero exception\n\n    const result = numerator / denominator;\n\n // skipped due to the exception\n\n    console.log(\"Result:\", result);\n\n} \n\ncatch (error) {\n\n// handle the exception\n\n    console.error(\"Error:\", error.message); \n\n}\nCatch block: The catch block catches and handles exceptions thrown within the try block.\n\nFor Example: try {\n\n    //...\n\n} catch (error) {\n\n// Handle the exception\n\n    console.error(\"Error:\", error.message);\n\n }\nFinally block (optional): The finally block executes regardless of whether an exception occurs or not. It is commonly used for cleanup tasks (e.g., closing files, releasing resources).\n\nFor Example: try {\n\n    //...\n\n} catch (error) {\n\n  // …\n\n} finally {\n\n// Executed always\n\n    console.log(\"Cleanup tasks here\"); \n\n}\n\nLearn more about Exception Handling in Selenium WebDriver.\n\n10. When choosing standards, think Closed vs. Open\n\nConsider CERT vs. MISRA. CERT emphasizes community cooperation and participation. It offers a coding standard that is freely available as a web-based wiki.\nWith CERT, users can comment on specific guidelines – comments are considered when the standards are reviewed and updated.\nOn the other hand, MISRA is a set of C and C++ coding standards developed and maintained by the Motor Industry Software Reliability Association (MISRA). It is primarily considered the de-facto coding standard for embedded industries.\nMISRA was created and is updated by working groups according to predetermined blueprints. While secure and reliable, it is not available for free, though it admits some community feedback when implementing updates.\nNaturally, CERT is easier to work with.
