{"title":"Stop Nesting Database Systems","author":"Christian Winter","date":null,"main_content":"The idea of embedding fast data processing engines into other systems to speed up analytics is not new.\nIn fact, DuckDBs started out as SQL Analytics on Pandas.\nMeta had a similar vision for its Velox project, which takes a slightly different approach by building an embeddable execution engine distributed as a C++ library.\nIf you’re considering building your own analytics engine, embedding an existing engine is almost always a better choice. Don’t roll your own crypto analytics engine.\n\nRecently, however, there has been a trend to embed database systems in other database systems for various reasons, which I find surprising and worrying.\nIt started with pg_duckdb, and now there is omni-sqlite, pg_clickhouse, and, just this week, a Planetscale to MotherDuck pipeline over pg_duckdb.\nWith this pipeline, you end up nesting four systems:\nPlanetScale packs PostgreSQL, which packs the MotherDuck-built extension pg_duckdb, which packs DuckDB and MotherDuck.\n\nSolutions like pg_duckdb and pg_clickhouse promise to solve your PostgreSQL analytics issues with HTAP-like capabilities.\nThey allow you to run transactions in PostgreSQL and claim to seamlessly hand over analytical workloads to Clickhouse and DuckDB with a sleek interface directly from PostgreSQL.\nBut only at first glance. To me, this approach is similar to opening a random closet and throwing all your clutter in there to hide it before guests arrive.\nDoes it look clean? Sure! But does it solve your issues? Probably not.\n\nTo find out if these scientists were too preoccupied with whether they could, let’s take a look at how these extensions work and what they can and can’t do for you.\n\nHow the Nesting of Database Systems Works\n\nWe’ll use pg_duckdb as an example because it is the oldest and most popular of the bunch.\nHowever, the points discussed here are not specific to pg_duckdb, but rather to the concept of nesting database systems in general.\n\nAmong the available solutions, pg_duckdb has the greatest potential to speed up analytics within a transactional host like PostgreSQL.\npg_duckdb has several features that enable elegant, nearly seamless integration.\nDuckDB was designed from the beginning to be embedded within other processes.\nDuckDB supports the PostgreSQL SQL dialect.\nDuckDB has an excellent query planner, joins, and analytical algorithms.\nYour queries run within the same transaction on transactionally consistent data within the bounds of PostgreSQL.\n\nWith the extension enabled, pg_duckdb controlls query execution for all queries you send to PostgreSQL.\nIt knows what it can handle itself and delegates the rest, mainly scanning the data on disk, to PostgreSQL.\nAll join and aggregation processing then takes place in DuckDB before the result is handed back to PostgreSQL.\nThe diagram below provides a rough overview and illustrates some optimizations that we will discuss below.\n\nWhat They Can Do Better than the Outer Engine\n\nAnalytical query processing has certainly come a long way since PostgreSQL was designed. Consequently, using an analytical system provides a wealth of functionality not available in PostgreSQL.\nUnlike PostgreSQL, DuckDB’s optimizer was designed for complex analytical workloads, so it can find much more efficient plans.\nThis allows DuckDB to fully parallelize queries and utilize system resources much more effectively.\nWhile PostgreSQL assumes memory is scarce, analytical engines can productively saturate powerful servers.\nThis has the potential to reduce execution times significantly.\nYou also gain access to features of the embedded engine that are not available in PostgreSQL. For example, you can query remote Parquet files and combine them with local subresults.\n\nWhere They Are On-Par\n\nOf course, nesting another engine does not magically solve all performance issues. There are parts where the embedded engine can only be as fast as the outer system allows.\nIn order to run queries on data managed by the host system, the embedded engine must retrieve the data from the host.\nRetrieving this data requires a regular scan, which PostgreSQL must perform.\npg_duckdb employs additional optimizations to reduce the impact of this process. It pushes filters to PostgreSQL scans where possible and runs scans in parallel.\nHowever, it still depends on PostgreSQL to fetch rows from disk using its buffer manager. It has no access to columnar layout or advanced techniques such as scan pruning.\nTherefore, while the upper parts of the query plan can leverage DuckDB’s full performance, all scans are expected to run at PostgreSQL speeds.\n\nWhere They Hold You Back\n\nCreating a Frankenstein system comes at a cost.\nFor one thing, you increase the complexity of your stack.\nRather than running PostgreSQL in isolation, you now need two engines that aren’t designed to work together to do so.\nThere are other limitations as well.\n\nWhile you can perform many analytical tasks, some things fundamentally do not work.\nThe most obvious issue is that you cannot mix and match functionality. For instance, you can’t run DuckDB plans that require PostgreSQL functions in the query plan.\nHowever, this is not only a surface-level issue of available functions.\nWhat if the systems have different sort orders or comparisons, e.g., through collations?\nSuddenly, the results could differ depending on which system executes the query, and you would not know.\n\nThe last downside is the missing upsides in the current state of these solutions.\nWhile there are obvious theoretical opportunities for performance improvements, neither pg_duckdb nor pg_clickhouse can reliably outperform PostgreSQL on TPC-H in their current forms. 1 So, you need to carefully decide which queries to use them for, as well as when sticking to PostgreSQL is the better choice.\n\nNesting Systems vs ETL\n\nAlthough nesting systems have their drawbacks, there are definitely use cases where they are preferable to typical ETL setups.\n\nFirst, if you need to perform analytics, they must occur somewhere. If you don’t want to adopt a full HTAP engine like CedarDB right away, nesting systems can be a good stopgap solution.\nFor many PostgreSQL users, the alternative to such integrations is setting up and maintaining a full ETL pipeline for infrequent, analytical “heavy hitter” queries, such as monthly reporting.\nIn that case, having the option to spin up an embedded engine at the click of a button and have it disappear without a trace is ideal.\nHowever, if you need to continuously run analytics, the benefits of a dedicated analytical or HTAP solution will quickly outweigh the cost.\n\nThis does not hold true for approaches like pg_clickhouse or the MotherDuck component for pg_duckdb, which act more like foreign data wrappers.\nRather than being fully embedded within the host, they provide access to query results on external systems through a single, convenient interface.\nHowever, they don’t help with moving data to the analytical system or keeping it up to date.\nTherefore, you still need to set up a data pipeline from PostgreSQL to Clickhouse or MotherDuck.\nWhile these solutions are more convenient, they are not fundamentally different from traditional ETL stacks.\n\nThese extensions’ main promise is HTAP, a workload that ETL pipelines will never be able to cover.\nRunning analytics within a transaction and persisting results with transactional guarantees is only possible if a single system controls both analytical reads and transactional writes.\nThese workloads are much more common than one might think. 2\nTo run HTAP workloads, your analytical queries require a consistent view of your transactional data and the ability to persist results with the same guarantees.\nHowever, is an analytical engine nested in a transaction system worthy of being called “HTAP”?\n\nIs This HTAP?\n\nThe MotherDuck post that inspired this semi-rant gives a “maybe yes” but I want to stick to a clear “no.”\nThe setup can be useful in the above scenarios, but the only part that manages to even scratch the surface of HTAP is the last one: a local, embedded analytical engine within a transactional host.\n\nAnd even a full embedding still has to delegate scans to the host system, meaning it can not overcome the most crucial bottleneck for analytics: scans.\nThe majority of time in analytics is spent on table scans, even in systems that heavily optimize their scans for analytical workloads. 3\nBoth MotherDuck and Clickhouse have such scans as well, but they are only useful on data they manage themselves, which does not come with the same transactional guarantees as your PostgreSQL query.\n\nThe only way to have your cake and eat it too is to use a system built for HTAP that can optimize the entire process, from scan to result, while still guaranteeing transactional integrity.\nGetting this right takes more than simply embedding an analytical engine.\n\nHere is a short list of what we had to build at CedarDB to achieve full performance for a mix of analytics and transactions:\nA storage layout that works for both point accesses and analytical scans.\nOptimistically synchronized data structures to reduce lock contention between parallel write and read operations.\nOptimistic snapshot isolation for transactions to prevent long reads from blocking writers.\nA scheduler that prioritizes small writes without starving long reads.\n\nThese features are in addition to optimizations for the analytical and transactional parts in isolation, many of which we highlight in other blog posts.\nBuilding an HTAP system is definitely a much larger project than nesting two database systems, but I believe the results justify the effort.\n\nIf you want to experience the performance and convenience of a true HTAP solution for yourself, try out our Community Edition with an HTAP workload.\n\nBoth pg_clickhouse and pg_duckdb report own comparisons against PostgreSQL.
