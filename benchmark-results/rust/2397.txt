ahead-of-time codegen for rustc, meaning that it\ncan be loaded by the existing rustc frontend, but benefits from GCC by having more architectures\nsupported and having access to GCC’s optimizations.\nIt is not to be confused with gccrs, which is a GCC frontend for Rust.\n\nGCC patches status\n\nThere’s been some nice progress on the GCC side these past months.\n\nMany patches were sent for review:\n\nlibgccjit: Make is_same_type_as() supporting floating point types\n\nlibgccjit: Allow casts between integers and pointers\n\nlibgccjit: Add support for the weak variable attribute\n\nlibgccjit: Add support for the x87 CPU feature\n\nlibgccjit: Fix get_size for long double\n\nlibgccjit: Add a function to check if LTO is supported\n\nlibgccjit: Fix for -fanalyzer used in libgccjit\n\nlibgccjit: Fix volatile loads and stores\n\nlibgccjit: Add support for Aarch64 CPU features\n\nlibgccjit: Allow set_location to various classes\n\nlibgccjit: Add support for packed struct\n\nlibgccjit: Fix UB in constructor functions\n\nlibgccjit: Fix test-cold-attribute.c\n\nI sent all the patches that were ready to be sent in the hope that we can be as close to upstream as possible for the next release.\n\nAlso, a few patches were merged upstream:\n\nlibgccjit: Add the function attributes for setting the ABI\n\nlibgccjit: Add _Float16, _Float32, _Float64 and __float128 support for jit\n\nlibgccjit: Fix error on Power architectures caused by wrong jit_target_objs\n\nlibgccjit: Add gcc_jit_context_new_array_type_u64\n\nlibgccjit: Do not treat warnings as [no-mentions] handler in our triagebot config (thanks to Urgau!)\n\nSome of these PRs were made to make it easier to cross-compile rustc to new platforms, which is currently my focus.\nThis needed a few PRs on the Rust side:\n\nMove the libgccjit.so file in a target directory\n\nBootstrap config: libgccjit libs dir (thanks to Kobzol for the help!)\n\nWith those, it is possible to build a native rustc for m68k with very few workarounds!\nCurrently, it requires to patch a few crates used by rustc since the support for m68k isn’t\nmerged yet.\n(A PR adding m68k support to rustix will also need to be done since both of these crates are dependencies of tempfile which is a rustc dependency.)\nThe resulting binary cannot yet correctly compile a simple program, but I’m going to work on improving this in the coming months.\n\nIn the next months, I plan to debug all this so that we can compile Rust programs on m68k and in doing so, I will make more tests pass on m68k.\n\nHere’s a rough summary
