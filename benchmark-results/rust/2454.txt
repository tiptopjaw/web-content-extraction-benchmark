Test Harnesses\n\nThere are four independent test harnesses used for testing the\ncore SQLite library.\nEach test harness is designed, maintained, and managed separately\nfrom the others.\n\nThe TCL Tests are the original tests for SQLite.\nThey are contained in the same source tree as the\nSQLite core and like the SQLite core are in the public domain. The\nTCL tests are the primary tests used during development.\nThe TCL tests are written using the\nTCL scripting language.\nThe TCL test harness itself consists of 27.2 KSLOC\nof C code used to create the TCL interface. The test scripts are contained\nin 1390 files totaling\n23.2MB in size. There are\n51445 distinct test cases, but many of the test\ncases are parameterized and run multiple times (with different parameters)\nso that on a full test run millions of\nseparate tests are performed.\n\nThe TH3 test harness is a set of proprietary tests, written in\nC that provide 100% branch test coverage\n(and 100% MC/DC test coverage ) to\nthe core SQLite library. The TH3 tests are designed to run\non embedded and specialized platforms that would not easily support\nTCL or other workstation services. TH3 tests use only the published\nSQLite interfaces. TH3 consists of about\n76.9 MB or 1055.4 KSLOC\nof C code implementing 50362 distinct test cases.\nTH3 tests are heavily parameterized, though, so a full-coverage test runs\nabout 2.4 million different test\ninstances.\nThe cases that provide 100% branch test coverage constitute\na subset of the total TH3 test suite. A soak test\nprior to release does about\n248.5 million tests.\nAdditional information on TH3 is available separately.\n\nThe SQL Logic Test\nor SLT test harness is used to run huge numbers\nof SQL statements against both SQLite and several other SQL database engines\nand verify that they all get the same answers. SLT currently compares\nSQLite against PostgreSQL, MySQL, Microsoft SQL Server, and Oracle 10g.\nSLT runs 7.2 million queries comprising\n1.12GB of test data.\n\nThe dbsqlfuzz engine is a\nproprietary fuzz tester. Other fuzzers for SQLite\nmutate either the SQL inputs or the database file. Dbsqlfuzz mutates\nboth the SQL and the database file at the same time, and is thus able\nto reach new error states. Dbsqlfuzz is built using the\nlibFuzzer framework of LLVM\nwith a custom mutator. There are\n336 seed files. The dbsqlfuzz fuzzer\nruns about one billion test mutations per day.\nDbsqlfuzz helps ensure\nthat SQLite is robust against attack via malicious SQL or database\ninputs.\n\nIn addition to the four main test harnesses, there are many other\nsmall programs that implement specialized tests. Here are a few\nexamples:\n\nThe \"speedtest1.c\" program\nestimates the performance of SQLite under a typical workload.\n\nThe \"mptester.c\" program is a stress test for multiple processes\nconcurrently reading and writing a single database.\n\nThe \"threadtest3.c\" program is a stress test for multiple threads using\nSQLite simultaneously.\n\nThe \"fuzzershell.c\" program is used to\nrun some fuzz tests.\n\nThe \"jfuzz\" program is a libfuzzer-based fuzzer for\nJSONB inputs to the JSON SQL functions.\n\nAll of the tests above must run successfully, on multiple platforms\nand under multiple compile-time configurations,\nbefore each release of SQLite.\n\nPrior to each check-in to the SQLite source tree, developers\ntypically run a subset (called \"veryquick\") of the Tcl tests\nconsisting of about\n304.7 thousand test cases.\nThe veryquick tests include most tests other than the anomaly, fuzz, and\nsoak tests. The idea behind the veryquick tests are that they are\nsufficient to catch most errors, but also run in only a few minutes\ninstead of a few hours.\n\n3. Anomaly Testing\n\nAnomaly tests are tests designed to verify the correct behavior\nof SQLite when something goes wrong. It is (relatively) easy to build\nan SQL database engine that behaves correctly on well-formed inputs\non a fully functional computer. It is more difficult to build a system\nthat responds sanely to invalid inputs and continues to function following\nsystem malfunctions. The anomaly tests are designed to verify the latter\nbehavior.\n\n3.1. Out-Of-Memory Testing\n\nSQLite, like all SQL database engines, makes extensive use of\nmalloc() (See the separate report on\ndynamic memory allocation in SQLite for\nadditional detail.)\nOn servers and workstations, malloc() never fails in practice and so correct\nhandling of out-of-memory (OOM) errors is not particularly important.\nBut on embedded devices, OOM errors are frighteningly common and since\nSQLite is frequently used on embedded devices, it is important that\nSQLite be able to gracefully handle OOM errors.\n\nOOM testing is accomplished by simulating OOM errors.\nSQLite allows an application to substitute an alternative malloc()\nimplementation using the sqlite3_config ( SQLITE_CONFIG_MALLOC,...)\ninterface. The TCL and TH3 test harnesses are both capable of\ninserting a modified version of malloc() that can be rigged to fail\nafter a certain number of allocations. These instrumented mallocs\ncan be set to fail only once and then start working again, or to\ncontinue failing after the first failure. OOM tests are done in a\nloop. On the first iteration of the loop, the instrumented malloc\nis rigged to fail on the first allocation. Then some SQLite operation\nis carried out and checks are done to make sure SQLite handled the\nOOM error correctly. Then the time-to-failure counter\non the instrumented malloc is increased by one and the test is\nrepeated. The loop continues until the entire operation runs to\ncompletion without ever encountering a simulated OOM failure.\nTests like this are run twice, once with the instrumented malloc\nset to fail only once, and again with the instrumented malloc set\nto fail continuously after the first failure.\n\n3.2. I/O Error Testing\n\nI/O error testing seeks to verify that SQLite responds sanely\nto failed I/O operations. I/O errors might result from a full disk drive,\nmalfunctioning disk hardware, network outages when using a network\nfile system, system configuration or permission changes that occur in the\nmiddle of an SQL operation, or other hardware or operating system\nmalfunctions. Whatever the cause, it is important that SQLite be able\nto respond correctly to these errors and I/O error testing seeks to\nverify that it does.\n\nI/O error testing is similar in concept to OOM testing; I/O errors\nare simulated and checks are made to verify that SQLite responds\ncorrectly to the simulated errors. I/O errors are simulated in both\nthe TCL and TH3 test harnesses by inserting a new\nVirtual File System object that is specially rigged\nto simulate an I/O error after a set number of I/O operations.\nAs with OOM error testing, the I/O error simulators can be set to\nfail just once, or to fail continuously after the first failure.\nTests are run in a loop, slowly increasing the point of failure until\nthe test case runs to completion without error. The loop is run twice,\nonce with the I/O error simulator set to simulate only a single failure\nand a second time with it set to fail all I/O operations after the first\nfailure.\n\nIn I/O error tests, after the I/O error simulation failure mechanism\nis disabled, the database is examined using\nPRAGMA integrity_check to make sure that the I/O error has not\nintroduced database corruption.\n\n3.3. Crash Testing\n\nCrash testing seeks to demonstrate that an SQLite database will not\ngo corrupt if the application or operating system crashes or if there\nis a power failure in the middle of a database update. A separate\nwhite-paper titled\nAtomic Commit in SQLite describes the\ndefensive measures SQLite takes to prevent database corruption following\na crash. Crash tests strive to verify that those defensive measures\nare working correctly.\n\nIt is impractical to do crash testing using real power failures, of\ncourse, and so crash testing is done in simulation. An alternative\nVirtual File System is inserted that allows the test\nharness to simulate the state of the database file following a crash.\n\nIn the TCL test harness, the crash simulation is done in a separate\nprocess. The main testing process spawns a child process which runs\nsome SQLite operation and randomly crashes somewhere in the middle of\na write operation. A special VFS randomly reorders and corrupts\nthe unsynchronized\nwrite operations to simulate the effect of buffered filesystems. After\nthe child dies, the original test process opens and reads the test\ndatabase and verifies that the changes attempted by the child either\ncompleted successfully or else were completely rolled back. The\nintegrity_check PRAGMA is used to make sure no database corruption\noccurs.\n\nThe TH3 test harness needs to run on embedded systems that do not\nnecessarily have the ability to spawn child processes, so it uses\nan in-memory VFS to simulate crashes. The in-memory VFS can be rigged\nto make a snapshot of the entire filesystem after a set number of I/O\noperations. Crash tests run in a loop. On each iteration of the loop,\nthe point at which a snapshot is made is advanced until the SQLite\noperations being tested run to completion without ever hitting a\nsnapshot. Within the loop, after the SQLite operation under test has\ncompleted, the filesystem is reverted to the snapshot and random file\ndamage is introduced that is characteristic of the kinds of damage\none expects to see following a power loss. Then the database is opened\nand checks are made to ensure that it is well-formed and that the\ntransaction either ran to completion or was completely rolled back.\nThe interior of the loop is repeated multiple times for each\nsnapshot with different random damage each time.\n\n3.4. Compound failure tests\n\nThe test suites for SQLite also explore the result of stacking\nmultiple failures. For example, tests are run to ensure correct behavior\nwhen an I/O error or OOM fault occurs while trying to recover from a\nprior crash.\n\n4. Fuzz Testing\n\nFuzz testing\nseeks to establish that SQLite responds correctly to invalid, out-of-range,\nor malformed inputs.\n\n4.1. SQL Fuzz\n\nSQL fuzz testing consists of creating syntactically correct yet\nwildly nonsensical SQL statements and feeding them to SQLite to see\nwhat it will do with them. Usually some kind of error is returned\n(such as \"no such table\"). Sometimes, purely by chance, the SQL\nstatement also happens to be semantically correct. In that case, the\nresulting prepared statement is run to make sure it gives a reasonable\nresult.\n\n4.1.1. SQL Fuzz Using The American Fuzzy Lop Fuzzer\n\nThe concept of fuzz testing has been around for decades, but fuzz\ntesting was not an effective way to find bugs until 2014 when\nMichal Zalewski invented the first practical profile-guided fuzzer,\nAmerican Fuzzy Lop or \"AFL\".\nUnlike prior fuzzers that blindly generate random inputs, AFL\ninstruments the program being tested (by modifying the assembly-language\noutput from the C compiler) and uses that instrumentation to detect when\nan input causes the program to do something different - to follow\na new control path or loop a different number of times. Inputs that provoke\nnew behavior are retained and further mutated. In this way, AFL is able\nto \"discover\" new behaviors of the program under test, including behaviors\nthat were never envisioned by the designers.\n\nAFL proved adept at finding arcane bugs in SQLite.\nMost of the findings have been assert() statements where the conditional\nwas false under obscure circumstances. But AFL has also found\na fair number of crash bugs in SQLite, and even a few cases where SQLite\ncomputed incorrect results.\n\nBecause of its past success, AFL became a standard part of the testing\nstrategy for SQLite beginning with version 3.8.10 (2015-05-07) until\nit was superseded by better fuzzers in version 3.29.0 (2019-07-10).\n\n4.1.2. Google OSS Fuzz\n\nBeginning in 2016, a team of engineers at Google started the\nOSS Fuzz project.\nOSS Fuzz uses a AFL-style guided fuzzer running on Google's infrastructure.\nThe Fuzzer automatically downloads the latest check-ins for participating\nprojects, fuzzes them, and sends email to the developers reporting any\nproblems. When a fix is checked in, the fuzzer automatically detects this\nand emails a confirmation to the developers.\n\nSQLite is one of many open-source projects that OSS Fuzz tests. The\ntest/ossfuzz.c source file\nin the SQLite repository is SQLite's interface to OSS fuzz.\n\nOSS Fuzz no longer finds historical bugs in SQLite. But it is still\nrunning and does occasionally find issues in new development check-ins.\nExamples:\n[1]\n[2]\n[3].\n\n4.1.3. The dbsqlfuzz and jfuzz fuzzers\n\nBeginning in late 2018, SQLite has been fuzzed using a proprietary\nfuzzer called \"dbsqlfuzz\". Dbsqlfuzz is built using the\nlibFuzzer framework of LLVM.\n\nThe dbsqlfuzz fuzzer mutates both the SQL input and the database file\nat the same time. Dbsqlfuzz uses a custom\nStructure-Aware Mutator\non a specialized input file that defines both an input database and SQL\ntext to be run against that database. Because it mutates both the input\ndatabase and the input SQL at the same time, dbsqlfuzz has been able to\nfind some obscure faults in SQLite that were missed by prior fuzzers that\nmutated only SQL inputs or only the database file.\nThe SQLite developers keep dbsqlfuzz running against trunk in about\n16 cores at all times. Each instance of dbsqlfuzz program is able to\nevalutes about 400 test cases per second, meaning that about 500 million\ncases are checked every day.\n\nThe dbsqlfuzz fuzzer has been very successful at hardening the\nSQLite code base against malicious attack. Since dbsqlfuzz has been\nadded to the SQLite internal test suite, bug reports from external\nfuzzers such as OSSFuzz have all but stopped.\n\nNote that dbsqlfuzz is not the Protobuf-based structure-aware\nfuzzer for SQLite that is used by Chromium and described in the\nStructure-Aware Mutator article.\nThere is no connection between these two fuzzers, other than the fact that they\nare both based on libFuzzer\nThe Protobuf fuzzer for SQLite is written and maintained by the Chromium\nteam at Google, whereas dbsqlfuzz is written and maintained by the original\nSQLite developers. Having multiple independently-developed fuzzers for SQLite\nis good, as it means that obscure issues are more likely to be uncovered.\n\nNear the end of January 2024, a second libFuzzer-based tool called\n\"jfuzz\" came into use. Jfuzz generates corrupt JSONB blobs and feeds\nthem into the JSON SQL functions to verify that the JSON functions\nare able to safely and efficiently deal with corrupt binary inputs.\n\n4.1.4. Other third-party fuzzers\n\nSQLite seems to be a popular target for third-parties to fuzz.\nThe developers hear about many attempts to fuzz SQLite\nand they do occasionally get bug reports found by independent\nfuzzers. All such reports are promptly fixed, so the product is\nimproved and that the entire SQLite user community benefits.\nThis mechanism of having many independent testers is similar to\nLinus's law:\n\"given enough eyeballs, all bugs are shallow\".\n\nOne fuzzing researcher of particular note is\nManuel Rigger.\nMost fuzzers only look for assertion faults, crashes, undefined behavior (UB),\nor other easily detected anomalies. Dr. Rigger's fuzzers, on the other hand,\nare able to find cases where SQLite computes an incorrect answer.\nRigger has found\nmany such cases.\nMost of these finds are obscure corner cases involving type\nconversions and affinity transformations, and a good number of the finds\nare against unreleased features. Nevertheless, his finds are still important\nas they are real bugs,\nand the SQLite developers are grateful to be able to identify and fix\nthe underlying problems.\n\n4.1.5. The fuzzcheck test harness\n\nHistorical test cases from AFL, OSS Fuzz, and dbsqlfuzz are\ncollected in a set of database files in the main SQLite source tree\nand then rerun by the \"fuzzcheck\" utility program whenever one runs\n\"make test\". Fuzzcheck only runs a few thousand \"interesting\" cases\nout of the billions of cases that the various fuzzers have\nexamined over the years. \"Interesting\" cases are cases that exhibit\npreviously unseen behavior. Actual bugs found by fuzzers are always\nincluded among the interesting test cases, but most of the cases run\nby fuzzcheck were never actual bugs.\n\n4.1.6. Tension Between Fuzz Testing And 100% MC/DC Testing\n\nFuzz testing and 100% MC/DC testing are in tension with\none another.\nThat is to say, code tested to 100% MC/DC will tend to be\nmore vulnerable to problems found by fuzzing and code that performs\nwell during fuzz testing will tend to have (much) less than\n100% MC/DC.\nThis is because MC/DC testing discourages defensive code with\nunreachable branches, but without defensive code, a fuzzer is\nmore likely to find a path that causes problems. MC/DC testing\nseems to work well for building code that is robust during\nnormal use, whereas fuzz testing is good for building code that is\nrobust against malicious attack.\n\nOf course, users would prefer code that is both robust in normal\nuse and resistant to malicious attack. The SQLite developers are\ndedicated to providing that. The purpose of this section is merely\nto point out that doing both at the same time is difficult.\n\nFor much of its history SQLite has been focused on 100% MC/DC testing.\nResistance to fuzzing attacks only became a concern with the introduction\nof AFL in 2014. For a while there, fuzzers were finding many problems\nin SQLite. In more recent years, the testing strategy of SQLite has\nevolved to place more emphasis on fuzz testing. We still maintain\n100% MC/DC of the core SQLite code, but most testing CPU cycles are\nnow devoted to fuzzing.\n\nWhile fuzz testing and 100% MC/DC testing are in tension, they\nare not completely at cross-purposes. The fact that the SQlite test\nsuite does test to 100% MC/DC means that when fuzzers do find problems,\nthose problems can be fixed quickly and with little risk of introducing\nnew errors.\n\n4.2. Malformed Database Files\n\nThere are numerous test cases that verify that SQLite is able to\ndeal with malformed database files.\nThese tests first build a well-formed database file, then add\ncorruption by changing one or more bytes in the file by some means\nother than SQLite. Then SQLite is used to read the database.\nIn some cases, the bytes changes are in the middle of data.\nThis causes the content of the database to change while keeping the\ndatabase well-formed.\nIn other cases, unused bytes of the file are modified, which has\nno effect on the integrity of the database.\nThe interesting cases are when bytes of the file that\ndefine database structure get changed. The malformed database tests\nverify that SQLite finds the file format errors and reports them\nusing the SQLITE_CORRUPT return code without overflowing\nbuffers, dereferencing NULL pointers, or performing other\nunwholesome actions.\n\nThe dbsqlfuzz fuzzer also does an excellent job of verifying\nthat SQLite responds sanely to malformed database files.\n\n4.3. Boundary Value Tests\n\nSQLite defines certain limits on its operation, such as the\nmaximum number of columns in a table, the maximum length of an\nSQL statement, or the maximum value of an integer. The TCL and TH3 test\nsuites both contains numerous tests that push SQLite right to the edge\nof its defined limits and verify that it performs correctly for\nall allowed values. Additional tests go beyond the defined limits\nand verify that SQLite correctly returns errors. The source code\ncontains testcase macros to verify that both sides of each boundary\nhave been tested.\n\n5. Regression Testing\n\nWhenever a bug is reported against SQLite, that bug is not considered\nfixed until new test cases that would exhibit the bug have been added\nto either the TCL or TH3 test suites.\nOver the years,\nthis has resulted in thousands and thousands of new tests.\nThese regression tests ensure that bugs that have\nbeen fixed in the past are not reintroduced into future versions of\nSQLite.\n\n6. Automatic Resource Leak Detection\n\nResource leak occurs when system resources\nare allocated and never freed. The most troublesome resource leaks\nin many applications are memory leaks - when memory is allocated using\nmalloc() but never released using free(). But other kinds of resources\ncan also be leaked: file descriptors, threads, mutexes, etc.\n\nBoth the TCL and TH3 test harnesses automatically track system\nresources and report resource leaks on every test run.\nNo special configuration or setup is required. The test harnesses\nare especially vigilant with regard to memory leaks. If a change\ncauses a memory leak, the test harnesses will recognize this\nquickly. SQLite is designed to never leak memory, even after\nan exception such as an OOM error or disk I/O error. The test\nharnesses are zealous to enforce this.\n\n7. Test Coverage\n\nThe SQLite core, including the unix VFS,\nhas 100% branch test coverage under TH3 in\nits default configuration as measured by\ngcov.\nExtensions such as FTS3 and RTree are excluded from this\nanalysis.\n\n7.1. Statement versus branch coverage\n\nThere are many ways to measure test coverage. The most popular\nmetric is \"statement coverage\". When you hear someone say that their\nprogram as \"XX% test coverage\" without further explanation, they usually\nmean statement coverage. Statement coverage measures what percentage\nof lines of code are executed at least once by the test suite.\n\nBranch coverage is more rigorous than statement coverage. Branch\ncoverage measures the number of machine-code branch instructions that\nare evaluated at least once on both directions.\n\nTo illustrate the difference between statement coverage and\nbranch coverage, consider the following hypothetical\nline of C code:\n\nif( a>b && c!=25 ){ d++; }\n\nSuch a line of C code might generate a dozen separate machine code\ninstructions. If any one of those instructions is ever evaluated, then\nwe say that the statement has been tested. So, for example, it might\nbe the case that the conditional expression is\nalways false and the \"d\" variable is\nnever incremented. Even so, statement coverage counts this line of\ncode as having been tested.\n\nBranch coverage is more strict. With branch coverage, each test and\neach subblock within the statement is considered separately. In order\nto achieve 100% branch coverage in the example above, there must be at\nleast three test cases:\n\na<=b\n\na>b && c==25\n\na>b && c!=25\n\nAny one of the above test cases would provide 100% statement coverage\nbut all three are required for 100% branch coverage. Generally speaking,\n100% branch coverage implies 100% statement coverage, but the converse is\nnot true. To reemphasize, the\nTH3 test harness for SQLite provides the stronger form of\ntest coverage - 100% branch test coverage.\n\n7.2. Coverage testing of defensive code\n\nA well-written C program will typically contain some defensive\nconditionals which in practice are always true or always false.\nThis leads to a\nprogramming dilemma: Does one remove defensive code in order to obtain\n100% branch coverage?\n\nIn SQLite, the answer to the previous question is \"no\".\nFor testing purposes, the SQLite source code defines\nmacros called ALWAYS() and NEVER(). The ALWAYS() macro\nsurrounds conditions\nwhich are expected to always evaluate as true and NEVER() surrounds\nconditions that are always evaluated to false. These macros serve as\ncomments to indicate that the conditions are defensive code.\nIn release builds, these macros are pass-throughs:\n\n#define ALWAYS(X) (X)\n#define NEVER(X) (X)\n\nDuring most testing, however, these macros will throw an assertion\nfault if their argument does not have the expected truth value. This\nalerts the developers quickly to incorrect design assumptions.\n\n#define ALWAYS(X) ((X)?1:assert(0),0)\n#define NEVER(X) ((X)?assert(0),1:0)\n\nWhen measuring test coverage, these macros are defined to be constant\ntruth values so that they do not generate assembly language branch\ninstructions, and hence do not come into play when calculating the\nbranch coverage:\n\n#define ALWAYS(X) (1)\n#define NEVER(X) (0)\n\nThe test suite is designed to be run three times, once for each of\nthe ALWAYS() and NEVER() definitions shown above. All three test runs\nshould yield exactly the same result. There is a run-time test using\nthe sqlite3_test_control ( SQLITE_TESTCTRL_ALWAYS,...) interface that\ncan be used to verify that the macros are correctly set to the first\nform (the pass-through form) for deployment.\n\n7.3. Forcing coverage of boundary values and boolean vector tests\n\nAnother macro used in conjunction with test coverage measurement is\nthe testcase() macro. The argument is a condition for which\nwe want test cases that evaluate to both true and false.\nIn non-coverage builds (that is to say, in release builds) the\ntestcase() macro is a no-op:\n\n#define testcase(X)\n\nBut in a coverage measuring build, the testcase() macro\ngenerates code that evaluates the conditional expression in its argument.\nThen during analysis, a check\nis made to ensure tests exist that evaluate the conditional to both true\nand false. Testcase() macros are used, for example, to help verify\nthat boundary values are tested. For example:\n\ntestcase( a==b );\ntestcase( a==b+1 );\nif( a>b && c!=25 ){ d++; }\n\nTestcase macros are also used when two or more cases of a switch\nstatement go to the same block of code, to make sure that the code was\nreached for all cases:\n\nswitch( op ){\ncase OP_Add:\ncase OP_Subtract: {\ntestcase( op==OP_Add );\ntestcase( op==OP_Subtract );\n/*... */\nbreak;\n}\n/*... */\n}\n\nFor bitmask tests, testcase() macros are used to verify that every\nbit of the bitmask affects the outcome. For example, in the following block\nof code, the condition is true if the mask contains either of two bits\nindicating either a MAIN_DB or a TEMP_DB is being opened.\nThe testcase()\nmacros that precede the if statement verify that both cases are tested:\n\ntestcase( mask & SQLITE_OPEN_MAIN_DB );\ntestcase( mask & SQLITE_OPEN_TEMP_DB );\nif( (mask & (SQLITE_OPEN_MAIN_DB|SQLITE_OPEN_TEMP_DB))!=0 ){... }\n\nThe SQLite source code contains 1184\nuses of the testcase() macro.\n\n7.4. Branch coverage versus MC/DC\n\nTwo methods of measuring test coverage were described above:\n\"statement\" and \"branch\" coverage. There are many other test coverage\nmetrics besides these two. Another popular metric is \"Modified\nCondition/Decision Coverage\" or MC/DC.\nWikipedia\ndefines MC/DC as follows:\n\nEach decision tries every possible outcome.\n\nEach condition in a decision takes on every possible outcome.\n\nEach entry and exit point is invoked.\n\nEach condition in a decision is shown to independently\naffect the outcome of the decision.\n\nIn the C programming language\nwhere && and ||\nare \"short-circuit\" operators, MC/DC and branch coverage are very nearly\nthe same thing. The primary difference is in boolean vector tests.\nOne can test for any of several bits in bit-vector and still obtain\n100% branch test coverage even though the second element of MC/DC - the\nrequirement that each condition in a decision take on every possible outcome -\nmight not be satisfied.\n\nSQLite uses testcase() macros as described in the previous\nsubsection to make sure that every condition in a bit-vector decision takes\non every possible outcome. In this way, SQLite also achieves 100% MC/DC\nin addition to 100% branch coverage.\n\n7.5. Measuring branch coverage\n\nBranch coverage in SQLite is currently measured\nusing gcov with the \"-b\"\noption. First the test program is compiled using options\n\"-g -fprofile-arcs -ftest-coverage\" and then the test program is run.\nThen \"gcov -b\" is run to generate a coverage report.\nThe coverage report is verbose and inconvenient to read,\nso the gcov-generated report is processed using\nsome simple scripts to put it into a more human-friendly format.\nThis entire process is automated using scripts, of course.\n\nNote that running SQLite with gcov is not a test of SQLite —\nit is a test of the test suite. The gcov run does not test SQLite because\nthe -fprofile-args and -ftest-coverage options cause the compiler to\ngenerate different code.\nThe gcov run merely verifies that the test suite provides 100% branch test\ncoverage. The gcov run is a test of the test - a meta-test.\n\nAfter gcov has been run to verify 100% branch test coverage,\nthen the test program is recompiled using delivery compiler options\n(without the special -fprofile-arcs and -ftest-coverage options)\nand the test program is rerun.\nThis second run is the actual test of SQLite.\n\nIt is important to verify that the gcov test run\nand the second real test run both give the same output. Any\ndifferences in output indicate either the use of undefined or\nindeterminate behavior in the SQLite code (and hence a bug),\nor a bug in the compiler.\nNote that SQLite has, over the previous decade, encountered bugs\nin each of GCC, Clang, and MSVC. Compiler bugs, while rare, do happen,\nwhich is why it is so important to test the code in an as-delivered\nconfiguration.\n\n7.6. Mutation testing\n\nUsing gcov (or similar) to show that every branch instruction is taken\nat least once in both directions is a good measure of test suite quality.\nBut even better is showing that every branch instruction makes\na difference in the output. In other words, we want to show\nnot only that every branch instruction both jumps and falls through but also\nthat every branch is doing useful work and that the test suite is able\nto detect and verify that work. When a branch is found that does not\nmake a difference in the output, that suggests that code associated with\nthe branch can be removed (reducing the size of the library and perhaps\nmaking it run faster) or that the test suite is inadequately testing the\nfeature that the branch implements.\n\nSQLite strives to verify that every branch instruction makes a difference\nusing mutation testing.\nA script\nfirst compiles the SQLite source code into assembly language\n(using, for example, the -S option to gcc). Then the script steps through\nthe generated assembly language and, one by one, changes each branch\ninstruction into either an unconditional jump or a no-op, compiles the\nresult, and verifies that the test suite catches the mutation.\n\nUnfortunately, SQLite contains many branch instructions that\nhelp the code run faster without changing the output.\nSuch branches generate false-positives during mutation testing.\nAs an example, consider the following\nhash function\nused to accelerate table-name lookup:\n\n55 static unsigned int strHash(const char *z){\n56 unsigned int h = 0;\n57 unsigned char c;\n58 while( (c = (unsigned char)*z++)!=0 ){ /*OPTIMIZATION-IF-TRUE*/\n59 h = (h<<3) ^ h ^ sqlite3UpperToLower[c];\n60 }\n61 return h;\n62 }\n\nIf the branch instruction that implements the \"c!=0\" test on line 58\nis changed into a no-op, then the while-loop will loop forever and the\ntest suite will fail with a time-out. But if that branch is changed\ninto an unconditional jump, then the hash function will always return 0.\nThe problem is that 0 is a valid hash. A hash function that always\nreturns 0 still works in the sense that SQLite still always gets the correct\nanswer. The table-name hash table degenerates into a linked-list\nand so the table-name lookups that occur while parsing SQL statements\nmight be a little slower, but the end result will be the same.\n\nTo work around this problem, comments of the form\n\" /*OPTIMIZATION-IF-TRUE*/ \" and\n\" /*OPTIMIZATION-IF-FALSE*/ \" are inserted into the SQLite\nsource code to tell the mutation testing script to ignore some branch\ninstructions.\n\n7.7. Experience with full test coverage\n\nThe developers of SQLite have found that full coverage testing is an\nextremely effective method for locating and preventing bugs.\nBecause every single branch\ninstruction in SQLite core code is covered by test cases, the developers\ncan be confident that changes made in one part of the code\ndo not have unintended consequences in other parts of the code.\nThe many new features and performance improvements that have been\nadded to SQLite in recent years would not have been possible without\nthe availability of full-coverage testing.\n\nMaintaining 100% MC/DC is laborious and time-consuming.\nThe level of effort needed to maintain full-coverage testing\nis probably not cost effective for a typical application.\nHowever, we think that full-coverage testing is justified for a\nvery widely deployed infrastructure library\nlike SQLite, and especially for a database library which by its very\nnature \"remembers\" past mistakes.\n\n8. Dynamic Analysis\n\nDynamic analysis refers to internal and external checks on the\nSQLite code which are performed while the code is live and running.\nDynamic analysis has proven to be a great help in maintaining the\nquality of SQLite.\n\n8.1. Assert\n\nThe SQLite core contains 6754 assert()\nstatements that verify function preconditions and postconditions and\nloop invariants. Assert() is a macro which is a standard part of\nANSI-C. The argument is a boolean value that is assumed to always be\ntrue. If the assertion is false, the program prints an error message\nand halts.\n\nAssert() macros are disabled by compiling with the NDEBUG macro defined.\nIn most systems, asserts are enabled by default. But in SQLite, the\nasserts are so numerous and are in such performance critical places, that\nthe database engine runs about three times slower when asserts are enabled.\nHence, the default (production) build of SQLite disables asserts.\nAssert statements are only enabled when SQLite is compiled with the\nSQLITE_DEBUG preprocessor macro defined.\n\nSee the Use Of assert in SQLite document\nfor additional information about how SQLite uses assert().\n\n8.2. Valgrind\n\nValgrind is perhaps the most amazing\nand useful developer tool in the world. Valgrind is a simulator - it simulates\nan x86 running a Linux binary. (Ports of Valgrind for platforms other\nthan Linux are in development, but as of this writing, Valgrind only\nworks reliably on Linux, which in the opinion of the SQLite developers\nmeans that Linux should be the preferred platform for all software development.)\nAs Valgrind runs a Linux binary, it looks for all kinds of interesting\nerrors such as array overruns, reading from uninitialized memory,\nstack overflows, memory leaks, and so forth. Valgrind finds problems\nthat can easily slip through all of the other tests run against SQLite.\nAnd, when Valgrind does find an error, it can dump the developer directly\ninto a symbolic debugger at the exact point where the error occurs, to\nfacilitate a quick fix.\n\nBecause it is a simulator, running a binary in Valgrind is slower than\nrunning it on native hardware. (To a first approximation, an application\nrunning in Valgrind on a workstation will perform about the same as it\nwould running natively on a smartphone.) So it is impractical to run the full\nSQLite test suite through Valgrind. However, the veryquick tests and\nthe coverage of the TH3 tests are run through Valgrind prior to every\nrelease.\n\n8.3. Memsys2\n\nSQLite contains a pluggable\nmemory allocation subsystem.\nThe default implementation uses system malloc() and free().\nHowever, if SQLite is compiled with SQLITE_MEMDEBUG, an alternative\nmemory allocation wrapper ( memsys2 )\nis inserted that looks for memory allocation\nerrors at run-time. The memsys2 wrapper checks for memory leaks, of\ncourse, but also looks for buffer overruns, uses of uninitialized memory,\nand attempts to use memory after it has been freed. These same checks\nare also done by valgrind (and, indeed, Valgrind does them better)\nbut memsys2 has the advantage of being much faster than Valgrind, which\nmeans the checks can be done more often and for longer tests.\n\n8.4. Mutex Asserts\n\nSQLite contains a pluggable mutex subsystem. Depending on\ncompile-time options, the default mutex system contains interfaces\nsqlite3_mutex_held() and sqlite3_mutex_notheld() that detect\nwhether or not a particular mutex is held by the calling thread.\nThese two interfaces are used extensively within assert() statements\nin SQLite to verify mutexes are held and released at all the right\nmoments, in order to double-check that SQLite does work correctly\nin multi-threaded applications.\n\n8.5. Journal Tests\n\nOne of the things that SQLite does to ensure that transactions\nare atomic across system crashes and power failures is to write\nall changes into the rollback journal file prior to changing the\ndatabase. The TCL test harness contains an alternative\nOS backend implementation that helps to\nverify this is occurring correctly. The \"journal-test VFS\" monitors\nall disk I/O traffic between the database file and rollback journal,\nchecking to make sure that nothing is written into the database\nfile which has not first been written and synced to the rollback journal.\nIf any discrepancies are found, an assertion fault is raised.\n\nThe journal tests are an additional double-check over and above\nthe crash tests to make sure that SQLite transactions will be atomic\nacross system crashes and power failures.\n\n8.6. Undefined Behavior Checks\n\nIn the C programming language, it is very easy to write code that\nhas \"undefined\" or \"implementation defined\" behavior.\nThat means that the code might work during development, but then give\na different answer on a different system, or when recompiled using different\ncompiler options.\nExamples of undefined and implementation-defined behavior in\nANSI C include:\n\nSigned integer overflow. (Signed integer overflow does not\nnecessarily wrap around, as most people expect.)\n\nShifting an N-bit integer by more than N bits.\n\nShifting by a negative amount.\n\nShifting a negative number.\n\nUsing the memcpy() function on overlapping buffers.\n\nThe order of evaluation of function arguments.\n\nWhether or not \"char\" variables are signed or unsigned.\n\nAnd so forth....\n\nSince undefined and implementation-defined behavior is non-portable\nand can easily lead to incorrect answers, SQLite works very hard to avoid it.\nFor example,\nwhen adding two integer column values together as part of an SQL statement,\nSQLite does not simply add them together using the C-language \"+\" operator.\nInstead, it first checks to make sure the\naddition will not overflow, and if it will, it does the addition using\nfloating point instead.\n\nTo help ensure that SQLite does not make use of undefined or\nimplementation defined behavior, the test suites are rerun using\ninstrumented builds that try to detect undefined behavior. For example,\ntest suites are run using the \"-ftrapv\" option of GCC. And they\nare run again using the \"-fsanitize=undefined\" option on Clang. And\nagain using the \"/RTC1\" option in MSVC. Then the test suites are rerun\nusing options like \"-funsigned-char\" and \"-fsigned-char\" to make sure\nthat implementation differences do not matter either. Tests are then repeated\non 32-bit and 64-bit systems and on big-endian and little-endian systems,\nusing a variety of CPU architectures.\nFurthermore, the test suites are augmented with many test cases that are\ndeliberately designed to provoke undefined behavior. For example:\n\" SELECT -1*(-9223372036854775808); \".\n\n9. Disabled Optimization Tests\n\nThe sqlite3_test_control ( SQLITE_TESTCTRL_OPTIMIZATIONS,...) interface\nallows selected SQL statement optimizations to be disabled at run-time.\nSQLite should always generate exactly the same answer with optimizations\nenabled and with optimizations disabled; the answer simply arrives quicker\nwith the optimizations turned on. So in a production environment, one always\nleaves the optimizations turned on (the default setting).\n\nOne verification technique used on SQLite is to run an entire test suite\ntwice, once with optimizations left on and a second time with optimizations\nturned off, and verify that the same output is obtained both times. This\nshows that the optimizations do not introduce errors.\n\nNot all test cases can be handled this way. Some test cases check\nto verify that the optimizations really are reducing the amount of\ncomputation by counting the number of disk accesses, sort operations,\nfull-scan steps, or other processing steps that occur during queries.\nThose test cases will appear to fail when optimizations are disabled.\nBut the majority of test cases simply check that the correct answer\nwas obtained, and all of those cases can be run successfully with and\nwithout the optimizations, in order to show that the optimizations do not\ncause malfunctions.\n\n10. Checklists\n\nThe SQLite developers use an on-line checklist to coordinate testing\nactivity and to verify that all tests pass prior to each SQLite release.\nPast checklists\nare retained for historical reference.\n(The checklists are read-only for anonymous internet viewers, but\ndevelopers can log in and update checklist items in their web\nbrowsers.)\nThe use of checklists for SQLite testing and other development activities\nis inspired by\nThe Checklist Manifesto\n.\n\nThe latest checklists contain approximately 200 items that are\nindividually verified for each release. Some checklist items only take\na few seconds to verify and mark off. Others involve test suites\nthat run for many hours.\n\nThe release checklist is not automated: developers run each item on\nthe checklist manually. We find that it is important to keep a human in\nthe loop. Sometimes problems are found while running a checklist item\neven though the test itself passed. It is important to have a human\nreviewing the test output at the highest level, and constantly asking\n\"Is this really right?\"\n\nThe release checklist is continuously evolving. As new problems or\npotential problems are discovered, new checklist items are added to\nmake sure those problems do not appear in subsequent releases. The\nrelease checklist has proven to be an invaluable tool in helping to\nensure that nothing is overlooked during the release process.\n\n11. Static Analysis\n\nStatic analysis means analyzing source code at compile-time to\ncheck for correctness. Static analysis includes compiler\nwarning messages and more in-depth analysis engines such as the\nClang Static Analyzer.\nSQLite compiles without warnings on GCC and Clang using\nthe -Wall and -Wextra flags on Linux and Mac and on MSVC on Windows.\nNo valid warnings are generated by the Clang Static Analyzer tool \"scan-build\"\neither (though recent versions of clang seem to generate many false-positives.)\nNevertheless, some warnings might be generated by other\nstatic analyzers. Users are encouraged not to stress over these\nwarnings and to instead take solace in the intense testing of SQLite\ndescribed above.\n\nStatic analysis has not been helpful in finding\nbugs in SQLite. Static analysis has found a few bugs in SQLite, but\nthose are the exceptions. More bugs have been\nintroduced into SQLite while trying to get it to compile without\nwarnings than have been found by static analysis.\n\n12. Summary\n\nSQLite is open source. This gives many people the idea that\nit is not well tested as commercial software and is perhaps unreliable.\nBut that impression is false.\nSQLite has exhibited very high reliability in the field and\na very low defect rate, especially considering how rapidly it is evolving.\nThe quality of SQLite is achieved in part by careful code design and\nimplementation. But extensive testing also plays a vital role in\nmaintaining and improving the quality of SQLite. This document has\nsummarized the testing procedures that every release of SQLite undergoes\nwith the hope of inspiring confidence that SQLite is\nsuitable for use in mission-critical applications.\n\nThis page was last updated on 2025-05-31 13:08:22Z"}
