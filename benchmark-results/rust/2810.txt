{"title":"leah blogs","author":null,"date":null,"main_content":"This year, I decided to use Advent of\nCode to learn the language\nSwift. Since there were only 12 days of\ntasks for 2025, here is my summary of experiences.\nAlso check out my solutions.\n\nTooling\n\nI used Swift 6.2 on Void Linux, which I compiled from scratch since there\nwere no prebuilt binaries that worked with a Python 3.13 system\n(needed for lldb). It’s possible to bootstrap Swift from just a\nclang++ toolchain, so this wasn’t too tedious, but it still required\nlooking up Gentoo ebuilds how to pass configuration properly. As an\nend user, this should not worry you too much.\n\nTooling in general is pretty nice: there’s an interpreter and you\ncan run simple “scripts” directly using swift foo.swift. Startup\ntime is short, so this is great for quick experiments. There’s also a\nREPL, but I didn’t try it yet. One flaw of the interpreter (but\npossibly related to my setup) is that there were no useful backtraces\nwhen something crashed. In this case, I compiled a binary and used\nthe included lldb, which has good support for Swift.\n\nThere’s also a swift-format tool included to format source code.\nIt uses 2 spaces by default, but most code in the wild uses 4 spaces\ncuriously. I’m not sure when that changed.\n\nSince I only write simple programs using a single source file,\nI didn’t bother looking at swift-build yet.\n\nBy default, programs are linked dynamically against the standard\nlibrary and are thus super compact.\nUnfortunately, many modern languages today don’t support this properly.\n(Statically linking the standard library costs roughly 10MB,\nwhich is fair too.)\n\nThe language\n\nIn general, the language feels modern, comfy, and is easy to pick up.\nHowever, I found some traps as well.\n\nThe syntax is inspired by the C family and less symbol-heavy than\nRust’s. There’s a block syntax akin to Ruby for passing closures.\n\nError handling can be done using checked exceptions, but there are\nalso Optional types and Result types like in Rust, and syntactic\nshortcuts to make them convenient.\n\nThe standard library has many practical functions, e.g. there’s a\nfunction Character.wholeNumberValue that works for any Unicode digit\nsymbol.\nThere’s a Sequence abstraction over arrays etc. which has many useful functions\n(e.g. split(whereSeparator:), which many other standard libraries lack).\nThe standard library is documented well.\n\nThe string processing is powerful, but inconvenient when you want to\ndo things like indexing by offsets or ranges, due to Unicode semantics.\n(This is probably a good thing in general.)\nI switched to using arrays of code-points for problems that required this.\n\nOn Day 2, I tried using regular\nexpressions, but I found serious performance issues: first I used a\nRegexp literal ( #/.../# ) in a loop, which actually resulted in\ncreating a new Regexp instance on each iteration; second, Regexp\nmatching itself is quite slow. Before I extracted the Regexp into a\nconstant, the program was 100x as slow as Ruby(!), and after it still\nwas 3x as slow. I then rewrote the solution to not use Regexps.\n\nPrefix (and suffix) operators need to “stick” to their expression, so\nyou can’t write if! condition. This is certainly a choice: you can\ndefine custom prefix and suffix operators and parsing them\nnon-ambiguously is easier, but it’s probably not a thing I would have\ndone.\n\nSwift functions often use parameter names (probably for compatibility\nwith Objective-C). They certainly help readability of the code, but I\nthink I prefer OCaml’s labeled arguments, which can be reordered and\npermit currying.\n\nThe language uses value semantics for collections and then optimizes\nthem using copy-on-write and or by detecting inout parameters (which\nare updated in-place). This is quite convenient when writing code\n(e.g day 4 )\nGarbage collection is done using reference counting. However, some\nAoC tasks turned out to make heavy use of the garbage collector, where\nI’d have expected the compiler to use a callstack or something for\nintermediate values. Substrings are optimized by a custom type\nSubstring, if you want to write a function to operate on either\nstrings or substrings, you need to spell this out:\nfunc parse<T>(_ str: T) ->... where T: StringProtocol\n\nThere’s a library swift-algorithms\nadding even more sequence and collection algorithms, which I decided not to use.\n\nDownsides\n\nThe compiler is reasonably fast for an LLVM-based compiler. However,\nwhen you manage to create a type checking error, error reporting is\nextremely slow, probably because it tries to find any variant that\ncould possibly work still. Often, type checking errors are also confusing.\n\n(Error messages unrelated to type checking are good and often really\nhelpful, e.g. if you accidentally use '' -quotes for strings\nor try to use [] as an empty map, it tells you how to do it right.)\n\nRanges can be inclusive... or right-exclusive..<. Constructing\na range where the upper boundary is smaller than the lower boundary\nresults in a fatal error, whereas in other languages it’s just an\nempty range.\n\nSome “obvious” things seem to be missing, e.g. tuples of Hashable\nvalues are not Hashable currently (this feature was removed in 2020,\nafter trying to implement the\nproposal\nthat introduced it, and no one bothered to fix it yet?), which is\npretty inconvenient.\n\nLikewise, the language has pattern matching for algebraic data types\nand tuples, but unfortunately not for arrays/sequences, which is\ninconvenient at times.\n\nSince I was just picking up Swift, I had to search stuff online a lot\nand read Stack Overflow. I noticed I found many answers for prior\nversions of Swift that changed in the mean time (even for basic tasks).\nFor a language that’s been around for over 10 years, this seems like\nquite some churn. I hope the language manages to stabilize better and\ndoesn’t just get new features bolted on continuously.\n\n⁂\n\nIn general, using Swift was fun and straight-forward for these programming tasks.\nFor writing serious applications on non-MacOS systems,\nthere’s also the question of library availability.\nSome parts of the language still feel unfinished or unpolished,\nin spite of being around for quite some time.\n\nNP: Adrianne Lenker—Promise is a Pendulum"}
