Utilizing the Agentic Lifecycle for Improved Generative Results It’s so easy to jump into the deep end, to vibe. With the rise of modern editors with agentic capabilities such as Cursor, Zed and of course Visual Studio Code, it’s never been easier to jump into the craze of vibe coding . We have never had it so easy to jump straight into agent mode! What many have come to learn the hard way is, this can lead to many unanticipated changes or assumptions, leading to surprise behaviours or hallucinations from the generative model. Providing a prompt such as write unit tests for the OrderServiceController within the context of a codebase would imply that it would write unit tests for said OrderServiceController . If a developer were to be more ambitious or worse, ambiguous in their prompt, what can we ascertain will be the result? Generative AI hallucinates quite often, sometimes in subtle, quiet ways which aren’t obvious either. They pull in dependencies which don’t exist, or rewrite entire modules because it doesn’t comply with their ideal result to your prompt; they rewrite the same function five times before opting to rewrite the entire package. That’s where understanding and employing a lifecycle which allows us to provide context, answer questions and even go over design and concepts before any code has been changed allows for a more effective partnership with AGI tools. Those agent-enabled editors mentioned support this entire lifecycle out of the box thanks to their chat-like widgets with the ‘assistant’ using one of three modes that can be used: Ask, Edit, and Agent. Ask “Ask mode for chat is optimized for asking questions about your codebase, coding and general technology concepts. Ask mode is particularly useful for getting a better understanding of your codebase, brainstorming ideas, and getting help with coding tasks.” Visual Studio Code Ontop of those listed, I find ask utterly fantastic for learning a new codebase, troubleshooting an error within a module, and planning how to approach my task within the file. We can use Ask to essentially chat with our AI assistant without any codebase changes occurring, allowing developers to “go over” their thinking and design, revising and prompting for additional insights or known issues as if you were pair programming. By going over intention, design and behaviour prior to using Edit or Agent mode, we can benefit from outcomes like: A concrete understanding of the proposed changes Details are more defined for the AGI to follow, leading to less hallucination potential or surprises. The ability to catch conflicting concepts before anything has been changed Edit “With [chat] edit mode in Visual Studio Code, you can use natural language to make edits across multiple files in your project. The edits are applied directly in your editor, where you can review them in-place, with the full context of the surrounding code” Visual Studio Code Edit allows for us to make surgical modifications to a single file; all within the scope that a developer can follow along and review. Use this when you have small changes within a single file, such as updating the behaviour of a function or writing new test cases, or improving the perfect README.md . The outcomes from this mode should be pretty clear: Visible, easy to review changes to your file Leverage Generative AI to update and improve a document, section, etc. Agent “Agent mode is optimized for complex coding tasks based on high-level requirements that might require running terminal commands and tools. The AI operates autonomously, determining the relevant context and files to edit, planning the work needed, and iterating to resolve issues as they arise.” Visual Studio Code Agent truly shows the power, danger and potential fallacy of giving an AGI access to an entire local codebase. An Agent can do far more than just edit a single file. It can scaffold directories and files until it’s blue in the face, it can write a TODO list of a hundred items and proceed to do them, it can run commands from your terminal. More impressively, it can also interface with Model Context Protocol (MCP) servers to carry out more complicated and intertwined tasks between services. Good — here’s a practical, actionable guide to improving your developer-image process and shrinking / speeding up your Containerfiles. I’ll split this into (A) process & workflow improvements, (B) Containerfile optimization techniques you can apply immediately, and (C) concrete examples and commands you can copy into your repo. A — Process & workflow improvements Separate concerns: keep a small, stable base image and build language/tool images that inherit from it. This reduces how often base layers change and speeds builds. Build-time vs runtime installs: install only build-time tools during image build; defer large or transient installs (like Bun/amp) to an entrypoint (you already implemented this — good). Use persistent volumes for caches & state: mount ~/.amp , language caches (cargo, go, npm/pnpm, pip) from the host so you avoid repeated downloads across container runs. Use explicit versioning: pin package/tool versions (or base image tags) so builds are reproducible. Leverage layer caching & CI cache: push intermediate images or use a cache-from mechanism in CI to avoid full rebuilds. Run as non-root and match UIDs: accept build args for UID/GID so files created by the container are owned on the host correctly. Developer ergonomics: mount repo into /work , add a dev entrypoint that sets up PATH and envs, provide quick run scripts in repo (e.g., make dev-shell ). B — Containerfile optimizations (practical rules) Combine package manager steps into a single RUN to limit layers and remove caches: e.g. RUN dnf install -y ... && dnf clean all && rm -rf /var/cache/dnf Don’t install large optional packages unless needed; disable weak deps where supported. Use multi-stage builds for compiled artifacts (build toolchain in one stage, copy only runtime files into final image). Defer heavy downloads to startup (entrypoint) for faster rebuilds; you already do this for Amp. Use BuildKit cache mounts for language package caches when building (npm, pip, cargo, go). If you use BuildKit or docker buildx, add -mount=type=cache in a Dockerfile/Containerfile to persist package caches between builds. Use an explicit .dockerignore / .containerignore to exclude large files from build context. Keep images small by removing docs and unnecessary locales if applicable. Create a “dev-tools” image with heavy tooling (clang, cmake, docker/gcloud CLI, large SDKs) that developers can podman run into — instead of baking everything into every language image. Use build args for UID/GID & optional features (e.g., -build-arg WITH_RUST=true ). Tag images by semantic version / commit hash; store dev image variants separately from CI/production ones.
